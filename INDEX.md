# WP-GJ-Builder — architecture index

Navigation map for this codebase. Not user docs — a lookup table of files/classes/functions and what they're for. Keep updated as files are added/renamed/removed.

## Editor front-end (assets/editor/)

Bundled by webpack (`webpack.config.js`) into `assets/build/editor.js` + `editor.css`. Enqueued only on the `wpgjb-editor` admin screen (`includes/Admin/EditorPage.php::maybe_enqueue()`).

- **index.js** — entry point. Boots `grapesjs.init()` with `grapesjs-preset-webpage` (stock panels/toolbar) + `@silexlabs/grapesjs-css-variables` (theme-token pickers in Style Manager). Owns: DOM shell (`.wpgjb-shell` → status strip + lock banner + `.wpgjb-workspace` with left/right panels flanking `#wpgjb-gjs-root`), i18n config (`grapesjs/locale/ru`/`en` chosen dynamically via `bootstrap.uiLang`, see language-switcher gotcha below), left-panel 3-tab filter (`setupLeftPanelTabs` — Элементы/Блоки/Страницы over one BlockManager instance via `BlockManager.render(filtered)`), right-panel 3-tab show/hide (`setupRightPanelTabs` — Свойства/Настройки компонента/Слои), `registerBlocksInBlockManager` (registers the manifest blocks as BlockManager entries, label via `blockDisplayName()` which strips a legacy `test-` id prefix and title-cases the rest), `editor.RichTextEditor.add('wrapUl'/'wrapOl', ...)` (list buttons in the native RTE toolbar — `rte.exec('insert{Unordered,Ordered}List')`, only ever surfaces on free Элементы since Blocks' text never gets native RTE at all, see `rte:enable` guard), `setupLangSwitch()` (status-strip button, reloads with `?wpgjb_lang=ru|en` toggled — language choice is NOT persisted client-side, URL is the only source of truth), autosave/save-indicator/conflict-prompt, publish/export-json/import-json commands (added to preset's `options` panel via `Panels.addButton`), Heartbeat lock/takeover, `rte:enable` guard (blocks inline-RTE bypass of the Content form for Block-internal text), `component:selected`/`component:deselected` → `renderComponentSettingsPanel()` (see gotcha below — replaced the old Modal-based Content form). Backbone `noConflict()` shim at top (GrapesJS's bundled Backbone silently breaks `wp.media()` otherwise).
- **blocks.js** — pure component-model/data layer for the manifest-driven "Блоки", UI-agnostic. `extractValuesFromTemplate`/`renderBlockElement`/`renderBlockHtml` (markup+values ⇄ HTML codec), `readValues()`'s array-slot branch reads **every** `[data-slot-item]` in a `[data-slot-repeat]` container via `findOwnScopeAll()` (not just the first — see gotcha below), `registerBlockType`/`registerAllBlocks` (Components.addType + BLOCK_TOOLBAR; **no traits** — Content form renders inline instead, see index.js), `getComponentValues`/`setComponentValues` (`data-wpb-values` JSON attribute read/write), `applyResolvedImagesToComponent` (async image src/srcset resolution post-insert), `resolveBlockRoot`/`getBlockIdForComponent` (walks up to nearest `data-wpb-block` ancestor — scopes Block selection/RTE-guard to whole-block, no-ops for free Elements). `BLOCK_TOOLBAR`'s titles/aria-labels are `t()`-wrapped (were hardcoded Russian literals until an i18n audit caught it — every Block AND every Element uses this toolbar).
- **elements.js** — static atomic "Элементы", grouped into BlockManager categories by `def.group` via `elementCategory(group)`/`ELEMENT_GROUP_LABELS` (`text`/`buttons-links`/`media`/`forms`/`layout`/`interactive` — same "grouped like Blocks" treatment Blocks already had via `section_type`; elements-interactive.js imports the same helper so both files' Elements land in one consistent category set, not two parallel ones). Members: heading/text/button(variant trait)/divider/video/container/columns/spacer/image/6 form-input variants(text/email/textarea/checkbox/radio/select)/accordion(native `<details>`, zero JS)/icon(trait-driven inline-SVG picker)/email-link/phone-link/4 list variants (bullet `<ul>`, numbered `<ol>`, checklist — fixed check-icon per `<li>`, icon-list — `wpgjb-icon-list` custom type, one `iconName` trait re-styles every item's icon span without touching each item's text). `ICON_PATHS`/`icon()` (exported, reused by elements-interactive.js) is the single source of SVG path data for both BlockManager preview media and the Icon/icon-list elements' selectable sets. `THEME_ACCENT`/`THEME_ACCENT_TEXT`/`THEME_BORDER` — `var( --wp--preset--color--X, #fallback )` constants reused across button/input/accordion/icon-list styles, so canvas content picks up the active theme's palette when available instead of a fixed hex (editor UI *chrome* — panels/tabs/toolbar — intentionally stays WP-admin-colored, unrelated to this). `registerButtonType`/`registerIconType`/`registerIconListType`/`registerSpacerType` are the real `Components.addType`s; `registerImageCommandOverride` overrides `open-assets` to open `wp.media()` for free images while no-op'ing (deferring to the Content form) for images inside a Block. Exports `ELEMENT_BLOCK_IDS` (consumed by index.js's left-tab filter).
- **elements-interactive.js** — the 4 interactive/animated "Элементы" (grouped `interactive`/`media`, see elements.js): countdown (custom `datetime` Trait type via `TraitManager.addType`, target stored as a real HTML attribute so the frontend runtime can read it), counter (changeProp trait mirrored into an attribute on change), gallery (multi-select `wp.media`, picked images become real `<img>` children directly — no template/regeneration layer needed), slider (multi-select `wp.media`, `component.find(selector)` locates the fixed track/dots sub-components to refill on pick). Canvas shows a static preview only — real behavior (ticking/animating/lightbox/sliding) comes from `assets/runtime/wpgjb-elements-runtime.js` on the published frontend, gated by `data-wpgjb-{countdown,counter,gallery,slider}` marker attributes (same principle as Blocks' per-block JS, but a parallel non-manifest path — see PageAssets.php). These marker/state attributes had to be added to `Sanitizer::richness_page()`'s whitelist explicitly (wp_kses has no `data-*` wildcard) — forgetting a new attribute here means it silently vanishes at publish and the frontend runtime finds nothing.
- **elements-infographic.js** — 4 data-driven "Элементы" (столбчатая/линейная/круговая диаграмма, таблица с сортировкой), grouped `interactive`. Deliberately hand-rolled CSS/SVG rendering, NOT Chart.js — `assets/runtime/wpgjb-elements-runtime.js` is a plain vanilla-JS file, not part of the webpack build (see below), so a bundled npm charting lib has no route to reach the published frontend without a whole new build entry + PHP asset-enqueue point; decided with the user not to add that mechanism for this. Bar/line/pie share one data model: a `chartDataText` changeProp Trait (`textarea`, one `label, value` line per row, parsed by `parseChartDataText()`), regenerated into static HTML/SVG by `onValueChange()` on every trait change — same "bake the real HTML once, no runtime reconstruction" principle as counter/gallery. The runtime does NOT need the JSON at all (unlike a hypothetical live-data widget): bars/SVG points are already real DOM at publish time, so `initCharts()` only attaches a scroll-triggered entrance animation (IntersectionObserver, same pattern as `initCounters()`) by toggling plain inline styles (`transform`, `stroke-dashoffset`) — no CSS file, no `@keyframes`, matches this file's "everything is inline style" convention. Data table (`wpgjb-datatable`) is a real `<table>`/`<thead>`/`<tbody>`, cells are plain GrapesJS `text` components (double-click-editable like any other text), toolbar carries 4 custom add/remove-row/column commands (`wpgjb-table-add-row` etc., operate via `component.find('[data-wpgjb-datatable-thead]')`/`-tbody` markers + `.append()`/`.remove()`) — sorting itself is 100% frontend-runtime (`initDataTables()`, numeric-aware compare, sets `aria-sort`).
- **elements-content.js** — 7 "Элементы": tabs, flip-card, hotspot, testimonial slider, icon box, image box, quote. Tabs (`wpgjb-tabs`): nav buttons + droppable panels (panels ARE droppable, unlike almost every other Element here, so users can drop arbitrary content into a tab), a real `hidden` attribute on inactive panels at bake time, `initTabs()` swaps `hidden`/`aria-selected` on click. Flip-card (`wpgjb-flipcard`): front/back droppable faces (`backface-visibility:hidden`, one rotated 180deg), flip driven ENTIRELY by JS (`initFlipCards()` — direct `mouseenter`/`mouseleave` listeners per card, NOT `window.WPGJB.onDelegate()`) because `mouseenter`/`mouseleave` don't bubble and a single document-level delegated listener (the `onDelegate` pattern used everywhere else in this runtime) would not reliably fire for descendants — click is still delegated normally (touch fallback, `hover-vs-tap` a11y principle). Hotspot (`wpgjb-hotspot`): toolbar "add point" command appends a `wpgjb-hotspot-dot` child (own Trait-driven `topPct`/`leftPct` position, `init()`/`onPosChange()` mirrors counter's changeProp pattern) with a nested tooltip `<span>`, `initHotspots()` toggles `display`/`aria-expanded` on click. Testimonial slider (`wpgjb-testimonial-slider`) deliberately REUSES the existing image-slider track/dots/prev/next DOM contract from `elements-interactive.js` byte-for-byte (`data-wpgjb-slider(-track|-dots|-dot|-prev|-next)`/`data-wpgjb-slide`) — `initSliders()` in the runtime matches by that marker generically, not by component type, so the shared prev/next/dot click behavior works with ZERO new runtime code; only the per-slide markup differs (avatar + name + role + static 5-star row + quote, all RTE-editable text). The base `wpgjb-slider` type's `isComponent()` had to gain an extra `!el.getAttribute('data-wpgjb-slider-variant')` guard so it doesn't shadow-match the testimonial variant's root div on HTML re-parse (both share `data-wpgjb-slider="true"`). Icon box / image box: compound elements pairing the EXISTING icon-picker (`ICON_PATHS`)/`wp.media` single-image picker with a heading+text pair, `direction` changeProp trait (vertical/horizontal), distinct from the pre-existing atomic Icon/Image Elements (elements.js).
- **elements-codeblock.js** — "Блок кода": language `select` trait + code `textarea` trait, hand-rolled `highlightCode(code, lang)` (one combined regex per call — comment/string/number/keyword alternation matched in a SINGLE pass against the RAW code, each captured/uncaptured slice individually `escapeHtml()`'d before insertion — deliberately NOT escape-then-regex, which would risk the keyword/string patterns matching *inside* HTML entities or double-escaping). Explicitly a **display-only** element per the user's clarification ("подсветка синтаксиса, без исполнения") — no `<script>`/iframe/raw-HTML execution path at all, so none of the Code Embed security questions (capability gating, sandboxed iframe) that a "paste arbitrary embed code" feature would have needed actually apply here. Copy-to-clipboard label text (`data-wpgjb-codeblock-copied-label`) is baked by the EDITOR (which has `@wordpress/i18n` access) into a data attribute and just read verbatim by `initCodeCopyButtons()` — the runtime file itself has no i18n import capability (see below), so any user-facing string the runtime needs to inject at runtime (not just toggle) must be pre-baked this way, not hardcoded in the runtime JS.
- **templates.js** — the "Страницы" tier. `registerTemplateBlocks(editor, blocks, templates)` builds one BlockManager entry per template by concatenating `renderBlockHtml()` output per member block (client-side, so inserted blocks stay real `data-wpb-block` components, not baked HTML). Exports `TEMPLATE_ID_PREFIX` (`wpgjb-tpl-`, used by index.js's left-tab filter).
- **content-form.js** — the manifest slot-editing form (Content form) for Blocks: `renderContentForm`/`renderSlotField`/`renderImageField`/`renderArrayField`. Takes a container element param; rendered inline into `#wpgjb-content-form` (right panel) by `index.js`'s `renderComponentSettingsPanel()`, not a Modal.
- **page-settings.js** — `openPageSettingsModal(editor, bootstrap, onSaved)`, Elementor-style page settings (title/slug/status/featured image via `wp.media`/page template) in a Modal, opened via the `wpgjb-open-page-settings` command/toolbar button (only registered when `bootstrap.type === 'page'`). Saves via `api.updatePageSettings()`.
- **images.js** — image resolution helpers: `cacheFromMediaModel` (sync cache from a just-picked wp.media Backbone model), `fetchAttachment` (async REST fallback for already-saved IDs), `applyResolvedImage` (sets src/srcset/sizes/loading|fetchpriority on a detached element).
- **api.js** — thin REST client: `loadDocument`, `saveDraft`, `publish`, `fetchThemeTokens`, `fetchThemeChrome`, `updatePageSettings` (single options-object param: `{title, slug, status, featuredMedia, pageTemplate}`). Only place that knows REST URLs/nonce.
- **lock.js** — `initLock()`, Heartbeat-based post-lock/takeover detection.
- **editor.css** — WP-admin color theme via GrapesJS's own CSS custom properties (`--gjs-primary-color` etc., see gotcha below), left/right panel (`flex: 0 0 clamp(280px, 20vw, 460px)` — responsive width, not fixed) + status-strip/lock-banner chrome, Content-form field styles, BLOCK_TOOLBAR icon glyphs.

### Gotchas found empirically (not in any doc)
- `grapesjs-preset-webpage` defaults `useCustomTheme: true`, which injects a **literal** 4-color `<style>` tag at runtime (`#463a3c`/`#b9a5a6`/`#804f7b`/`#d97aa6` for `.gjs-one-bg` etc.) that wins over our CSS-variable-based theme override since it's added later in `<head>`. Must pass `useCustomTheme: false` in `pluginsOpts['grapesjs-preset-webpage']`.
- The preset calls `editor.Panels.getPanels().reset([...])` on load — replaces the whole panel set with `commands`/`devices-c`/`options`/`views`. We `removePanel('views')` post-init since its buttons (open-sm/open-tm/open-layers/open-blocks) are superseded by the persistent left/right tab panels.
- `editor.on('rte:enable', view => ...)`: `view.disableEditing()` called synchronously inside the handler does NOT win — GrapesJS sets `contentEditable` after the event fires in the same synchronous flow. Must defer via `setTimeout(() => view.disableEditing(), 0)`.
- Trait-type `button` commands don't receive `opts.component` — always use `editor.getSelected()` inside the command (reliable since the Trait Manager only ever shows traits for the current selection).
- Right panel's "Настройки компонента" tab hosts TWO sibling containers inside `#wpgjb-traits-panel`: `#wpgjb-traits-native` (the real `traitManager.appendTo` target — mounted once, never re-parented) and `#wpgjb-content-form` (plain div, `renderContentForm()` re-renders into it on every selection change). `component:selected`/`component:deselected` toggle `hidden` on whichever one applies (Block → content form, Element/nothing → native traits) — never clear/rebuild the native-traits container itself, that would break the mounted TraitManager view for future selections.
- `Sanitizer::richness_page()` — the contour-2 (publish) HTML whitelist — has NO `data-*` wildcard support (wp_kses limitation, each attribute is an exact key). Any new component that needs a `data-*` attribute to survive to the published frontend (i.e. anything read by a runtime script, like the interactive Elements) must add that exact attribute name here, or it silently vanishes at publish with no error — found by a real publish+frontend-fetch Puppeteer check, not by code review (the countdown's marker attribute was present in `project_data` and the runtime script was correctly enqueued, but the marker was missing from the actual served HTML). Attributes that are purely editor-side recognition markers with no frontend behavior depending on them (`data-wpgjb-spacer`, `data-wpgjb-icon-el`, `data-wpgjb-button`) are NOT whitelisted and are fine to lose at publish.
- `BlockManager.render(filteredArray)` must run AFTER all blocks are registered — calling it before (e.g. at tab-setup time, before `registerAllBlocks`/`registerElementBlocks`/`registerTemplateBlocks` have run) filters an empty collection and every tab ends up showing everything.
- Device switcher (`devices-c` panel) renders visible by default in this preset+core version pairing despite the preset's own source setting `config.showDevices = false` — no extra re-enable code was needed (verified empirically, contradicts a naive source read).
- Canvas theme-style inheritance: font-family/color/font-weight already match exactly between canvas and published frontend for bare/unclassed tags (global-styles.css cascades correctly with no wrapper markup needed) — only fluid/`clamp()` font-size differs slightly, and only because the canvas iframe width differs from the real viewport at check-time; this is inherent to any iframe-based canvas, not a real gap.
- Real header/footer chrome (Elementor-style "edit in page context") is injected client-side: `editor.on('load', ...)` fetches `GET /theme/chrome` (server renders `<!-- wp:template-part {slug:header|footer} /-->` via `do_blocks()`, block themes only — classic themes get `isBlockTheme:false` and no chrome) and inserts the HTML as siblings of the GrapesJS wrapper element directly inside the canvas iframe body (confirmed empirically: the wrapper is a plain child of `<body>`, siblings survive undisturbed). Chrome gets `pointer-events:none` via a `<style>` tag injected into the iframe head — never touches `editor.Css`/project data, so it can never leak into `getCss()`/publish output.
- **Reverted attempt** ("Вариант А" in a design discussion with the user): extending the above chrome injection to classic themes (`get_header()`/`get_sidebar()`/`get_footer()` output buffering + DOMDocument body-fragment extraction) plus splicing in a real sidebar (moving the GrapesJS wrapper DOM node into a new flex-row container) broke the editor on the user's real site — reverted (commit reverting `a0607d2`) without root-causing exactly which part failed, per explicit instruction to roll back rather than debug further. Do not re-attempt this without first understanding what broke; the code for reference is in that commit's diff/revert.
- Two SAFE, additive style-inheritance fixes (no DOM restructuring, no calling theme template functions) landed instead, addressing the same underlying complaint ("canvas doesn't look like the real site, especially background") at lower risk: (1) `EditorPage::compute_body_class()` — a hand-computed, non-`WP_Query` approximation of `body_class()` (`page`, `page-id-N`, `page-template-X`, `custom-background`) applied directly to the canvas iframe's `<body className>` on `editor.on('load', ...)`, since classic themes commonly scope background/typography rules to these classes rather than a bare `body{}`; (2) `ThemeStylesController::get_custom_background_css()` — a new `/theme/custom-background.css` canvasStyles entry that captures the theme's REAL registered `add_theme_support('custom-background')` `wp-head-callback` (not hardcoded to `_custom_background_cb`, in case a theme overrides it) via output buffering — this WP Customizer background feature is entirely separate from a theme's own style.css/theme.json and was invisible to the canvas before. Both were empirically verified against a throwaway classic theme with a Customizer-configured background color (canvas background now matches exactly, `#336699` end to end) with zero regression on the existing block-theme header-chrome path.
- `_custom_background_cb()`'s emitted CSS is ALWAYS scoped to selector `body.custom-background` (hardcoded in WP core, `wp-includes/theme.php`) — the class itself is normally added by core's `get_body_class()` under a specific condition (`current_theme_supports('custom-background') && (get_background_color() !== default-color || get_background_image())`, `wp-includes/post-template.php`). Missing this exact condition in `compute_body_class()` means the fetched CSS rule technically loads but matches nothing — found by comparing the fetched CSS text against the actually-applied `getComputedStyle` background in a live Puppeteer check, not by reading WP's docs (which don't spell out the selector-vs-class coupling).
- GrapesJS's own `.gjs-cv-canvas` (the canvas element inside `#wpgjb-gjs-root`) hardcodes `width: calc(100% - var(--gjs-left-width))` with `--gjs-left-width: 15%` by default — a reserved strip for the preset's OWN left-docked Blocks/Layers panel, which we don't use (ours live in the sibling `.wpgjb-left-panel` via `appendTo`). That 15% was pure dead space inside the canvas container. Fix: `#wpgjb-gjs-root { --gjs-left-width: 0%; }` in editor.css — custom properties inherit down to the descendant `.gjs-cv-canvas`, no GrapesJS API involved.
- `readValues()`'s array-slot handling in blocks.js used to read only `repeatContainer.querySelector('[data-slot-item]')` (singular — first match) and always produced a one-element values array, REGARDLESS of how many `[data-slot-item]`s actually exist in a block's markup.html. This silently capped every freshly-dragged repeatable block (pricing/features/testimonials/gallery/timeline/...) at exactly 1 item even after markup.html was edited to ship 3 or 6 — found by an actual BlockManager `content` inspection in Puppeteer, not by reading the code. `applyValues()` (the render/apply side) was NEVER limited this way — it already looped over the full `values[key]` array correctly. Fixed by adding `findOwnScopeAll()` (collects every own-scope match, not just the first) and using it in the array branch of `readValues()`.
- Language switcher (`?wpgjb_lang=ru|en`, `EditorPage::resolve_ui_lang()`): do **not** use `switch_to_locale()` for this — `WP_Locale_Switcher::switch_to_locale()` silently returns `false` (no error) if the target locale isn't in `get_available_languages()`, i.e. it requires a full core language pack to be installed. A site without an installed `ru_RU` pack would make the switch a silent no-op. The actual fix: hook `'pre_determine_locale'` (short-circuits `determine_locale()`, which `load_script_textdomain()`/`_load_script_textdomain_from_src()` use to pick the JS translation file) and `'locale'` (short-circuits `get_locale()`, which PHP-side `__()` calls use) directly — both filters are unconditional, no installed-language-pack gate, and are exactly what those two lookup paths already consult internally. Confirmed by reading `wp-includes/l10n.php` source directly (not from memory/docs) after empirically observing `switch_to_locale('ru_RU')` do nothing on a Playground instance with only `en_US` installed.
- `wp_set_script_translations($handle, $domain, $path)`'s translation filename convention CHANGED in WP 7.0: `_load_script_textdomain_from_src()` now builds `"{$domain}-{$locale}-{$handle}.json"` — the **literal script handle**, not `md5($handle)`. The pre-7.0 convention (`{domain}-{locale}-{md5(handle)}.json`, widely documented/remembered from older WP) no longer applies. For this plugin: `languages/wp-gj-builder-en_US-wpgjb-editor.json` (handle is `wpgjb-editor`). Verified by reading the actual bundled core source (`wp-includes/l10n.php`) in the WP Playground temp directory, not assumed from memory — got it wrong on the first attempt (built the file with the md5 name, translations silently didn't load) before checking.

- GrapesJS's built-in `type: 'image'` component renders an internal placeholder (a `data:image/svg+xml;base64,...` background) when the component has no real `src` yet — this is meant as editor-canvas-only UX, but `editor.getHtml()`/publish serializes it as a literal `src` attribute, and `wp_kses()` (via `Sanitizer::sanitize_html()`) strips the disallowed `data:` URI scheme, leaving a mangled fragment (`image/svg+xml;base64,...` with the scheme cut off) that the browser then requests as a relative URL — a real 404 on the published page, found via a Puppeteer `response` listener, not by reading the code. Fix used for every new element with an optional image (hotspot's background photo, image box, testimonial-slider avatar): never use `type: 'image'` for a slot that might get published unpicked — use a plain `<div>` placeholder (solid background color, no `src` at all) plus a button-Trait/toolbar command that calls `wp.media()` and REPLACES the placeholder with a real `<img>` (`parent.append({tagName:'img', attributes:{...}}, {at: placeholder.index()})` then `placeholder.remove()`) only once a real image is actually chosen — same shape as gallery/slider's existing multi-image pickers, just single-image. The pre-existing atomic "Изображение" Element (elements.js) has this same latent risk if published without ever picking an image — not fixed here (out of scope for this task), but the same fix pattern applies if it's ever hit in practice.
- `data-*` attributes that exist ONLY so an in-editor command can `component.find('[data-wpgjb-x]')` a child (hotspot's `data-wpgjb-hotspot-bg`, image box's `data-wpgjb-imagebox-img`, testimonial slider's `data-wpgjb-testimonial-avatar`, the `data-wpgjb-slider-variant` type-disambiguation marker) do NOT need a `Sanitizer::richness_page()` whitelist entry — nothing on the published frontend reads them, and re-editing always goes through `project_data` (GrapesJS's own serialization, which keeps the real `type`/model state regardless of what survives in the HTML), never a re-parse of published HTML. Same bucket as the pre-existing `data-wpgjb-spacer`/`data-wpgjb-icon-el`/`data-wpgjb-button` markers noted above — only whitelist a `data-*` attribute here if a *runtime script* or *PageAssets marker check* actually reads it back.

### Test-writing gotcha (from debugging a "broken" feature that wasn't)
When a REST-round-trip-dependent UI action (e.g. a Modal's save button) appears to do nothing in a Puppeteer test, check wait time before assuming a code bug — this WP Playground WASM-PHP instance can occasionally take &gt;1s to respond under load from a long dev session's accumulated processes. A `window.__DEBUG_*` trace showing the promise never settling (not rejecting, not resolving) within the test's wait window was the tell; the real fix was a longer `setTimeout` in the test, not any source change.

## PHP (includes/)

### Core/
- **Plugin.php** — `boot()`, wires up all module `register_hooks()`/`register_routes()` calls.
- **Activation.php** — `activate()`/`deactivate()`.
- **Diagnostics.php** — `log()`/`recent()`, simple local log store.

### Storage/ — dual-storage document persistence, UI-agnostic
- **DocumentsTable.php** — `{prefix}wpb_documents` schema (`table_name()`, `maybe_upgrade()`).
- **DocumentRepository.php** — CRUD + optimistic locking: `get_for_edit`, `save_draft` (checks `expected_doc_version`, 409 on mismatch), `publish`.
- **MigrationRunner.php** — `register(Migration)`, `migrate(project_data, from, to)`.
- **RetentionPolicy.php** — scheduled revision pruning (`prune_all`/`prune_document`, configurable `keep_count`).
- **Cleanup.php** — `full_cleanup()` for uninstall.

### Blocks/ — manifest-driven "Блоки" catalog, server-side data contract
- **ManifestSchema.php** — constants: `SECTION_TYPES`, `SLOT_TYPES`, required keys.
- **ManifestValidator.php** — `validate_manifest()`, `validate_values()` (shared by editor bootstrap, insert-block REST, assemble-page REST, and `TemplateLibrary`'s own validation).
- **BlockLibrary.php** — loads/validates `blocks-library/{id}/{manifest.json,markup.html,style.css}`; `all_visible_to_current_user()` filters by `wpgjb_insert_raw_code` capability.
- **TemplateLibrary.php** — mirrors BlockLibrary for `templates-library/{id}/manifest.json` (`{title, blocks:[{block_id,slots}]}`, same shape `/pages/assemble` accepts). Validates every member block via `ManifestValidator::validate_values`; drops (doesn't fail the whole catalog for) an invalid template.
- **BlockRenderer.php** — `render_component()`, server-side mirror of the client's markup+values→component codec (used by REST insert-block/assemble-page, not by the static-HTML publish path, which is always client-generated).
- **BlockSnapshotter.php** — regression snapshot system: `generate_snapshot`/`write_baseline`/`compare_all` against `tests/php/Snapshots/blocks/*.json`.
- **BlockFaultTolerance.php** — `quarantine()`, degrades a page referencing a since-removed block id instead of failing the whole page.
- **ChecklistLinter.php** — `lint_block`/`lint_all`, section-6-checklist style lint (hardcoded hex/font-family, missing alt, etc.).

### AI/ — page generation via Timeweb Cloud AI Agent API (`docs/TWC AI API.md`)
Built entirely on top of the pre-existing "AI-assembly-ready" contract (spec §9, principle П7: "the AI module assembles pages with the same operations as a human") — no new page-assembly logic exists here, only prompt construction + one existing REST endpoint call.
- **TwcAiClient.php** — first outbound-HTTP integration in this plugin. `chat_completion(array $messages): string|WP_Error` wraps `wp_remote_post()` to TWC's OpenAI-compatible `/v1/chat/completions` (Bearer + `x-proxy-source` headers, `stream:false`, 60s timeout). Credentials via `get_option(OPTION_AGENT_ACCESS_ID/OPTION_BEARER_TOKEN)`, set on `SecurityPage`. The API has no JSON-schema/structured-output mode (`response_format.type` only documents `text`) — strict JSON is enforced entirely via prompt instructions + our own parsing, not by the API.
- **PageGenerator.php** — `generate(array $brief): array`. Builds a system prompt embedding `BlockLibrary::export_ai_digest()` + slot-grammar rules mirroring `ManifestValidator`, calls `TwcAiClient`, parses the reply as `{title, blocks:[{block_id,slots}]}`, and dispatches it through an **internal REST call** (`rest_do_request()` against `POST /wpgjb/v1/pages/assemble`) — reuses `ManifestValidator`/`ProjectDataSanitizer`/`BlockRenderer`/`DocumentRepository` unchanged. On a 422 (either shape — see below) or unparseable JSON, retries once with the exact errors fed back to the model (`MAX_ATTEMPTS = 2` total). AI never fills `image` slots (no media-library access) and uses `#` for `link` slots.
- **AiRequestLogger.php** — file-based (not `Diagnostics::log()`'s WP-options ring buffer) request/response log at `logs/ai-requests.log` in the plugin directory, by explicit user request for debuggability. `.htaccess`+`index.php` auto-created alongside it (plugin lives under `wp-content/plugins/`, theoretically web-reachable otherwise). Bearer token is never written to this file. **Gotcha found by a real run, not docs**: `file_put_contents(..., FILE_APPEND | LOCK_EX)` silently wrote nothing (0-byte file, no error) against this dev sandbox's virtual filesystem (WP Playground) — dropped `LOCK_EX` (acceptable here: single admin triggers one generation at a time, no real concurrent-write scenario).
- `assemble_page()` (in `BlocksCatalogController.php`) signals its two 422 cases differently — empty `blocks` array returns a `WP_Error` (REST-framework-wrapped as `{code, message, data:{status}}`), invalid sequence items return a direct `WP_REST_Response{status:'invalid', errors:[{index,block_id,errors}]}`. `PageGenerator::extract_errors()` handles both shapes when building the self-correction prompt.
- **BlockLibrary::export_ai_digest()** (in `Blocks/BlockLibrary.php`, not `AI/`) — compact per-block projection (`id`/`section_type`/`purpose`/`slots`, no markup/style/context) built on top of the already-capability-filtered `all_visible_to_current_user()`, so raw-html-gated blocks never leak into a prompt for a user without that capability.
- **AiController.php** (in `Rest/`) — `POST /wpgjb/v1/ai/generate-page`, thin, delegates to `PageGenerator::generate()`. Reuses `wpgjb_edit_pages`+`publish_pages` (same as `assemble_page` itself requires) — no new capability. Simple per-user `transient`-based throttle (20s) — debounce, not real quota/cost control (explicitly out of scope, see the phase's plan).
- UI entry point lives in `Admin/EditorPage.php` (`render_ai_generate_button()`/`maybe_enqueue_ai_admin_assets()`) + `assets/admin/index.js` (finally implemented — was a Phase-0 placeholder ever since, nothing else has used the `admin` webpack entry until now): a button next to "Добавить в конструкторе" on the Pages list opens a vanilla-JS modal (niche/page_type/tone/details), POSTs to the endpoint, redirects to the returned `editor_url` on success — same as `assemble_page`'s own response shape already provides.

### Sanitize/ — two-contour sanitizer, the ONLY sanitization implementation (never add a second)
- **Sanitizer.php** — shared whitelist primitives: `richness_none/rich_text/image/page()`, `sanitize_html()`, `sanitize_css()`. `richness_page()` (contour-2/publish whitelist) enumerates every exact `data-*` attribute that must survive publish (no wildcard support in wp_kses) — see the interactive-Elements gotcha above — plus tags/attrs for form-input Elements (`input`/`textarea`/`select`/`option`), `details`/`summary` (accordion), `svg`/`path`/`polyline`/`circle` (icon Element + line/pie chart shapes), `table`/`thead`/`tbody`/`tr`/`th`/`td` (data table), `blockquote`/`cite` (quote Element), `pre`/`code` (code block), and `aria-sort`/`aria-expanded`/`aria-selected`/`hidden`/`scope` (table sorting, tabs, hotspot).
- **ProjectDataSanitizer.php** — `sanitize()` walks the whole `project_data` component tree (generic-node path covers any component without `data-wpb-block`, e.g. free Elements); `sanitize_values()` for slot values specifically.
- **Contour1EditorSave.php** — `sanitize_document()`, runs on every save.
- **Contour2PrePublish.php** — `sanitize_document()`, runs again independently right before writing `post_content` (catches anything that bypassed contour 1 via direct REST).

### Render/ — publish pipeline, consumes client-generated HTML/CSS
- **Publisher.php** — `publish()`: contour-2 sanitize → `wp_update_post()` → new publish-revision row. Also resolves+stores `PageAssets::META_JS` (block JS) and `PageAssets::META_INTERACTIVE` (interactive Elements flag) as postmeta, once per publish. Wraps the final HTML in `<div class="alignfull">` when `PageTemplates::is_full_width_on_block_theme()` is true — see PageTemplates.php below.
- **PageTemplates.php** — fixes a real reported bug: a live AI-generated page (classic Elementor-compatible theme "neon-style") rendered narrow and visually disconnected from the theme, because the theme's own default page template assumes a classic narrow blog-post reading column, which a page-builder-generated full-section page doesn't fit. Registers two virtual page templates via `theme_page_templates` (they show up in the existing page-settings "Макет страницы" dropdown automatically, no client-side change needed) — `wpgjb-canvas.php` ("Конструктор: Пустой холст", zero theme markup at all, works identically on any theme via `template_include`) and `wpgjb-full-width.php` ("Конструктор: Во всю ширину", real theme header/footer but an unconstrained content area). `DEFAULT_TEMPLATE` (= the full-width one) is assigned via `update_post_meta( $post_id, '_wp_page_template', ... )` right after `wp_insert_post()` in both `EditorPage::handle_new_page()` and `BlocksCatalogController::assemble_page()` — every new constructor page (human-created or AI-generated) starts with it, no manual template pick needed.
- Classic vs. block theme branch for the full-width template: `template_include` only overrides for CLASSIC themes (`includes/Render/templates/wpgjb-full-width.php` calls `get_header()`/`get_footer()` directly, with content in its own unconstrained `<main>`, bypassing the theme's normal narrow content wrapper entirely). Block themes have no `header.php`/`footer.php` for those calls to find — `get_header()`/`get_footer()` silently no-op on them — so for block themes `maybe_override_template()` deliberately does NOT override at all; the theme's native FSE page template renders normally (real header/content/footer), and `Publisher::is_full_width_on_block_theme()` + the `alignfull` wrap is the only intervention needed: WP core's own generated stylesheet always includes a `.is-layout-constrained > .alignfull` escape-hatch rule (confirmed by reading an actual theme's `wp_get_global_stylesheet()` output, not assumed), so a single wrapping `<div class="alignfull">` around the published HTML is enough to escape whatever narrow `content-size` that specific block theme's `theme.json` declares — no theme-specific CSS knowledge required. Verified empirically on both a real block theme (Twenty Twenty-Five, which already defaults its own page template's post-content to `alignfull` — the wrap is a harmless no-op there) and a throwaway classic theme with a hard `max-width` on its content div (the wrap/bypass is load-bearing there).
- Testing gotcha found while verifying this: don't grep the full rendered HTML for a class name like `wp-site-blocks`/`masthead` to detect "is theme chrome present" — block themes' `wp_head()` always prints the theme's global-styles `<style>` block, which MENTIONS selectors like `.wp-site-blocks{...}` as plain CSS text regardless of which template rendered the page (canvas templates that never output the class still have it appear in a stylesheet). Check the actual `<body>` content for real header/nav markup, not the `<head>`'s CSS text.
- **FrontendRenderer.php** — `resolve_dynamic_tags()` (batch tag resolution on serve), `output_page_css()`/`enqueue_page_assets()` (reads both `PageAssets::META_JS` and `META_INTERACTIVE` postmeta; enqueues `wpgjb-elements-runtime.js`, dependent on `wpgjb-runtime.js`, only when the interactive flag is set — independent of whether any block declared its own JS).
- **PageAssets.php** — `collect_block_ids()`/`resolve_js_assets()` from `project_data` (manifest Blocks path), feeds the per-page asset enqueue. `has_interactive_elements()` — parallel non-manifest scan for `INTERACTIVE_MARKER_ATTRIBUTES` (`data-wpgjb-{countdown,counter,gallery,slider,chart,datatable,tabs,flipcard,hotspot,codeblock}`), free "Элементы" have no `data-wpb-block` so they need their own tree walk. Testimonial slider and the static Elements (icon box/image box/quote) intentionally do NOT add their own marker here — testimonial slider reuses `data-wpgjb-slider` (already in the list), and the static ones have no runtime JS to gate.

### DynamicTags/

- **BatchResolver.php** — `resolve(html, post)`, ≤3-query resolution regardless of tag-instance count (groups by data source, not occurrence count).

- **PublishController.php** — `/documents/{post_id}/{type}/publish`.
- **BlocksCatalogController.php** — `/blocks`, `/blocks/{id}`, `/documents/{post_id}/{type}/insert-block`, `/pages/assemble` (creates a NEW page from a block sequence — does NOT insert into an already-open document, that's out of scope per its own ADR), `/templates` (list, mirrors `wpgjbEditorData.templates` for REST-only/future-AI-module consumers).
- **SitePartsController.php** — `/parts` list/create, display-conditions get/set.
- **ThemeStylesController.php** — `/theme/global-styles.css`, `/theme/tokens`, `/theme/chrome` (`get_chrome()` — real rendered header/footer HTML for block themes, via `do_blocks()` on a synthetic `core/template-part` block; empty/`isBlockTheme:false` for classic themes), `/theme/custom-background.css` (`get_custom_background_css()` — captures whatever `wp-head-callback` the active theme registered for `add_theme_support('custom-background')`, see style-inheritance gotchas above), `/blocks/style.css` (long-cached platform block CSS), `blocks_style_version()` cache-buster.
- **Dev*Controller.php** (DevPhase1-7, DevSpike, DevSpike2, DevSpike4, DevSecondTier, DevSnapshots, DevChecklist) — `WP_DEBUG`-gated, `manage_options`-gated regression-verification endpoints, intentionally kept (not dead code). `DevSecondTierController`'s docblock is stale (describes sidebars/ACF-tags but actually runs the page-assembly test suite — known inconsistency, not fixed yet).

### Admin/
- **EditorPage.php** — the full-screen editor host. `render_page()` emits one empty `#wpgjb-editor-root` div (zero DOM assumptions — all UI structure is client-built, confirmed safe for the preset-webpage UI swap). `maybe_enqueue()` localizes `window.wpgjbEditorData` (postId/type/postTitle/postSlug/postStatus/featuredMedia/featuredMediaUrl/previewUrl/adminUrl/restRoot/restNonce/namespace/canvasStyles[]/bodyClass/document/blocks/templates/pageTemplates/lock/uiLang — `pageTemplates` only populated when `type==='page'`). `compute_body_class()` — see style-inheritance gotchas above; deliberately hand-computed (no `WP_Query`) to keep this a low-risk addition after the classic-theme-chrome revert. `export_blocks_for_js()`/`export_templates_for_js()`/`export_page_templates_for_js()` feed `blocks`/`templates`/`pageTemplates` (the latter via core `get_page_templates()`, works for both classic and block themes). `resolve_ui_lang()`/`UI_LOCALES` — reads `?wpgjb_lang=ru|en`, defaults from `determine_locale()`'s natural resolution; when it differs, hooks `'pre_determine_locale'`+`'locale'` (NOT `switch_to_locale()` — see language-switcher gotcha above) so `wp_set_script_translations('wpgjb-editor', 'wp-gj-builder', WPGJB_PLUGIN_DIR . 'languages')` and PHP-side `__()` both pick up the requested locale for this one request.
- **SitePartsPage.php** — parts list/create/conditions admin UI.
- **SecurityPage.php** — capability matrix, raw-code flag, retention policy, full-cleanup admin UI, and (new) AI credentials section (`render_ai_settings_section()`/`handle_save_ai_settings()`) — Agent Access ID + Bearer Token, `TwcAiClient::OPTION_*`. The token field never echoes the saved secret back into the form value (only a "already set" placeholder) — an empty submit leaves the stored token untouched, doesn't clear it.

## Data files
- **assets/runtime/wpgjb-runtime.js** — tiny shared `WPGJB.onDelegate(selector, event, handler)` helper, loaded raw (no webpack/babel) via `wp_register_script` — plain ES5 `var`, no build step, deliberately (matches `lint-js`'s pre-existing baseline for this directory).
- **assets/runtime/wpgjb-elements-runtime.js** — frontend behavior for all interactive Elements. Original 4: countdown ticking via `setInterval`, counter count-up via `requestAnimationFrame`+`IntersectionObserver`, gallery lightbox via `WPGJB.onDelegate`, slider prev/next/dot via `WPGJB.onDelegate`+`translateX` (also drives the testimonial-slider variant unchanged, see elements-content.js). Added for the infographic/content elements: `initCharts()` (bar/line/pie entrance animation, IntersectionObserver + inline-style toggles, `prefersReducedMotion()` skips it entirely), `initDataTables()` (click-`<th>` sort, numeric-aware compare, `aria-sort`, brief fade-reorder), `initTabs()` (nav click → panel `hidden`/`aria-selected`), `initFlipCards()` (direct per-element `mouseenter`/`mouseleave`, NOT `onDelegate` — see elements-content.js gotcha), `initHotspots()` (dot click → tooltip `display`/`aria-expanded`), `initCodeCopyButtons()` (clipboard write + pre-baked "copied" label swap). Enqueued only when `PageAssets::META_INTERACTIVE` is set — see FrontendRenderer.php. No companion CSS file was added for any of this: every animation is a plain inline-style mutation (`transform`/`opacity`/`stroke-dashoffset`), since `@keyframes`/`:hover`/media-query rules are the only things inline style genuinely can't express, and none of the new elements ended up needing those.
- **blocks-library/{id}/{manifest.json,markup.html,style.css}** — the platform blocks. Repeatable-slot wrapper elements (`[data-slot-repeat]`/`[data-slot-item]`) carry `data-gjs-droppable="false"` — without it they parse as GrapesJS's generic `droppable:true` default and become valid drop targets for free Elements, silently corrupting the manifest slot structure. Card/tariff/review/feature-style blocks (pricing-comparison, test-pricing, test-features, test-testimonials, team-compact, test-team, services-grid, test-faq, stats-counters) default to 3 `[data-slot-item]`s; strip-style blocks (test-gallery, timeline, test-steps, logos-strip) default to 6 — all with distinct per-item placeholder content (not N identical copies) and real visual polish (box-shadow/hover/grid, using the existing `var( --wp--preset--color--X, #fallback )` convention every block already followed). `test-hero` also has an array slot but its manifest `purpose` field states it's a `ManifestValidator` test fixture, not a real content block — deliberately left untouched.
- **languages/wp-gj-builder-{locale}-{handle}.json** — Jed-format JS translation catalogs for `wp_set_script_translations()` (WP 7.x filename convention — see gotcha above). Only `en_US` exists (hand-authored, ~150 strings); `ru_RU` needs none since the source `t()`/`__()` msgids already ARE the Russian display strings. No PHP-side `.mo` file exists for the plugin's own domain — the rare PHP-side strings (e.g. the `wp_die()` "post not found" message) stay Russian even when `?wpgjb_lang=en` is active; only the JS-side editor UI (the vast majority of user-facing text) actually switches.
- **templates-library/{id}/manifest.json** — the "Страницы" catalog, `{title, blocks:[{block_id, slots}]}`. Currently one demo entry (`landing-page`).
- **logs/ai-requests.log** — runtime-generated (not committed, `*.log` is repo-gitignored), created on first AI generation call by `AiRequestLogger`. `.htaccess`/`index.php` in the same dir are also auto-created, harmless to commit if ever tracked but not currently.
