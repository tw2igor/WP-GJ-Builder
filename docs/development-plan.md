# WP-Builder (WP-GJ-Builder): план разработки

## Контекст

`WP-GJ-Builder` — новый плагин-конструктор страниц для WordPress на основе GrapesJS, встроенный в хостинг для партнёров-веб-студий и их клиентов (модель Tilda: готовые смысловые блоки, наследующие стиль темы, минимум решений на плечах пользователя). Полная продуктовая спецификация — `docs/wp-builder-dev-spec.md` (15 разделов: принципы, модель данных, структура админки, редактор, библиотека блоков, части сайта, динамические данные, манифесты блоков, безопасность, производительность, надёжность, MVP-состав, критерии приёмки). Обоснование выбора GrapesJS как фундамента (против Elementor/Gutenberg/Bricks/Breakdance/Oxygen) уже зафиксировано пользователем в `docs/builder-analysis.md` — решение принято и не пересматривается; этот документ также фиксирует конкретные архитектурные обязательства (см. ниже), вытекающие из выбора GrapesJS.

Проект полностью greenfield: в папке `WP-GJ-Builder` пока нет кода. В остальном репозитории (`C:\AI_Dev\WP\WP-Plugin`) нет ни Node-, ни PHP-тулинга — единственный прецедент, `utm-link-constructor`, простой плагин без сборки, не масштабируется на задачи этого проекта (своя таблица БД, REST API, сборка GrapesJS-бандла).

Цель этого документа — дорожная карта до MVP (раздел 14 спеки) и конкретная техническая архитектура для самых рискованных узлов интеграции GrapesJS+WordPress. По решению пользователя, сразу после утверждения плана начинается реализация Phase 0 в этой же сессии.

**Обязательства из `builder-analysis.md`, которые являются жёсткими архитектурными ограничениями (не темами для обсуждения):**
1. JSON project data (`editor.getProjectData()`/`loadProjectData()`) — единственный источник правды для повторного редактирования. GrapesJS **никогда** не восстанавливает страницу из HTML (официально задокументированное ограничение Component Recognition — теряются типы кастомных компонентов, атрибуты, traits).
2. Финальный HTML/CSS (`editor.getHtml()`/`getCss()`) генерируется **на клиенте** в момент публикации и один раз пишется в `post_content` — не пересобирается на каждый визит (иначе воспроизводится архитектурная проблема Elementor: PHP-парсинг JSON на каждый запрос).
3. Серверная санитизация HTML/CSS при каждом сохранении — обязательна и небазовая: у GrapesJS были реальные CVE на XSS в собственном UI (`SNYK-JS-GRAPESJS-2935960`, `SNYK-JS-GRAPESJS-2342412`, issue #3082 — импорт HTML с `onload`, issue #4076 — инъекция через атрибуты компонента). В мультитенантном контексте (тысячи сайтов хостинга) это не гипотетический риск.
4. На первом этапе доступ к произвольному HTML/JS в редакторе — только за явным флагом прав, выключен по умолчанию; готовые блоки — единственный режим для основной массы пользователей, пока не построен полноценный контур санитизации.
5. Наследование стиля темы: визуальное наследование (реальный CSS темы в canvas iframe) работает "из коробки"; понимание дизайн-токенов темы редактором — конкретная решаемая задача через `grapesjs-css-variables` + разбор `theme.json`, не автоматика.

## Репозиторий и файловая структура

Новый билд-тулинг для этого workspace (оправдан масштабом — своя таблица, REST, сборка редактора):

```
WP-GJ-Builder/
  wp-gj-builder.php           # bootstrap, header, activation/uninstall hooks
  uninstall.php               # стандартный WP uninstall entry point
  composer.json                # PSR-4 автозагрузка + phpcs/phpstan
  package.json                 # JS-сборка (@wordpress/scripts)
  includes/
    Core/                      # bootstrap, activation, регистрация capability
    Storage/                   # wpb_documents: схема, CRUD, миграции
      Migrations/
    Rest/                      # REST-контроллеры: каталог, save/load документа, dynamic-tags registry
    Render/                    # publish pipeline: project_json -> статический HTML, батч-резолвер тегов
    Sanitize/                  # общий whitelist-санитайзер (вызывается ОБОИМИ контурами)
    DynamicTags/                # реестр тегов, батч-резолвер
    Blocks/                    # загрузчик/валидатор манифеста блока
    SiteParts/                 # шапка/подвал/сайдбар, display conditions, classic/block theme injection
    Security/                  # матрица прав, audit log
    Admin/                     # PHP-сторона админ-экранов (Части сайта, Настройки)
    Cli/                       # (опционально) wp-cli: rebuild-all-pages, diagnostics
  assets/
    editor/                    # исходники редактора (GrapesJS + grapesjs-css-variables), только для экрана редактора
    admin/                     # React/@wordpress/components админ-экраны
    build/                     # скомпилированный вывод
  blocks-library/
    {block-id}/manifest.json, markup, style.css, assets/
  templates-library/           # готовые страницы (вкладка "Страницы")
  languages/                   # ru+en с самого начала
  tests/{php,js}
  docs/adr/                    # последующие ADR (builder-analysis.md уже есть, будущие решения — сюда)
```

`includes/Sanitize` — намеренно единственный: оба контура безопасности (раздел 10 спеки) обязаны вызывать одну и ту же логику, различаться должна только точка вызова, не правила.

## Технические решения по самым рискованным узлам

### 1. Сборка редактора: `@wordpress/scripts` (webpack), не Vite/esbuild
Обоснование: раздел 13 спеки требует, чтобы вся "хром"-часть редактора (панели, меню, уведомления) следовала визуальному языку `@wordpress/components`. `@wordpress/dependency-extraction-webpack-plugin` (часть `@wordpress/scripts`) автоматически исключает `@wordpress/*`-импорты из бандла, подключая их как уже загруженные ядром `wp.*`-глобалы, и генерирует `{handle}.asset.php` с точным списком зависимостей и версией по хэшу контента — это напрямую скармливается в `wp_register_script($handle, $url, $deps, $ver)` без ручной логики. Аналогов у Vite/esbuild для WP нет. Два отдельных входа сборки: editor-бандл (тяжёлый, только на экране редактора) и мелкие runtime-скрипты фронтенд-блоков (слайдеры/табы — по одному на блок, для точечного enqueue).

**Жёсткое требование раздела 11 ("ассеты редактора не должны попадать ни на фронтенд, ни в остальную админку") — это вопрос PHP-регистрации, не сборки**: `admin_enqueue_scripts`-колбэк обязан проверять `get_current_screen()->id === $wpb_editor_hook_suffix` перед `wp_enqueue_script/style` — никогда не вешать enqueue без проверки текущего экрана.

### 2. Схема `{prefix}_wpb_documents`
```
id, post_id, type (page|header|footer|sidebar|block|template),
revision_type ('draft'|'publish'), is_current (bool, ровно одна строка на post_id+type+revision_type),
project_json LONGTEXT, schema_version SMALLINT, doc_version INT (монотонный, для optimistic locking),
updated_at, updated_by
KEY (post_id, type, revision_type, is_current); KEY (post_id, updated_at); KEY (type)
```
Без FK (стандартная практика для кастомных таблиц WP — dbDelta/репликация плохо работают с FK); очистка при удалении поста — через хук `before_delete_post`. Конфликты сохранения: клиент шлёт загруженный `doc_version`, сервер отклоняет несовпадение 409-м с текущей строкой (запускает UI сравнения, раздел 5.5). Присутствие второго редактора — штатный WP-механизм (`_edit_lock`/`_edit_last` + Heartbeat), не своя таблица блокировок.

### 3. Публикация и статический HTML
Клиент вызывает `editor.getHtml()`/`getCss()` в момент клика "Опубликовать" и шлёт `{project_json, html, css}` на `POST /wpb/v1/documents/{id}/publish`. Сервер: капабилити+nonce → **контур-2 санитизация** независимо от контура-1 → нормализация динамических тегов к каноничному плейсхолдеру (защита от прямого forge-POST) → `wp_update_post()` (не сырой SQL — чтобы штатные хуки `save_post`/`transition_post_status`/`clean_post_cache` сработали для кэш-плагинов хостинга автоматически) → новая строка `revision_type='publish'`.

### 4. Динамические теги (раздел 8)
Формат плейсхолдера: `{{wpb:tag_id;param=value}}`, хранится буквально в `post_content`. Резолвер использует `WP_HTML_Tag_Processor` (без нового парсера) — один проход, экранирование по позиции в дереве (text/attr/url), а не по доверию к автору тега. 1–3 запроса **вне зависимости от числа тегов**: группировка идёт по источникам данных (уже загруженный `$post`, один `get_userdata()` только если есть тег автора, один `get_post_meta($post->ID)` только если есть кастомное поле) — не по количеству вхождений тегов.

### 5. Манифест блока → компонент GrapesJS
Один манифест (JSON: id, section_type, purpose, slots[] с типами/ограничениями, context, requirements, style_whitelist) обслуживает три независимых потребителя одной и той же функцией валидации/вставки: (1) форма "Контента" в редакторе генерируется итерацией по `slots[]`, (2) валидация человеческого ввода при сохранении, (3) REST-эндпоинт вставки блока для будущего AI-модуля. Один компонент-тип GrapesJS на `section_type`, `stylable` берётся из `style_whitelist` — это прямой механизм, реализующий П2/П3 (не свободная стилизация). Изображения — через `wp.media`, атрибут ID (не URL), чтобы бесплатно получить `srcset`/lazy из WP core.

### 6. Наследование стиля темы
Токены — через `wp_get_global_settings()` (блочные темы) с фолбэком на `get_theme_support('editor-color-palette'/'editor-font-sizes')` для классических тем. В canvas iframe грузится **реальный** сгенерированный stylesheet темы (`wp_get_global_stylesheet()`, уже содержит `--wp--preset--*` custom properties) — не переизобретённая копия токенов, чтобы гарантировать байт-в-байт совпадение с фронтендом. `grapesjs-css-variables` конфигурируется на эти `--wp--preset--*` имена, токены — первыми в Style Manager, произвольное значение — вторым. "Изменено от темы" — структурно (значение — `var(...)` = унаследовано, литерал = переопределено), не отдельный флаг для поддержки.

### 7. Санитизация (раздел 10)
Контур 1 (при каждом сохранении): `wp_kses()` по whitelist тегов на слот (большинство слотов — ноль тегов); `<script>`/`on*` отклоняются самим фактом отсутствия в allowed-списке. Раздел "Вставка кода" и Zero-режим обходят `wp_kses` только за капабилити `wpb_insert_raw_code` (через `map_meta_cap`), каждый такой обход пишется в `{prefix}_wpb_audit_log`. **CSS-санитайзер — единственный настоящий пробел в ядре WP** (own core sanitizer for CSS doesn't exist) — строить на `sabberworm/php-css-parser` (Composer), не ручной токенизацией: whitelist свойств (per-block или общий для Zero-режима), явный запрет `expression()`/`-moz-binding`/`behavior`/`@import`, проверка синтаксиса значения по типу свойства. Контур 2 (перед публикацией, независимо от контура 1): та же санитайзер-функция повторно проходит по финальному собранному HTML/CSS — обязателен даже если запрос обошёл обычный UI редактора (прямой REST-вызов), это единственное, что удовлетворяет критерию приёмки №7 из раздела 15.

### 8. Производительность (раздел 11)
Манифест блока декларирует `requirements.assets`; при публикации разница блоков на странице резолвится один раз в `_wpb_page_assets` (postmeta), не пересчитывается на каждый запрос — `wp_enqueue_scripts` просто читает список. Общий "рантайм конструктора" подключается как `dep` любого интерактивного блока — раздел 11 требование "только если на странице есть интерактивные блоки" получается бесплатно через граф зависимостей WP, без ручной проверки. CSS — двумя файлами: (1) общий, долгокэшируемый "платформенные блоки" (переиспользуется между страницами сайта), (2) маленький per-page файл только с инстанс-переопределениями (раздел 6 "изменено от темы").

## Дорожная карта (8 фаз)

Каждая фаза самодостаточна (можно приостановиться между любыми двумя); фазы 3 и 4 можно вести параллельно двум инженерам после выполнения exit-критериев фазы 1.

### Phase 0 — Основы и риск-спайки
- Спайк 1 (наследование стиля темы, classic+block): реальная тема в iframe canvas + theme.json токены через `grapesjs-css-variables`, визуальное сравнение.
- Спайк 2 (батч-резолвинг тегов, раздел 8 п.2): прототип реестра тегов + резолвера на ≤3 запроса вне зависимости от числа тегов.
- Спайк 3 (dual-storage + миграции, разделы 3, 12): схема `wpb_documents` + мини цепочка миграций на фейковый `schema_version`, доказать fallback "заморозка на последнем валидном HTML" при повреждении JSON.
- Спайк 4 (санитизация, раздел 10): изолированный whitelist-санитайзер + **адверсариальный корпус тестов на основе реальных GrapesJS CVE** (`SNYK-JS-GRAPESJS-2935960`, `2342412`, issue #3082 `onload`-импорт, issue #4076 инъекция через атрибуты) — прогнать через оба контура, подтвердить что контур 2 ловит обход контура 1.
- Скелет плагина: `wp-gj-builder.php`, `composer.json`+PSR-4, `package.json`+`@wordpress/scripts`, activation/uninstall-заглушки, регистрация капабилити-заглушки.
- CI-lint база: phpcs (WordPress-Extra), phpstan baseline, JS lint.
- **Exit:** все 4 спайка задокументированы вердиктом; плагин активируется/деактивируется без фич; сборка даёт плейсхолдер-бандл.

### Phase 1 — Dual Storage и модель документа
- Таблица `wpb_documents` (схема выше) + activation-time создание + dbDelta upgrade path.
- CRUD-слой + REST save/load `project_json` (капабилити-гейт, без рендера).
- Миграционная цепочка (из спайка 3) на реальном `schema_version`.
- Схема манифеста блока + валидатор (стабилизировать форму раздела 9 сейчас — REST-каталог и будущий AI зависят от этого контракта) — сознательно переспецифицировать необязательные поля (`context`/`requirements`) уже сейчас.
- **Exit:** создание/сохранение/загрузка `project_json` через REST с фейковым апгрейдом `schema_version` мигрирует корректно на чтении; повреждённый документ замораживается, а не падает.

### Phase 2 — Publish pipeline: статический HTML + динамические теги
- Рендер: `project_json` → HTML/CSS в `post_content` при публикации (клиентская генерация, см. решение №3 выше).
- Реестр динамических тегов (группы сайт/страница/автор, первая очередь раздела 8) + батч-резолвер (из спайка 2), подключён к реальному publish/serve пути.
- Контур-2 санитизация непосредственно перед записью `post_content` (из спайка 4), общий код с заглушкой контура-1.
- Хуки инвалидации кэша при публикации (стандартные хуки WP).
- **Exit:** вручную собранный `project_json` с 2-3 динамическими тегами публикуется на реальную WP-страницу; обычный посетитель не грузит ни одного ассета редактора и видит корректно резолвленные значения; батч-резолвер доказуемо укладывается в ≤3 запроса независимо от числа тегов (проверено профилированием).

### Phase 3 — Editor MVP (только блочный режим, без Zero-режима)
- Полноэкранная оболочка редактора (раздел 5.1): заголовок, переключатель устройств, undo/redo, индикатор сохранения, просмотр, публикация.
- Canvas iframe с реальным CSS активной темы + токенами theme.json (по вердикту спайка 1).
- Панель выделенного блока (Контент/Настройки, вверх/вниз/дублировать/скрыть/удалить).
- Минимальная библиотека (3-5 тестовых блоков) для проверки потока добавления через "+".
- Автосохранение (по изменениям, троттлинг 15-30с), одна обновляемая ревизия на сессию + постоянная на публикацию, Heartbeat-блокировка, optimistic-locking конфликт-промпт.
- Контур-1 санитизация подключена к каждому сохранению.
- **Exit:** пользователь открывает страницу, добавляет блок, редактирует контент, меняет настройку раскладки, публикует — фронтенд идентичен канвасу. Две сессии редактирования одной страницы корректно триггерят lock/conflict UX.

### Phase 4 — Каталог библиотеки блоков и REST API (может идти параллельно Phase 3)
- UI библиотеки: категории по смыслу (принцип П1), живые превью в стиле темы, переключатель Платформа/Мои блоки, поиск, фильтр "использованные на сайте".
- REST-каталог (раздел 9): список блоков+манифесты с фильтрами, блок целиком, вставка блока в страницу, "собрать страницу из последовательности" (read-only/insert — по объёму MVP раздела 14).
- Чек-лист выпуска блока (раздел 6, 6 пунктов) как lint-проверка (жёсткий hex/font-family, отсутствие alt).
- Первая партия блоков (10-15, не сразу 60-100) — достаточно, чтобы прогнать через каждое поле манифеста.
- **Exit:** внешний REST-клиент (критерий приёмки №9, раздел 15) получает список блоков с манифестами и вставляет блок с заполненными слотами в черновик, не зная о GrapesJS. Партия блоков проверена на 3 контрастных темах (раздел 6 п.3, раздел 15 п.3).

### Phase 5 — Zero-режим, части сайта, матрица безопасности
- Zero-режим: полный Style/Layer/Trait Manager, хлебные крошки, токены темы первыми, метка "кастомизировано" — за флагом прав, выключен по умолчанию.
- Части сайта: контекст редактирования шапки/подвала, движок display conditions (специфичность в духе Elementor), classic-theme через перехват `get_header`/`get_footer` с честной индикацией поддержки, block-theme через замену template parts.
- Безопасность: экран матрицы ролей→прав, блок "Вставка кода" за явным флагом, audit log, whitelist тегов в настройках.
- Экраны "Части сайта"/"Настройки" (React/@wordpress/components).
- **Exit:** публикация изменённой шапки инвалидирует кэш всех затронутых страниц с явным сообщением пользователю; непривилегированный пользователь физически не может достичь Zero-режима/вставки кода, и прямой REST-запрос с `<script>` отклоняется контуром-1 (и был бы пойман контуром-2 при обходе).

### Phase 6 — Производительность, надёжность, очистка
- Аудит загрузки ассетов: редактор нигде вне своего экрана; блочный CSS/JS — только при реальном присутствии блока.
- Изображения: srcset/lazy через WP core, `fetchpriority=high` без lazy для первого экрана, WebP-фолбэк.
- Плановая очистка ревизий + настраиваемая политика хранения.
- Регрессионные снапшоты (раздел 12): JSON-снимки версий блоков, прогон при апдейтах библиотеки.
- Отказоустойчивость на масштабе: несколько повреждённых блоков на одной странице — страница всё равно публикуется.
- Полный uninstall: таблицы/опции/cron/transients за явным чекбоксом + отдельное "полная очистка сейчас"; HTML остаётся читаемым после удаления плагина.
- Замер по метрикам раздела 11 (TTFB ±15%, ≤60КБ CSS/JS конструктора, PageSpeed mobile ≥85) на эталонной теме, без кэш-плагина.
- **Exit:** все 9 критериев раздела 15 проходят чек-листом, включая тест "6-8 блоков за 15 минут" с реальным нетренированным тестировщиком.

### Phase 7 — Приёмка MVP и подготовка к раскатке
- Security review (обязателен перед массовым релизом, раздел 10) — желательно независимый от команды разработки, с учётом адверсариального корпуса из спайка 4.
- План поэтапной раскатки (staged rollout) с возможностью отката.
- Базовая доступность: клавиатурный проход по редактору, точечная проверка скринридером (полный WCAG-аудит — вторая очередь).
- Проверка полноты i18n (ru/en).
- Финальный прогон 9 критериев раздела 15 с фиксацией (скриншоты/логи/цифры бенчмарка).
- **Exit:** все критерии раздела 15 закрыты с доказательствами; security review подписан; путь отката проверен на staging.

## Вторая очередь: закладывать точку расширения сейчас, но не строить

- **ACF-теги**: реестр тегов (Phase 2) — без хардкода "только core WP" в резолвере.
- **AI-модуль / сборка страницы целиком**: REST list/get/insert уже в MVP (Phase 4); эндпоинт "собрать всю страницу по последовательности [{block_id, slots}]" (раздел 9) — набросать форму контракта уже сейчас (можно застабить как 501), чтобы манифест/insert-контракт не ломался позже.
- **Экспорт/импорт в файл**: держать `project_json` сериализуемо чистым (без PHP-ссылок на объекты, без server-only полей) — тогда экспорт добавляется аддитивно.
- **Сайдбары**: движок display conditions и редактор частей сайта (Phase 5) — строить обобщённо над "тип части", не хардкодить только шапку/подвал.
- **Тарифные ограничения**: матрица прав и REST-контроллеры (Phase 5) — оставить естественную точку для будущего hook/filter лимита, саму логику лимитов не строить.

**Игнорировать полностью для MVP:** централизованная кросс-сайтовая телеметрия (только единый локальный формат лога с точкой расширения), полный WCAG-аудит, углублённая SEO-интеграция сверх ссылки на Yoast/Rank Math.

## Критические файлы

- `includes/Storage/` — схема+миграции, фундамент для всего (раздел 3).
- `includes/Blocks/` — схема/валидатор манифеста, контракт для редактора и будущего AI/REST (раздел 9).
- `includes/Render/` — publish pipeline + батч-резолвер тегов, критический путь производительности/корректности (разделы 8, 11).
- `includes/Sanitize/` — общий двухконтурный санитайзер, критический для безопасности модуль (раздел 10).
- `assets/editor/` — интеграция canvas GrapesJS с темой, самая рискованная с продуктовой точки зрения поверхность (критерий приёмки 5.2).

## Первые шаги сразу после утверждения плана (Phase 0, старт в этой же сессии)

1. Создать скелет плагина: `wp-gj-builder.php` (header, константы, activation/deactivation-заглушки), `uninstall.php`-заглушка.
2. `composer.json` (PSR-4 `WPGJBuilder\` → `includes/`) + `package.json` с `@wordpress/scripts`.
3. Спайк 3 первым (dual-storage — самый блокирующий): черновая схема `wpb_documents` + dbDelta + минимальная миграция на фейковом `schema_version` + тест "заморозки" на повреждённом JSON.
4. Параллельно — спайк 1 (тема в canvas): поднять минимальный GrapesJS canvas, подключить реальный CSS тестовой темы, визуально сверить рендер блока.
5. Затем спайк 2 (батч-резолвер тегов) и спайк 4 (санитайзер + адверсариальный корпус по перечисленным CVE).

## Проверка

- Phase 0 спайки проверяются напрямую: скриншот-сравнение (спайк 1), профилирование числа запросов через query log (спайк 2), намеренная порча JSON + проверка что страница не падает (спайк 3), прогон адверсариального корпуса через оба контура с ожидаемым отказом (спайк 4).
- Дальше — по exit-критериям каждой фазы выше; раздел 15 спеки (9 сквозных критериев приёмки) — финальная проверка на Phase 7.
- Использовать WordPress Playground (`npx @wp-playground/cli@latest server --auto-mount`) для локальной проверки каждой фазы на реальном WP — уже проверенный в этой сессии рабочий способ поднять окружение без Docker.
