# Выбор основы для конструктора страниц WordPress: аналитика

Дата: 18 июля 2026
Контекст: встроенный в хостинг конструктор страниц для партнёров (веб-студии/агентства), массовый shared-хостинг WordPress, готовые элементы и страницы, наследующие стиль темы, с прицелом на подключение AI-сборки страниц в будущем.

## 1. Критерии оценки

Для этой задачи ключевые критерии — не «какой конструктор лучше для конечного пользователя», а какой можно **встроить в собственный плагин, распространяемый бесплатно тысячам сайтов на общем хостинге**:

- Лицензия и стоимость при массовом распространении (нельзя платить за сайт/за домен на каждый из тысяч сайтов партнёров).
- Возможность оплаты и легальной работы из РФ без обходных схем.
- Производительность и нагрузка на shared-хостинг (CPU/RAM, вес фронтенда).
- Насколько легко элементы могут наследовать CSS/стиль активной темы, а не тянуть свой дизайн-язык.
- Зрелость, активность разработки, размер комьюнити.
- Пригодность как фундамента для AI-сборки страниц (открытый JSON/HTML-контент, API для программной вставки блоков).
- Трудозатраты на разработку собственного плагина поверх системы.

## 2. Рассмотренные варианты

### Elementor

Elementor Pro в 2026 году стоит от $59/год за один сайт до $399/год за 1000 сайтов (Agency), лицензию нужно активировать отдельно на каждом сайте — то есть даже старший тариф не даёт неограниченного покрытия, а модель «активаций» плохо ложится на десятки тысяч сайтов хостинга. Официальная оплата из России фактически недоступна: сайт Elementor принимает только PayPal и зарубежные карты через Stripe, россияне вынуждены использовать платёжные мосты и виртуальные карты — то есть партнёрам придётся оплачивать зарубежный сервис в обход, что уже сегодня создаёт трение (сам запрос это подтверждает). Кроме лицензии, Elementor тяжёлый: генерирует около 570 КБ базового веса страницы против 42 КБ у современных конкурентов вроде Breakdance, что критично на shared-хостинге с ограниченными ресурсами. Как основа для *своего* бесплатного плагина Elementor не подходит в принципе — это закрытый коммерческий продукт, встроить его в инфраструктуру хостинга и раздавать бесплатно нельзя.

### Gutenberg (стандартный редактор блоков / FSE)

Встроен в ядро WordPress, бесплатен, с 2023 по 2026 год доля FSE-совместимых блочных тем выросла с 5% до примерно 18% — экосистема растёт. Блочные темы дают чистый HTML и минимум JS (Interactivity API добавляет всего ~10 КБ), что хорошо для shared-хостинга. Но у Gutenberg два системных минуса для этой задачи: (1) интерфейс действительно не интуитивен для конечных пользователей-партнёров, привыкших к перетаскиванию в духе Elementor — это подтверждается и рынком (агентства массово уходят на Bricks/Breakdance/Elementor именно из-за UX); (2) кастомные блоки могут быть ресурсоёмкими, и производительность сильно зависит от качества конкретных блоков — гарантий «из коробки» нет. Развивать полноценный конструктор поверх Gutenberg возможно (это открытая система на React), но UX-разрыв с привычным партнёрам Elementor-like опытом придётся закрывать большим объёмом собственной разработки поверх чужого API, который не создавался как продукт для конечного пользователя.

### GrapesJS

Открытый фреймворк (лицензия BSD-3-Clause), не готовый продукт, а конструктор для встраивания в своё приложение — то есть именно то, что нужно для собственного плагина. На июль 2026 года это самый зрелый инструмент в категории open-source: более 25 800 звёзд на GitHub, актуальные коммиты (последний push в апреле-мае 2026), переписан на TypeScript, есть зрелая React-обвязка. Лицензия BSD-3 позволяет свободно встраивать GrapesJS в закрытый коммерческий плагин и распространять его бесплатно на неограниченное число сайтов — без по-сайтовых лицензий и без проблем с оплатой из РФ, так как ядро не требует платной подписки вообще. GrapesJS изначально проектировался для загрузки CSS активной темы прямо в canvas редактора (пример из документации: `canvas: { styles: ['/wp-content/themes/your-theme/style.css'] } }`), что напрямую закрывает требование «элементы должны наследовать стиль темы». Важно и то, что GrapesJS уже используется в продакшене для SaaS-редакторов (email-маркетинг, CMS, e-commerce, e-learning) — то есть это протестированный на масштабе фундамент, а не экзотика.

Отдельный плюс под будущую задачу: у GrapesJS уже есть экосистема AI-надстроек (GPT-плагин, `@silexlabs/grapesjs-ai-copilot`, коммерческий Grapes Studio AI от самой команды GrapesJS, который по промпту генерирует целую страницу из блоков). Это прямое доказательство, что архитектура GrapesJS (компонентное дерево + JSON-представление страницы) хорошо ложится на программную/AI-сборку страниц — ровно то, что запланировано на будущее.

Минусы: GrapesJS — это фреймворк, а не «плагин из коробки». Готового WordPress-плагина с элементами и шаблонами под задачу партнёров не существует — большую часть UI, набор блоков, интеграцию с WP (сохранение контента, права, шорткоды/блочный рендеринг) придётся написать самостоятельно. Это ожидаемо и совпадает с задачей — плагин планируется разрабатывать с нуля в любом случае.

### Дополнительно рассмотренные варианты (по запросу)

- **Bricks Builder** — современный, производительный (25-30 КБ HTML, ~0.8с LCP на shared-хостинге), но коммерческий продукт с лицензией (Ultimate — $599 разово), не предназначен для встраивания в чужой бесплатный продукт и распространения по тысячам сайтов; та же проблема оплаты/легальности, что и у Elementor.
- **Breakdance** — ещё легче по фронтенду (42 КБ, PageSpeed 85-95 из коробки), 130+ готовых элементов, но тоже закрытый коммерческий продукт с посайтовым лицензированием — не подходит как основа для бесплатного встроенного в хостинг плагина по тем же причинам.
- **Oxygen 6** — полная переработка на Twig/PHP, тоже коммерческий продукт (лицензия Ultimate $179 разово), делит ~80% кода с Breakdance; хорош для агентств, которые лицензируют его сами, но не для встраивания в чужую бесплатную дистрибуцию.

Общий вывод по этой группе: Bricks/Breakdance/Oxygen — сильные продукты для *партнёров, которые сами покупают лицензию и строят сайты клиентам*. Это не то же самое, что «встроить готовый бесплатный конструктор в хостинг» — по лицензии их так распространять нельзя.

## 3. Итоговая рекомендация

Взять за основу **GrapesJS**.

Обоснование в одном абзаце: это единственный из рассмотренных вариантов, который одновременно (1) можно легально и бесплатно встроить в собственный плагин и раздать на неограниченное число сайтов хостинга без по-сайтовых лицензий и без проблем оплаты из РФ, (2) архитектурно спроектирован для наследования стилей произвольной темы через прямую загрузку CSS в canvas, (3) современный и активно поддерживаемый (TypeScript, регулярные релизы, растущее AI-комьюнити), (4) уже имеет рыночное доказательство пригодности к AI-сборке страниц через собственный Studio AI и сторонние AI-плагины — то есть путь к будущей автоматической сборке страниц AI не потребует смены фундамента. Gutenberg проигрывает по готовому UX для партнёров и по гибкости встраивания собственного визуального языка; Elementor и другие коммерческие конструкторы (Bricks/Breakdance/Oxygen) исключены лицензией и моделью оплаты, несовместимой с бесплатным массовым встраиванием в хостинг.

Плата за этот выбор — весь UI, набор блоков, готовые шаблоны страниц и интеграцию с WordPress придётся спроектировать и разработать с нуля поверх фреймворка; готового продукта «из коробки» не будет. Это ожидаемо и соответствует следующему этапу задачи — разработке самого плагина.

## 4. Что важно учесть на этапе разработки плагина (кратко)

- Хранить контент страницы как структурированный JSON (компонентное дерево GrapesJS) + рендерить в чистый HTML/CSS при сохранении — это даст и лёгкий фронтенд для shared-хостинга, и удобный формат для будущей AI-генерации.
- Библиотеку блоков и готовых страниц строить как отдельный слой поверх GrapesJS (Plugins API), с наследованием CSS-переменных/классов активной темы, а не с собственной жёсткой стилистикой.
- Закладывать в архитектуру точку расширения под AI уже сейчас (например, единый метод «вставить блок/страницу по JSON-описанию»), чтобы в будущем AI-модуль мог просто вызывать тот же API, которым пользуется человек в редакторе.
- Учесть нагрузку на shared-хостинг отдельно от нагрузки на клиентский браузер: сам редактор (GrapesJS) выполняется в браузере партнёра/клиента и не нагружает сервер хостинга — на сервере остаётся только хранение и рендеринг сохранённого HTML/CSS, что снимает основные опасения по ресурсам PHP/MySQL на массовом хостинге.

## 5. Раунд 2: углублённая проверка возражений (18 июля 2026, доп. запрос)

### 5.1. Elementor — бесплатная версия + готовые бесплатные аддоны: меняет ли это вывод?

Факт подтверждается: рынок бесплатных аддонов для Elementor Free большой и зрелый. Essential Addons (2 млн+ активных установок, 95+ бесплатных виджетов, 1000+ шаблонов секций), The Plus Addons (800+ настроек, 300+ блоков), Element Pack (300+ виджетов, 2700+ готовых страниц/блоков) — всё это распространяется бесплатно через каталог WordPress.org. Технически действительно можно сделать «свой» бесплатный плагин с готовыми блоками в духе этих аддонов.

Но это не снимает фундаментальную проблему, а перемещает её на уровень ниже. Три причины, почему Elementor всё равно не годится в основу:

1. **Free — это осознанно урезанный продукт, а не полноценная платформа.** Elementor Free не умеет работать с произвольными полями/CPT, не имеет попапов и большей части шаблонов — это стратегическое ограничение вендора, а не техническая случайность: у Elementor коммерческий интерес держать Free ограниченным, чтобы продавать Pro. Значит «полноценный конструктор с готовыми страницами», который упомянут в задаче (шаблоны сайтов, шапки/подвалы, динамический контент), в парадигме Free принципиально недостижим без апгрейда на Pro — а Pro возвращает всю проблему оплаты из РФ и посайтовых лицензий из первого раунда анализа.
2. **Фундамент остаётся собственностью конкурента, а не вашего продукта.** Даже собрав свой аддон-плагин поверх Elementor Free, вы физически зависите от чужого ядра: Elementor Ltd может в любой версии изменить или ограничить бесплатный тариф (уже делали это исторически — убрали lifetime-лицензию в 2022 году), поменять API, потребовать аккаунт/телеметрию и т.д. Партнёры хостинга в этом случае строят свой бизнес на фундаменте, которым управляет прямой рыночный конкурент этой ниши, а не хостинг.
3. **Формат данных Elementor непрозрачен и не документирован официально.** Elementor хранит разметку страницы как сложный вложенный JSON в мета-поле `_elementor_data` — это внутренний формат, который меняется между версиями без публичной спецификации. Строить на нём будущую AI-генерацию страниц значит завязываться на реверс-инжиниринг чужого закрытого формата, который может измениться при любом обновлении плагина и сломать интеграцию.

Вывод по Elementor не меняется: наличие бесплатных аддонов снижает *стоимость входа для партнёра*, но не решает вопрос *кому принадлежит платформа* — а это ключевой вопрос при выборе фундамента для собственного продукта хостинга.

### 5.2. Gutenberg — есть ли на рынке решения проблемы UI?

Да, целый класс плагинов существует именно для этого: Kadence Blocks (400 тыс.+ активных установок), Stackable, Spectra, GenerateBlocks, а также более точечные инструменты вроде EditorsKit, GutenKit и «Blocks Editor Interface Customizer». Они действительно делают Gutenberg удобнее: добавляют визуальные контролы стилизации в реальном времени, готовые секции/паттерны, где-то — режим фронтенд-редактирования, приближающий опыт к классическим конструкторам.

Критическая оговорка: все они **надстройки над блочной моделью**, а не замена интерфейса. Kadence Blocks описывается рынком как усиление стандартного редактора блоков, Stackable — как трансформация Gutenberg в полноценный конструктор через дизайн-ориентированный подход поверх тех же блоков, Spectra — как визуальный drag-and-drop поверх блочного размещения. Ни один из них не меняет фундаментальную модель взаимодействия «блок → инспектор настроек в сайдбаре»; это не свободный canvas, к которому привыкли пользователи Elementor/Bricks/Breakdance. То есть рынок решает проблему «Gutenberg скучный и без вижуал-контролов», но не полностью решает исходную проблему «интерфейс не интуитивен для непривычного пользователя» — разрыв в UX с Elementor-подобным опытом сокращается, но не закрывается.

Это усиливает, а не ослабляет исходный вывод: если строить свой продукт поверх Gutenberg, придётся либо (а) повторить путь Kadence/Stackable и получить тот же частичный результат, либо (б) написать полноценный альтернативный UI поверх Block API — что по объёму работы сопоставимо с разработкой собственного редактора на GrapesJS, но с меньшей гибкостью, потому что вы всё ещё связаны моделью данных и рендеринга блоков WordPress.

### 5.3. GrapesJS + WordPress — почему прошлые агенты могли советовать не смешивать

Разобрал вопрос предметно, вот что нашлось, и это меняет тональность (не вывод, но добавляет важные оговорки):

**Архитектурно интеграция описана и работает.** Официальный паттерн (в том числе в материалах самой GrapesJS.com) — «WordPress отвечает за пользователей, права и хранение, GrapesJS — за визуальное редактирование», разметка загружается в canvas через iframe, что изолирует стили/скрипты редактора от админки WP и избегает конфликтов версий React с самим wp-admin. Скрипты подключаются штатно через `wp_enqueue_script/style`. Технической несовместимости с WordPress как таковой нет.

**Настоящая причина осторожности — не техническая совместимость, а модель доверия и безопасность на многопользовательском хостинге.** У GrapesJS как открытого редактора был не один CVE на XSS именно в собственном UI редактора: `SNYK-JS-GRAPESJS-2935960` и `2342412`, issue #3082 (XSS в Live Preview через импорт HTML с `onload`-атрибутами в тегах вроде `<img>`), issue #4076 (инъекция через атрибуты компонента, например ID, которая исполняется при клике в live preview). Смысл этих уязвимостей — редактор по своей природе работает с «сырым» HTML/CSS/иногда JS, и, если ввод не дезинфицирован на каждом уровне (селекторы классов, атрибуты компонентов, импорт HTML), пользователь редактора может внедрить исполняемый код. На одном сайте это управляемый риск (доверенный админ), но в контексте задачи — **один плагин, встроенный в тысячи сайтов на общем хостинге, где редакторами могут быть партнёры и, в перспективе, их клиенты** — это уже вопрос периметра безопасности всей хостинг-платформы, а не отдельного сайта. Именно поэтому «не смешивайте» — вероятно, был осторожный совет по умолчанию: чужой открытый редактор, спроектированный для доверенной среды (один разработчик встраивает его в своё приложение и сам решает, кому давать доступ), нельзя просто отдать в руки неопределённо широкого круга конечных пользователей без серьёзного слоя серверной санитизации и ограничения прав — а этот слой в GrapesJS не поставляется «из коробки», его целиком проектирует и поддерживает ваша команда.

**Вторая причина — на GrapesJS полностью ложится эксплуатационная нагрузка, которую у готовых продуктов берёт на себя вендор.** Раз GrapesJS — «фреймворк, а не продукт», сохранение/автосохранение, история версий, права доступа, дедупликация контента при копировании страниц, поведение при смене темы, экспорт/импорт — всё это нужно спроектировать и поддерживать самостоятельно, без апдейтов от вендора «продуктовой» части. Показательный прецедент: даже Mautic, которые в своё время перешли на GrapesJS вместо собственного редактора, и SalesReady, которые построили на GrapesJS процесс генерации документов, в итоге доросли до использования платного Studio SDK от команды GrapesJS — то есть сама GrapesJS-команда предлагает коммерческий слой именно потому, что голый open-source фреймворк для серьёзного продакшен-продукта требует ощутимых дополнительных вложений, которые проще купить, чем построить с нуля.

**Третья причина — потеря нативной интеграции с экосистемой WordPress.** GrapesJS отдаёт результат как чистый HTML/CSS (не блоки WordPress), поэтому контент не будет «переанализирован» под блочные фичи SEO/кэш-плагинов, точно так же как это уже происходит у Elementor и любых non-Gutenberg конструкторов — SEO-плагины (Rank Math, Yoast) всё равно проверяют финальный `post_content`/HTML и работают, но не получают блочно-специфичные преимущества (SEO-блоки, блочные паттерны валидации и т.п.). Это не уникальный минус GrapesJS, а общий побочный эффект отказа от блочной модели — тот же компромисс существует и у Elementor/Bricks/Breakdance, так что он не должен сам по себе разворачивать решение в сторону Gutenberg, но его стоит держать в голове.

**Итог по разделу:** предыдущие рекомендации «не смешивать» скорее всего были осторожностью по умолчанию в адрес любого нетривиального встраивания open-source редактора в мультитенантную CMS-инфраструктуру, а не специфичным техническим приговором GrapesJS. Технически интеграция работает и имеет успешные прецеденты (Mautic, SalesReady, Mailchimp-экосистема, множество SaaS). Но вывод честно нужно скорректировать: **выбор GrapesJS остаётся верным, только если в архитектуру заранее закладывается серверная санитизация HTML/CSS при сохранении, разграничение прав («кто может добавлять произвольный HTML/JS-блок» vs «кто работает только с готовыми брендированными блоками»), и бюджет на security-ревью перед раскаткой на массовый хостинг** — это не блокер, но обязательная статья работ, которую нельзя пропустить, в отличие от закрытых коммерческих продуктов, где эта работа уже сделана вендором за много лет.

## 6. Обновлённая рекомендация с учётом раунда 2

Рекомендация не меняется: **GrapesJS остаётся правильным фундаментом.** Раунд 2 не нашёл контраргумента, который бы перевесил исходные причины (лицензия, стоимость на масштабе, оплата из РФ, наследование стиля темы, AI-совместимость) — он лишь уточнил цену этого выбора: помимо разработки UI и блоков с нуля, в бюджет проекта нужно заранее включить (1) серверный слой санитизации/валидации сохраняемого HTML/CSS/JS, (2) модель прав доступа к «свободным» HTML/JS-блокам отдельно от готовых брендированных блоков, (3) security-ревью перед массовым релизом на хостинг. Альтернатива — не менять фундамент, а компенсировать риск ограничением: на первом этапе не давать партнёрам/клиентам доступ к произвольному HTML/JS-коду в редакторе вообще, разрешив только готовые блоки из собственной библиотеки (аналог того, как Elementor Free ограничивает виджеты) — это резко снижает поверхность атаки и даёт время построить полноценную санитизацию до открытия более свободного режима редактирования.

## 7. Раунд 3: производительность, нагрузка на БД и разница архитектур Gutenberg vs GrapesJS (18 июля 2026, доп. запрос)

### 7.1. Ваш опыт (Elementor медленнее Gutenberg) — почему так и есть, данные подтверждают

Это не субъективное ощущение, а задокументированный на рынке эффект, у которого две независимые причины — фронтенд и БД/сервер.

**Фронтенд (вес страницы, число запросов в браузере).** Elementor добавляет ~150-300 КБ веса страницы сверх сопоставимой страницы на Gutenberg ещё до всякой оптимизации, а с учётом полного набора ассетов (иконки, анимации, виджеты аддонов) разница достигает 3-5х по CSS/JS payload. Ключевая техническая причина: Elementor по умолчанию подгружает CSS/JS *всех* используемых на сайте виджетов почти на каждой странице, где он вообще активен, а не только тех виджетов, что реально есть на конкретной странице — «Improved Asset Loading» в настройках частично чинит это, но это опция, которую нужно осознанно включить, а не поведение по умолчанию. На необорудованном сайте это доходит до 80-120 HTTP-запросов и 2.5-4 МБ веса страницы против 35-70 запросов и 0.8-1.8 МБ у чистого Gutenberg.

**Сервер и БД.** Elementor хранит разметку страницы как сериализованный вложенный JSON в `wp_postmeta` (`_elementor_data`), и на каждый запрос страницы на фронтенде этот JSON нужно распарсить и превратить в HTML через PHP-рендерer Elementor — то есть по умолчанию (без кэша страниц) это дополнительная работа CPU на каждый визит, а не разовая при сохранении. На магазине с 30+ шаблонами Elementor это стабильно даёт TTFB выше 1.5с на shared-хостинге именно потому, что виджеты парсятся при каждом запросе перед рендером. Отдельная, часто недооценённая проблема — «мусорные» автозагружаемые (`autoload = yes`) записи Elementor в `wp_options` (глобальный CSS, кэш шаблонов, настройки): WordPress подтягивает *все* такие записи в память при абсолютно каждом запросе к сайту, независимо от того, какая это страница; сайты сообщают о записях вроде `_elementor_global_css` весом 2+ МБ, которые не удаляются даже после переезда на другой конструктор — то есть штраф за Elementor можно унаследовать «навсегда», если его не чистить руками.

Итог: ваш опыт совпадает с рыночными данными и объясняется архитектурно, а не случайно — Elementor хуже по обеим осям (фронтенд-вес и серверный CPU/БД), потому что интерпретирует свой формат на каждый запрос и не сегментирует ассеты по умолчанию.

### 7.2. Как будет с GrapesJS

Здесь важно разделить два разных вопроса: производительность **редактора** (внутри wp-admin, для партнёра) и производительность **готовой публичной страницы** (для посетителя сайта) — у GrapesJS они принципиально разделены, и это его главное архитектурное преимущество перед Elementor.

GrapesJS как JS-библиотека грузится и работает только в интерфейсе редактирования (в wp-admin, у партнёра/клиента в браузере) — обычный посетитель сайта её вообще не скачивает. На выходе GrapesJS отдаёт чистый статический HTML/CSS, который можно сохранить в `post_content` ровно так же, как сохраняется результат **статического** блока Gutenberg: без обёртки в чужой JSON-формат, без PHP-парсинга виджетов на каждый визит. При таком подходе (а именно так и планируется — раздел 4 этого документа) публичная страница на GrapesJS-фундаменте по нагрузке на сервер и БД должна быть на уровне статических блоков Gutenberg или лучше: WordPress просто достаёт готовый HTML из `wp_posts` и отдаёт его, без каких-либо дополнительных JOIN-запросов и без раздутых `wp_options`, характерных для Elementor. Разница с Elementor в проектировании принципиальная: Elementor хранит *инструкции по сборке страницы* и пересобирает её на лету на каждый запрос; правильно спроектированный GrapesJS-плагин хранит уже *готовый результат* и просто отдаёт его.

Важная оговорка, а не автоматическая гарантия: это преимущество не «встроено» в GrapesJS само по себе — оно появляется только если ваш плагин будет реализован именно так (кэшировать/сохранять финальный HTML при публикации, а не пересобирать вёрстку из JSON на каждый визит на PHP). Если пойти по пути «хранить только компонентный JSON и рендерить его в HTML на сервере при каждом запросе» — вы своими руками воспроизведёте ту же архитектурную проблему, что и у Elementor. Это прямой пункт в архитектурные требования к разработке плагина (см. раздел 4): рендерить и кэшировать HTML при сохранении/публикации, JSON-дерево хранить отдельно только как «источник правды» для повторного редактирования, а не как то, что парсится на лету для показа посетителям.

Как только в продукт добавится динамика (формы, персонализация, данные из БД, будущие AI-виджеты) — на неё расходы на сервер/БД появятся неизбежно, как и в Gutenberg с динамическими блоками (`render_callback`, который выполняется на каждый запрос) или в Elementor Pro (динамические поля/попапы). Здесь GrapesJS не имеет структурного преимущества — цена динамического контента одинакова для всех трёх систем, разница только в статической части, которая в реальных сайтах партнёров хостинга обычно и составляет большинство страниц.

### 7.4. Тезисно: в чём разница архитектур Gutenberg и GrapesJS

Раз в раз — коротко, без терминов на вырост:

- **Что это, в одном предложении.** Gutenberg — часть ядра WordPress, редактор, который *знает*, что такое WordPress, права, посты, темы, REST API — из коробки. GrapesJS — независимая JS-библиотека, которая ничего не знает про WordPress вообще; ей всё равно, куда сохранять результат — в WordPress, в Node.js-бэкенд или в файл на диске. WordPress-специфику в GrapesJS добавляете вы сами.
- **Единица содержимого.** В Gutenberg всё состоит из «блоков» — именованных, зарегистрированных в системе сущностей (`core/paragraph`, `core/columns`, ваш `myplugin/hero`), у каждого блока есть строгая схема атрибутов. В GrapesJS всё состоит из «компонентов» в собственном дереве GrapesJS — более низкоуровневая и гибкая модель (по сути дерево HTML-тегов+стилей+поведения), которая не обязана соответствовать какой-либо внешней схеме.
- **Где хранится результат.** Gutenberg хранит содержимое как читаемый текст с HTML-комментариями-маркерами прямо в поле `post_content` (`<!-- wp:paragraph -->...<!-- /wp:paragraph -->`) — WordPress core умеет это парсить обратно в блоки функцией `parse_blocks()`. GrapesJS хранит своё «состояние» (дерево компонентов + стили) как произвольный JSON через свой Storage Manager — формат этого JSON не имеет отношения к WordPress и не понимается им «из коробки»; связь с `post_content` нужно построить самим (например, при сохранении рендерить в HTML и класть в `post_content`, а JSON держать отдельно для повторного редактирования).
- **Как рендерится на фронтенде.** Статический блок Gutenberg — просто HTML, уже лежащий в базе, отдаётся как есть. Динамический блок Gutenberg — PHP-функция (`render_callback`) выполняется на сервере при каждом запросе и генерирует HTML на лету. GrapesJS вообще не участвует в рендере на фронтенде — весь GrapesJS-рантайм живёт только в редакторе; на фронтенде посетитель получает чистый HTML/CSS без какой-либо GrapesJS-логики (если только вы сознательно не подключите её отдельно для интерактивных элементов).
- **Как редактор понимает тему сайта.** Gutenberg берёт настройки темы автоматически из `theme.json` (палитра, типографика, отступы) — это встроенный контракт между темой и редактором. У GrapesJS такого контракта нет: тема должна быть явно подключена в конфигурацию canvas (`canvas: { styles: [...] }`), чтобы редактор вообще увидел, как элементы будут выглядеть на реальном сайте — интеграцию со стилями темы нужно спроектировать вручную.
- **Как расширяется.** Gutenberg расширяется через официальный Block API: `block.json`, JS-компоненты `edit`/`save`, `InspectorControls` для настроек в сайдбаре — жёстко заданный WordPress-way. GrapesJS расширяется через собственную систему плагинов (`grapesjs.plugins.add`), Block Manager, Style Manager, Trait Manager — независимую от WordPress архитектуру, которую вы адаптируете под задачи хостинга сами.
- **Кто отвечает за целостность данных при обновлениях.** В Gutenberg WordPress core гарантирует обратную совместимость формата постов между версиями (с механизмом «deprecated» версий блока). В GrapesJS обратную совместимость формата вашего собственного JSON и рендера вы обеспечиваете сами — апдейт вашего плагина не должен «ломать» ранее сохранённые страницы, и эту миграционную логику придётся писать вам, а не ждать от вендора.

## 8. Раунд 4: GrapesJS vs Gutenberg — надёжность, наследование стиля темы, механизм повторного редактирования (18 июля 2026, доп. запрос)

Elementor из сравнения убираем — решение по нему закрыто в предыдущих раундах. Ниже — прямые ответы на три сомнения.

### 8.1. «Gutenberg встроен в экосистему и выглядит надёжнее» — так ли это на практике

Ощущение объяснимо, но у него есть оборотная сторона, которая как раз объясняет ваш опыт с «критическими ошибками блоков». Механизм называется **валидация блока**: при каждом открытии поста Gutenberg сравнивает сохранённый в базе HTML блока с тем HTML, который прямо сейчас выдаёт функция `save()` этого блока в текущей версии кода. Если они не совпадают побайтово — редактор показывает «Этот блок содержит неожиданный или недопустимый контент» и предлагает «восстановить» блок, потому что не может гарантировать отсутствие потери данных. На GitHub-трекере WordPress/Gutenberg таких issue — сотни, начиная с 2018 года; типичные триггеры: обновление плагина/темы чуть меняет разметку блока, паттерны, сохранённые в одной версии WP, не читаются в другой, `target="_blank"` в кастомной ссылке ломает валидацию при каждой перезагрузке, синхронизированные паттерны (бывшие reusable blocks) валидны в одном контексте и невалидны в другом.

Прямое следствие для вашей задачи: если вы пишете **свои** блоки для Gutenberg (а это неизбежно для готовых элементов из ТЗ), ответственность за то, чтобы будущая версия блока не «сломала» уже сохранённые у партнёров страницы, целиком на вас — нужно вручную писать `deprecated`-версии блока при любом изменении разметки. Это ровно тот же класс риска «апдейт плагина ломает старый контент», который в разделе 5.3 обсуждался применительно к GrapesJS, только у Gutenberg он оформлен как официальный, ожидаемый механизм ядра (значит, более предсказуем и документирован), а не как результат кастомной реализации. Итог: «Gutenberg = встроенное в ядро» действительно даёт предсказуемость самого движка блоков, но не избавляет от собственных ошибок валидации в **ваших** блоках поверх него — тот же источник «глючности», который вы наблюдали, воспроизведётся и в вашем продукте, если не проектировать миграции блоков заранее. Это не аргумент против Gutenberg, а поправка к тезису «раз он в ядре, значит автоматически надёжнее для контента, который строите вы».

### 8.2. Наследование стиля темы у GrapesJS — что реально происходит и в чём разница с Gutenberg

Здесь стоит разделить «выглядит правильно» и «редактор понимает дизайн-язык темы» — это два разных уровня, и путаница в вопросе именно на стыке.

**Уровень 1 — визуальная точность (это работает «из коробки»).** Как описано в разделе 2, GrapesJS рендерит canvas в `iframe` и туда можно подключить настоящий CSS-файл темы (`canvas: { styles: ['/wp-content/themes/тема/style.css'] } }`). Браузер применяет каскад CSS ровно так же, как на реальном сайте — визуально блоки в редакторе будут выглядеть как на фронтенде. Это не эмуляция и не приближение, а тот же самый CSS.

**Уровень 2 — «понимание» дизайн-токенов темы редактором (вот тут разница с Gutenberg настоящая).** У Gutenberg это встроенный контракт: тема описывает палитру, шрифты, отступы в `theme.json`, и редактор автоматически превращает это в готовые UI-контролы (например, палитра цветов темы сразу появляется как набор кружочков в панели настройки блока). У GrapesJS такого контракта по умолчанию нет — Style Manager не знает про `theme.json` и не построит контролы сам. Но это решаемая и хорошо описанная задача, а не тупик: (1) существует официальный плагин `grapesjs-css-variables`, который умеет считывать CSS custom properties (`--primary`, `--spacing` и т.п.) и превращать их в контролы Style Manager с привязкой `background-color: var(--primary)`; (2) современные блочные темы WordPress уже генерируют свою палитру и типографику именно как CSS custom properties из `theme.json` (`--wp--preset--color--primary` и подобные) — то есть данные для такой интеграции уже есть в теме «из коробки», и типовая задача разработки — написать скрипт, который на старте редактора считывает `theme.json` активной темы через WP REST API и генерирует из него сектора Style Manager/палитру автоматически. Это ровно тот же путь, каким Kadence/Stackable закрывают дизайн-интеграцию поверх Gutenberg, только придётся построить его самостоятельно, а не получить бесплатно от ядра.

Вывод по разделу: наследование стиля темы у GrapesJS работает визуально из коробки, а на уровне «редактор подсказывает токены темы» требует одной конкретной, хорошо документированной инженерной задачи (парсинг `theme.json` → генерация Style Manager секторов через `grapesjs-css-variables` или аналог). Это входит в объём разработки плагина, но не является архитектурным риском или неизвестностью.

### 8.3. Как GrapesJS повторно редактирует сохранённую страницу — важное уточнение к разделу 7

В разделе 7 уже было сказано, что для страницы нужно хранить рендер (HTML) отдельно от JSON-дерева компонентов — теперь по документации GrapesJS это можно сформулировать точнее и жёстче, чем просто «рекомендация»:

**GrapesJS никогда не должен полагаться на чистый HTML как источник для повторного редактирования — и это прямо написано в его официальной документации.** GrapesJS действительно умеет импортировать сырой HTML/CSS и «на лету» превращать теги в компоненты — процесс называется **Component Recognition**: парсер проходит по DOM-дереву и для каждого элемента спрашивает по очереди все зарегистрированные типы компонентов «это твой элемент?» (`isComponent`), пока какой-то тип не откликнется; если ни один кастомный тип не откликнулся, элемент становится компонентом-заглушкой (обычно generic `div`/`text`). Это делает возможным «открыть произвольный HTML» — но именно поэтому это подходит для первичного импорта или демонстрации, а не для надёжного повторного редактирования: сам GrapesJS предупреждает, что при этом «многие данные могут быть потеряны» (типы кастомных компонентов, атрибуты, привязанные trait'ы, специфичные настройки), потому что чистый HTML физически не содержит этой информации — в нём остаётся только то, что можно выразить тегами/классами/атрибутами.

Правильная и единственная официально поддерживаемая схема — хранить **JSON project data** (`editor.getProjectData()` на сохранении → `editor.loadProjectData(json)` при повторном открытии). Этот JSON — не «сырая разметка», а полное состояние редактора: дерево компонентов с их точными типами, стили, страницы, настройки — именно оттуда редактор восстанавливает страницу без потерь при повторном открытии. HTML в этой схеме — производный артефакт только для показа посетителям сайта (тот самый «готовый результат», который в разделе 7 предлагалось класть в `post_content`), а не то, что редактор когда-либо пытается «прочитать назад».

Практический вывод для архитектуры плагина (уточнение к разделу 4): в БД партнёрского сайта нужно хранить два артефакта на каждую страницу — (1) JSON project data GrapesJS (источник правды для редактирования, может лежать в отдельном мета-поле или таблице) и (2) сгенерированный из него статический HTML/CSS в `post_content` (то, что видит посетитель и что даёт лёгкую производительность из раздела 7). Открывать страницу в редакторе нужно всегда через JSON, никогда через попытку распарсить сохранённый HTML обратно — так поведение будет предсказуемым и без потерь данных при каждом повторном открытии.

### 8.4. Синтез

Оба сомнения решаемы и не меняют рекомендацию:

- «Gutenberg надёжнее, потому что в ядре» — верно для самого движка блоков, но не защищает от валидационных ошибок в ваших собственных блоках; тот же класс проблем, что раздражал вас в Gutenberg, будет и в кастомных Gutenberg-блоках хостинга, если не писать миграции версий блоков вручную — у GrapesJS аналогичный риск управляется просто другим способом (версионирование JSON project data), а не отсутствует в принципе.
- «Не наследуются стили темы» — наследуются визуально всегда (реальный CSS в iframe), а «умное» использование дизайн-токенов темы в UI редактора — конкретная и решённая на рынке инженерная задача (`grapesjs-css-variables` + чтение `theme.json`), а не открытый вопрос.
- «Непонятно, как редактируется сохранённая как HTML страница» — она не должна редактироваться через HTML вообще; повторное редактирование идёт через JSON project data, HTML — это только витрина для посетителей. Это снимает саму предпосылку сомнения.

Рекомендация остаётся: **GrapesJS**, при условии, что оба технических требования (генерация Style Manager из `theme.json` темы и хранение JSON project data как источника правды отдельно от публичного HTML) закладываются в архитектуру плагина с самого начала, а не добавляются постфактум.
