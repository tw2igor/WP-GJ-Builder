# Phase 6c: Инфраструктура per-page JS/CSS ассетов — вердикт

Дата: 19 июля 2026. Статус: **PASS**, проверено реальной публикацией через REST и реальным curl-сравнением фронтенд-HTML трёх разных страниц (с интерактивным блоком / только со статичным блоком / обычная страница вне конструктора). Дев-раннер `/wpgjb/v1/dev/phase6` — 8/8 (было 5/5, добавлено 3 новые проверки).

Последний оставшийся крупный кусок Phase 6 из раздела 11 спеки: "манифест блока декларирует `requirements.assets`; при публикации резолвится один раз в postmeta, не пересчитывается на каждый запрос" + "CSS — двумя файлами: общий долгокэшируемый и маленький per-page с переопределениями".

## Что реализовано

- `includes/Render/PageAssets.php` (новый) — `collect_block_ids()` (перенесено из `Publisher`, теперь общее место обхода дерева `project_data`) и `resolve_js_assets()`: для каждого РЕАЛЬНО присутствующего на странице блока читает `manifest.requirements.assets.js` и строит прямые URL на `blocks-library/{id}/assets/{file}`. Резолвится ОДИН РАЗ в `Publisher::publish()`, сохраняется в postmeta `_wpb_page_js_assets` — `wp_enqueue_scripts` на фронтенде просто читает готовый список, не пересчитывает на каждый запрос.
- **Разделение CSS (раздел 11/6):** раньше (Phase 4) `Publisher::publish()` дублировал ПОЛНЫЙ `style.css` каждого использованного блока ИНЛАЙНОМ в `_wpb_page_css` на КАЖДОЙ странице сайта. Теперь:
  - Общий файл `GET /wpgjb/v1/blocks/style.css` (уже существовал с Phase 3 для canvas редактора) теперь ТАКЖЕ подключается на фронтенде через `wp_enqueue_style()` (`FrontendRenderer::enqueue_page_assets()`) — одна и та же версионированная ссылка на каждой странице сайта, браузер скачивает её один раз и переиспользует между страницами.
  - `_wpb_page_css` теперь хранит и инлайнит ТОЛЬКО маленький per-page override (реальный вывод `editor.getCss()` — переопределения Style Manager), не базовые стили блоков.
  - Кэш-бастинг: `ThemeStylesController::blocks_style_version()` — md5-хэш содержимого ВСЕХ `style.css` библиотеки, используется и как `?ver=` в `wp_enqueue_style()` (фронтенд), и как query-параметр в canvasStyles редактора (`EditorPage.php`) — при изменении любого блока версия меняется САМА, поэтому можно безопасно поставить `Cache-Control: public, max-age=31536000, immutable` на этот маршрут (было `private, max-age=60`), не рискуя показать посетителю/редактору устаревший CSS.
- **Реальный интерактивный блок для сквозной проверки:** `test-faq` стал настоящим аккордеоном — вопрос теперь `<button aria-expanded>`, ответ скрыт через нативный `hidden` до клика. `assets/runtime/wpgjb-runtime.js` (новый, вне webpack-сборки редактора — простой standalone-файл) — общий делегирующий обработчик кликов; `blocks-library/test-faq/assets/accordion.js` — сам toggle-логика, объявляет `wpgjb-runtime` своей зависимостью (`wp_register_script`/`wp_enqueue_script` c `deps`) — именно так реализуется "рантайм подключается как dep интерактивного блока, только если он есть на странице" из раздела 11, без ручной проверки в коде.
- `ManifestValidator::validate_slot_def()` — по пути также добавлена валидация `first_screen` (см. Phase 6b), не относится к этой части, но того же PR.

## Проверено

**Дев-раннер (`/wpgjb/v1/dev/phase6`, 8/8):** `PageAssets::resolve_js_assets()` находит `accordion.js` для `test-faq` и возвращает ПУСТОЙ список для `test-cta` (не объявляет JS); `Publisher::publish()` с блоком `test-cta` (реальный `style.css` с `.wpb-block--cta`) — `_wpb_page_css` содержит пользовательский override-маркер, но НЕ содержит `.wpb-block--cta`.

**Реальная сквозная проверка (REST publish + curl трёх разных фронтенд-страниц):**
1. Страница с `test-faq` → `<script id="wpgjb-runtime-js" src=".../assets/runtime/wpgjb-runtime.js?ver=0.1.0-dev">`, `<script id="wpgjb-block-test-faq-js" src=".../blocks-library/test-faq/assets/accordion.js?ver=0.1.0-dev">`, `<link id="wpgjb-platform-blocks-css" href=".../wpgjb/v1/blocks/style.css?ver=b4d4e0fb83ed">` — все три присутствуют.
2. Страница ТОЛЬКО с `test-cta` (без JS-требований) → CSS-ссылка есть (подключается на любой построенной конструктором странице), но НИ рантайма, НИ accordion.js — нет.
3. Обычная страница вне конструктора (`_wpb_built` отсутствует) → ни CSS-ссылки, ни JS вообще.

Это прямое доказательство раздела 11 "только если на странице есть интерактивные блоки" — не по коду, а по факту трёх разных реальных HTTP-ответов.

**Регрессия:** `/wpgjb/v1/dev/phase{1,2,5,6}` — все PASS после рефакторинга `Publisher`/`FrontendRenderer`.

## Осознанно не сделано

- Ни один из существующих платформенных блоков, кроме `test-faq`, не получил реальный интерактивный JS — остальные 10 блоков остаются чисто статичной разметкой (что честно и корректно: спецификация не требует интерактивности у каждого блока, только механизм подключения по требованию).
- Минификация/конкатенация нескольких block-специфичных JS-файлов в один HTTP-запрос (если на странице несколько разных интерактивных блоков) не реализована — каждый блок подключается своим отдельным `<script>`; для текущего размера библиотеки (1 интерактивный блок) это не проблема, но при росте библиотеки может стать пунктом дальнейшей оптимизации.
