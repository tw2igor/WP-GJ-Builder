# Переход редактора на Elementor-стиль (3 колонки) — вердикт

Дата: 20 июля 2026. Статус: **PASS**, проверено реальным браузером (Puppeteer) на живом WP Playground, не только код-ревью.

## Контекст

Пользователь указал на серьёзное расхождение между реализованным редактором (Tilda-стиль: плавающая кнопка "+", модалка-пикер, клик-вставка, ноль drag-and-drop) и запрошенным направлением — полноценный GrapesJS-редактор, ориентированный на Elementor: слева панель элементов (+ вкладка "Блоки"), в центре канвас, справа панель свойств. Предварительный план и его валидация — `nifty-mixing-cocke.md` (план-файл сессии) и spike-вердикт `elementor-rework-spike-verdict.md`.

## Что сделано

- **Левая колонка** (`#wpgjb-blocks`): нативный GrapesJS `BlockManager`, смонтированный через `appendTo` (не своя модалка). Переключатель "Элементы"/"Блоки" — тонкий UI-слой поверх ОДНОГО инстанса менеджера через `BlockManager.render(filteredArray)` (не DOM-постпроцессинг категорий).
- **Новые "Элементы"** (`assets/editor/elements.js`, 9 типов): Заголовок/Текст/Кнопка (нативные `text`/`link`-типы GrapesJS-ядра, без своего `addType`), Разделитель/Видео/Контейнер/Колонки (generic `default`-тип), Отступ (единственный реальный новый `addType('spacer', …)` с number-trait), Изображение (нативный `image`-тип + оверрайд команды `open-assets` на `wp.media`).
- **26 существующих "Блоков"**: без изменений в манифест/слот-логике — теперь дополнительно зарегистрированы как реальные записи `BlockManager` (drag-and-drop вместо клика в удалённой модалке), категории — по `SECTION_TYPE_LABELS`.
- **Правая панель**: для Блоков — Контент (форма по манифесту, без изменений) + Настройки; для Элементов — Настройки(Style)+Свойства(Traits), обе смонтированы тем же паттерном, что уже использовался (`editor.StyleManager.render()`/`editor.TraitManager.render()` один раз, переиспользование DOM-узла).
- **StyleManager**: убран кастомный 3-секторный конфиг (нестандартные id) — GrapesJS падает на свои штатные сектора (`general/flex/dimension/typography/decorations/extra`), на стандартные id/свойства которых жёстко завязан `@silexlabs/grapesjs-css-variables`.
- **Критический фикс, вскрытый переходом**: внутренние обёртки повторяемых слотов (`[data-slot-repeat]`/`[data-slot-item]`, 16 из 26 блоков) не имели переопределения GrapesJS-типа — парсились как generic `droppable:true`, что после появления drag-and-drop сделало бы их валидной точкой дропа для произвольных Элементов (тихая порча структуры слотов). Исправлено добавлением `data-gjs-droppable="false"` во все 16 файлов `blocks-library/*/markup.html`.
- **RTE-guard** (`index.js`, `editor.on('rte:enable', …)`): найдено эмпирически, что GrapesJS активирует inline-редактирование по dblclick на ЛЮБОМ `text`-компоненте независимо от `resolveBlockRoot` — заголовки/абзацы внутри существующих Блоков получили бы contenteditable в обход формы "Контент". `view.disableEditing()`, вызванный синхронно внутри обработчика, не побеждает (GrapesJS выставляет `contentEditable` уже ПОСЛЕ события) — рабочий вариант: отложить вызов на следующий тик (`setTimeout(…, 0)`).

## Проверено (реальный браузер, не код-ревью)

- Изолированный спайк GrapesJS 0.22.16 (без WP): `data-gjs-droppable="false"` реально блокирует дроп; дефолтные сектора Style Manager — `general/flex/dimension/typography/decorations/extra`; `<h2>`/`<p>` распознаются нативным `text`-типом; object-content регистрация BlockManager с вложенными детьми и стилями работает корректно; `BlockManager.getAll()`/`render(filtered)` — рабочий, задокументированный механизм фильтрации.
- **В реальном редакторе** (WP Playground + Puppeteer): вкладка "Элементы" активна по умолчанию (9 элементов), вкладка "Блоки" показывает все 26 блоков в 18 категориях; вставка блока `services-grid` — корневой компонент `droppable:false`, с `data-wpb-block`; попытка дропа внутрь `[data-slot-repeat]`/`[data-slot-item]` того же блока — оба узла `droppable:false` (фикс подтверждён на РЕАЛЬНОЙ, а не тестовой разметке); вложенность Контейнер→Заголовок — родитель верно `droppable:true`, вложенный Заголовок не несёт `data-wpb-block`; клик по внутреннему заголовку блока корректно выделяет корень блока (панель показывает "Контент"); выделение свободного Элемента показывает Настройки+Свойства (без вкладки Контент).
- **RTE-скоуп** (до фикса — контрольный прогон, после фикса — итоговый): dblclick по заголовку ВНУТРИ блока `contentEditable` **false** (после фикса; было `true` до фикса — регрессия подтверждена и устранена), dblclick по свободному текстовому Элементу — `contentEditable` **true**, RTE-тулбар видим.
- **Изображения**: dblclick по image-слоту ВНУТРИ блока (`test-hero`) НЕ открывает `.media-modal`; dblclick по свободному Элементу "Изображение" ОТКРЫВАЕТ `.media-modal`.
- **Style Manager**: секции реально отрендерены как `General/Flex/Dimension/Typography/Decorations/Extra` (стандартные id GrapesJS) — структурно подтверждает фикс; визуальная проверка дропдауна токенов темы `grapesjs-css-variables` не дала данных на этом тестовом WP (у активной темы нет color/font presets — `dev/theme/tokens` вернул `{tokens: [], source: "none"}"`), это ограничение тестового окружения, не кода.
- **Публикация**: `project_data`+`html`+`css` с миксом Блока и Элементов опубликованы через реальный REST (`/documents/{id}/{type}/publish`, статус 200); фронтенд содержит `wpb-block--services-grid`; `data-gjs-droppable`/`data-wpb-block` атрибуты НЕ протекают на фронтенд (единственный путь генерации HTML — `editor.getHtml()` на уже распарсенном дереве компонентов, что стрипает служебные атрибуты).
- **Полная регрессия**: `dev/phase{1,2,5,6,7}`, `dev/second-tier`, `dev/checklist` — все PASS без изменений. `dev/snapshots/compare` — сначала ожидаемый FAIL (16 блоков с новым `data-gjs-droppable`), после перегенерации baseline (`tests/php/Snapshots/blocks/*.json`, 26 файлов) — PASS, 26/26 match.

## Осознанно вне объёма

- Реальный drag-and-drop мышью не симулировался через Puppeteer (нативный HTML5 DnD через границу iframe плохо поддаётся синтетическим событиям CDP) — вместо этого вставка проверялась через тот же публичный путь, которым пользуется само перетаскивание (`BlockManager` `content` → `editor.getWrapper().append()`), что покрывает всю логику ПОСЛЕ вставки (санитайзер, droppable, resolveBlockRoot, панель) — сам факт "элемент реально начинает перетаскиваться мышью" не автоматизирован, это стандартная, многократно проверенная в других проектах функциональность самого GrapesJS (`nativeDnD`), не наш код.
- Визуальное подтверждение дропдауна токенов темы в Style Manager — не получено на этом тестовом WP из-за отсутствия presets у активной темы; структурная часть фикса (правильные id секторов/свойств) подтверждена напрямую через `StyleManager.getProperty()`.
- Search-фильтр по блокам внутри вкладки "Блоки" — не реализован (26 блоков, управляемо через категории; поиск — возможное будущее улучшение, не входит в объём этого перехода).
