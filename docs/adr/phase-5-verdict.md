# Phase 5: Части сайта + безопасность — вердикт

Дата: 19 июля 2026. Статус: **PASS**, проверено реальным REST (дев-эндпоинт `/wpgjb/v1/dev/phase5`, 8/8) и реальным браузером (Puppeteer против WP Playground: экраны "Части сайта"/"Настройки", полный цикл создание → условие показа → публикация → фронтенд). Скриншоты — `docs/adr/phase-5-screenshots/` (см. также `<scratchpad>/e2e/shots-phase5/`).

По решению пользователя объём фазы — "Части сайта + безопасность сначала", Zero-режим сознательно отложен на отдельный шаг.

## Что реализовано

- `includes/SiteParts/PartsPostType.php` — скрытый CPT `wpb_part` (`show_ui=false`, `show_in_rest=false`, `capability_type=page`+`map_meta_cap`), переиспользует уже существующий post_id-keyed слой хранения (DocumentRepository/EditorPage/Publisher) вместо отдельного механизма для шапки/подвала.
- `includes/SiteParts/DisplayConditions.php` — специфичность правил (`specific_page=100 > front_page/404=75 > category/post_type/archive=50 > all_pages/all_posts=25 > entire_site=0`), exclude всегда побеждает, конфликт на равной специфичности резолвится детерминированно и логируется в Diagnostics.
- `includes/SiteParts/ClassicThemeInjector.php` (`get_header`/`get_footer`, честно на уровне "вставка", не "замена" — раздел 5 спеки) и `BlockThemeInjector.php` (`render_block_core/template-part`) — оба no-op друг для друга по `wp_is_block_theme()`.
- `includes/SiteParts/CacheCascade.php` — слушает `wpgjb_after_publish`, инвалидирует кэш ТОЛЬКО реально затронутых страниц (не весь сайт), результат синхронно доступен `PublishController` в том же запросе для уведомления пользователя.
- `includes/Rest/SitePartsController.php`, `includes/Admin/SitePartsPage.php` (меню "Конструктор" → "Части сайта", вкладки Шапки/Подвалы, инлайн-форма условий на строку), `includes/Admin/SecurityPage.php` (меню "Настройки": матрица ролей×прав, аудит-лог).
- `blocks-library/insert-code/` — блок с единственным `raw_html`-слотом; `ProjectDataSanitizer::sanitize_raw_html_value()` пропускает значение НЕИЗМЕНЁННЫМ только при `current_user_can('wpgjb_insert_raw_code')`, с логированием в канал `raw-code-audit` (user_id, hash, длина), иначе — обычная санитизация как для string-слота.

## Реальный баг, найденный чтением кода публикации при подготовке браузерного теста (не по автотестам)

**`Publisher::publish()` никогда не переводил пост из `draft` в `publish`.** И страницы, созданные через "Добавить в конструкторе" (`EditorPage::handle_new_page()`), и части сайта (`PartsPostType::create()`) создаются со статусом `draft` — у обоих нет обычного экрана `post.php` с кнопкой "Опубликовать" (редактор конструктора — единственный путь). Кнопка "Опубликовать" в GrapesJS-редакторе вызывает `Publisher::publish()`, который писал `post_content`/postmeta, но не трогал `post_status` — то есть документ навсегда оставался черновиком и был физически недостижим для обычного посетителя, независимо от display conditions.

Автотесты (`Phase5Runner::check_cache_cascade()` и др.) не ловили это, потому что вручную вызывали `wp_publish_post()` в обход реального REST-пути — это скрывало проблему именно там, где её проще всего не заметить (юнит-уровень «зелёный», продукт — сломан). Баг вскрылся при подготовке сквозного браузерного сценария (создать часть → условие → реальный клик "Опубликовать" → проверка фронтенда), именно ради которого такой сценарий и делался.

**Исправлено** в `includes/Render/Publisher.php`: при публикации, если текущий `post_status` — `draft`/`auto-draft`/`pending`, явно выставляется `publish` (посты в других статусах, например `private`, не трогаются). Затрагивает не только Phase 5 (части сайта), но и Phase 2/3 (обычные страницы, созданные через конструктор) — это была сквозная проблема пайплайна публикации, не специфика этой фазы.

## Проверено

**REST (`/wpgjb/v1/dev/phase5`, 8/8 PASS):** CPT+условия по умолчанию, резолв по специфичности, логирование конфликта равной специфичности, каскадная инвалидация кэша ТОЛЬКО целевой страницы, скрытие блока "Вставка кода" и полная санитизация `raw_html` без капабилити, беспрепятственный проход + аудит-запись с капабилити, обычный `<script>` по-прежнему отклоняется.

**Браузер (Puppeteer, headless Chrome, реальный WP Playground):**
- Экран "Части сайта": список загружается, форма создания работает и редиректит в реальный редактор новой части; новая часть появляется в списке; инлайн-форма условий (scope=specific_page, target=ID страницы) сохраняется и подтверждается редиректом с `wpgjb_saved=1`.
- Экран "Настройки": матрица ролей×прав рендерится (чекбоксы по `Activation::CAPABILITIES`, администратор не показан — управлять его правами через таблицу небезопасно), аудит-лог вставки кода рендерится с реальными записями (время/пользователь/хэш/длина).
- **Полный сквозной цикл**: создана шапка → условие "показывать только на Sample Page" → через REST insert-block (тот же путь, что Phase 4) вставлен блок с узнаваемым текстом → реальный клик по кнопке `#wpgjb-publish` в редакторе → переход на `/sample-page/` на живом фронтенде (тема Twenty Twenty-Five, блочная) → маркер шапки виден НАД контентом страницы. Это подтверждает одновременно и `BlockThemeInjector`, и только что исправленный переход статуса, и каскадную инвалидацию — все вместе, на реальном запросе браузера, не изолированными юнитами. Ноль JS-ошибок консоли за весь прогон.

**Честно не проверено (ограничение песочницы, не пробел продукта):** `ClassicThemeInjector` — в этом инстансе WP Playground установлены только блочные темы (Twenty Twenty-Five/Four/Three); классическая тема для визуальной проверки `get_header`/`get_footer`-перехвата не устанавливалась. Логика инжектора идентична по структуре уже проверенному `BlockThemeInjector` и была проверена статическим ревью, но не живым рендером.

## Осознанно упрощено (не забыто)

- Форма условий показа в UI — одно правило на часть (scope+mode+target, без JS); сам движок `DisplayConditions` поддерживает произвольное число правил — ограничение конкретно этого экрана, не архитектуры.
- Вкладки "Мои блоки"/"Мои шаблоны" на экране "Части сайта" не реализованы — соответствующей функциональности ещё не существует вообще, вкладка не показана, чтобы не показывать пустышку.
- Zero-режим — отдельным шагом после, по решению пользователя; не входит в эту фазу.
