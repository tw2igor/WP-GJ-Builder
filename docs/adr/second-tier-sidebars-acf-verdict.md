# Вторая очередь: Сайдбары + ACF-теги — вердикт

Дата: 19 июля 2026. Первый пункт бэклога после закрытия MVP (Phase 7). Статус: **PASS**, проверено реальным REST/браузером на живом WP Playground.

## Сайдбары как третий тип части сайта

Раздел 7 спеки + план изначально требовали строить движок обобщённо над "типом части", не хардкодить только шапку/подвал — это было сделано ещё в Phase 5. Добавление `sidebar` подтвердило это: `PartsPostType`, `DisplayConditions`, `DocumentsController::ALLOWED_TYPES`, `EditorPage`'s bootstrap-роутинг не потребовали ни единой правки — они уже были обобщены. Реально изменившиеся места:

- `includes/SiteParts/PartsPostType.php` — новая константа `PART_TYPES = ['header','footer','sidebar']`, единый источник правды (раньше список дублировался хардкодом в 3 разных местах).
- `includes/Admin/SitePartsPage.php` — третья вкладка "Сайдбары".
- `includes/SiteParts/ClassicThemeInjector.php` — хук `get_sidebar` (тот же паттерн, что `get_header`/`get_footer` — штатный шаблонный тег, срабатывает ТОЛЬКО при вызове `get_sidebar()` без имени; именованные сайдбары `get_sidebar('shop')` стреляют отдельным хуком `get_sidebar_{name}` — сознательно не покрыты первым проходом, тот же честный предел "вставка", не "замена").
- `includes/SiteParts/BlockThemeInjector.php` — `sidebar` добавлен в whitelist slug'ов template part. **Честная оговорка**: сработает только если у активной темы вообще есть template part с таким slug — в отличие от header/footer это НЕ универсально для блочных тем (многие FSE-темы вообще не имеют отдельного слота sidebar). В этой песочнице (Twenty Twenty-Five/Four/Three) такого слота нет ни у одной темы — визуальная проверка на реальной странице невозможна здесь по той же причине, по которой в Phase 5 не была визуально проверена `ClassicThemeInjector` (нет классической темы в окружении).
- `includes/SiteParts/CacheCascade.php` — убран хардкод `in_array($type, ['header','footer'])`, теперь `PartsPostType::PART_TYPES`.

**Реальный баг, найденный ПРИ добавлении sidebar (не специфичный для sidebar — существовал для header/footer тоже, с самого Phase 5):** `includes/Rest/SitePartsController.php` регистрировал маршрут `/parts/(?P<id>\d+)/conditions`, но переиспользуемый `permission_callback` (`DocumentsController::check_permission`) читает `$request->get_param('post_id')` — рассинхронизация имени параметра означала, что условия показа НИКОГДА не удавалось прочитать/сохранить через REST ни для одной части сайта, ни разу с Phase 5. Исправлено переименованием параметра маршрута в `post_id`.

**Проверено:** REST полный цикл (создать sidebar-часть → 404 на conditions до фикса → 200 после фикса → установить условие specific_page → опубликовать → `cascade: {type: "sidebar", affected_pages: 1}`, корректно scoped только к целевой странице). Браузер: экран "Части сайта" → вкладка "Сайдбары" показывает созданную часть со статусом и условием показа. Дев-раннер `/wpgjb/v1/dev/second-tier` — 7/7 PASS, включая регрессионный тест на сам баг параметра маршрута (реальный REST-диспетч, не интроспекция).

## ACF-теги

`includes/DynamicTags/TagRegistry.php` — новый тег `acf_field`, зарегистрированный **только если ACF реально активен** (`function_exists('get_field')`) — иначе tag_id честно отсутствует в реестре и `{{wpb:acf_field;...}}` резолвится в пустую строку тем же путём, что любой опечатанный тег (раздел 8 правило 3), а не "тег есть, но всегда пуст" непонятного происхождения.

**Архитектурное решение (не вызывает `get_field()`):** резолвер тега обязан оставаться чистой функцией без обращений к БД (инвариант `BatchResolver`, обеспечивающий "≤3 запроса вне зависимости от числа тегов") — вызов `get_field()` ACF мог бы стоить дополнительный запрос НА КАЖДОЕ РАЗНОЕ имя поля (резолв объекта поля/схемы), что сломало бы эту гарантию при нескольких разных ACF-полях на странице. Вместо этого `acf_field` читает СЫРОЕ значение из уже загруженного батчем `$context['post_meta']` (тот же группа/бюджет запросов, что `post_meta`) — корректно для простых скалярных типов (text/textarea/number/url/email/одиночный select), и явно фильтрует сериализованные значения (repeater/gallery/relationship и т.п.) в пустую строку, а не выводит `serialize()`-мусор в разметку.

**Честно вне объёма этого прохода:** форматирование ACF (true_false→"Да"/"Нет", select→label вместо raw value, кастомный date_format) и сложные типы (image/gallery/relationship/repeater/flexible content) — все они требуют либо резолва объекта поля (риск для query-бюджета), либо структурного разбора (не текстовая подстановка). ACF не установлен в этой песочнице — реальное поведение с активным плагином эмпирически не проверялось, только логика резолвера (юнит-проверка на сконструированном `$context`) и корректность условной регистрации.

**Проверено:** `/wpgjb/v1/dev/second-tier` — тег отсутствует в реестре (ACF неактивен, как и есть на самом деле в этой среде), резолвер корректно читает простое значение и фильтрует сериализованное — оба через прямой вызов `TagRegistry::resolve_acf_raw_field()` с сконструированным контекстом, независимо от реального присутствия ACF.
