# Phase 3: Editor MVP (блочный режим) — вердикт

Дата: 19 июля 2026. Статус: **PASS**, проверено интерактивным реальным браузером (headless Chrome через `puppeteer-core`, реальные клики/ввод текста — не только curl/REST) поверх реального WordPress (WP Playground), плюс независимая проверка опубликованной страницы обычным `curl` как анонимный посетитель. Скриншоты — `docs/adr/phase-3-screenshots/`.

## Что реализовано

- `includes/Admin/EditorPage.php` — скрытая полноэкранная страница (`add_submenu_page(null, ...)`), капабилити-гейт, экранно-гейтированный enqueue редакторского бандла, штатная WP-блокировка (`wp_check_post_lock`/`wp_set_post_lock`), row action «Редактировать в конструкторе» + кнопка «Добавить в конструкторе» на списке страниц.
- `includes/Rest/ThemeStylesController.php` — CSS активной темы + CSS платформенных блоков + JSON-токены для canvas.
- `includes/Blocks/BlockLibrary.php` — загрузчик каталога `blocks-library/*` (манифест + markup + style), общий для бутстрапа редактора и будущего REST-каталога (Phase 4).
- `blocks-library/{test-hero,test-cta,test-text}/` — 3 тестовых блока (скалярные слоты + один с richtext + один с array/image), реальные манифесты из Phase 1.
- `assets/editor/{index,blocks,api,lock,editor.css}.js` — GrapesJS-редактор: canvas с реальным CSS темы (конфиг из вердикта спайка 1), манифест-управляемые компоненты (Content-форма генерируется из `slots[]`, Settings — нативный Style Manager GrapesJS, ограниченный `style_whitelist`), автосохранение (троттлинг 20с) с optimistic-locking конфликт-промптом, Heartbeat-блокировка редактирования, публикация через Phase 2 REST.
- `includes/Sanitize/ProjectDataSanitizer.php` — контур 1 (раздел 10): каждое сохранение прогоняет значения слотов и CSS-правила project_data через уже проверенный `Sanitizer` (Spike 4), подключено в `DocumentsController::save_document()`.

## Реальные баги, найденные ТОЛЬКО благодаря интерактивному браузерному тесту (не curl, не код-ревью)

REST-эндпоинты Phase 1/2 были полностью покрыты curl-тестами и прошли ещё в своих фазах — все нижеперечисленные баги проявлялись ИСКЛЮЧИТЕЛЬНО в реальном браузере:

1. **`Content-Type: application/json` на CSS-эндпоинте** — `WP_REST_Server` форсирует JSON-сериализацию тела ответа независимо от вручную выставленного заголовка; браузер отказывался применять такой `<link rel="stylesheet">` (strict MIME checking). Исправлено через `rest_pre_serve_request` (штатный способ отдать не-JSON тело из REST API).
2. **REST cookie-авторизация несовместима с `<link>`-тегами** — `rest_cookie_check_errors()` требует `X-WP-Nonce` для ЛЮБОГО запроса (не только небезопасных методов), а `<link>` физически не может нести кастомный заголовок → гарантированный 401 для canvas.styles. Реальной уязвимости в решении нет: сделали `theme/global-styles.css` и новый `blocks/style.css` публичными (`__return_true`) — это те же самые CSS-переменные/классы, что тема и так публично отдаёт каждому посетителю сайта.
3. **CSS не хватало** — забыт REST-эндпоинт CSS платформенных блоков (`blocks-library/*/style.css`) в `canvas.styles`; без него вставленный блок в canvas выглядел совсем неоформленным.
4. **`[hidden]` не работал** — `.wpgjb-picker`/`.wpgjb-lock-banner` объявляли `display: flex` с той же CSS-специфичностью, что и `[hidden]` в UA-стиле, и побеждали его по порядку каскада — пикер блоков был виден ПОСТОЯННО, с самой первой загрузки редактора. Добавлен явный `[hidden] { display: none !important; }`.
5. **Клик по внутреннему элементу блока (`<h1>` и т.п.) выбирал ЕГО САМОГО, а не родительский блок** — GrapesJS по умолчанию парсит КАЖДЫЙ вложенный элемент markup-шаблона как свой отдельный выбираемый компонент; панель Контент/Настройки при этом не находила манифест (он привязан к корню блока) и показывала «этот элемент не редактируется отдельно». Исправлено: при выборе поднимаемся по родителям (`resolveBlockRoot`) до компонента с `data-wpb-block` и переселяем выделение GrapesJS туда же.
6. **Автосохранение падало с HTTP 400 на новой странице** — REST-схема объявляет `doc_version` как `integer`, а клиент для документа без предыдущих сохранений отправлял буквальный JSON `null` (не отсутствие ключа) — WP отклонял как невалидный тип. Исправлено на клиенте: ключ `doc_version` теперь просто отсутствует в теле запроса, если версии ещё нет, а не передаётся как `null`.
7. (Некритично, но найдено тем же прогоном) `style_whitelist` в манифесте test-hero изначально содержал плейсхолдерные имена кастомных CSS-переменных (`--wpb-bg-color` и т.п.) вместо реальных имён CSS-свойств — приведено в соответствие с реальными свойствами (`background-color`, `padding` и т.д.) и с явной конфигурацией секторов Style Manager в `index.js`.

## Проверено интерактивно (полный пользовательский сценарий, один прогон)

1. Вход как `admin`, открытие `admin.php?page=wpgjb-editor&post_id=X&type=page` — редактор грузится, canvas показывает реальный CSS темы, ошибок в консоли нет.
2. Клик «+ Добавить блок» → пикер показывает 3 блока с реальными `manifest.purpose` → выбор `hero` → блок вставлен, стилизован (пилюля-кнопка, центрирование, карточка пункта — всё как в `style.css`).
3. Клик по заголовку блока в canvas → выделяется БЛОК ЦЕЛИКОМ (не текстовый узел) → открывается панель «Контент» с формой по `slots[]`.
4. Правка поля `title` → **тот же текст мгновенно обновляется в canvas** (двусторонняя связь форма↔canvas через `component.components()`).
5. Переключение на «Настройки» → рендерится нативный Style Manager GrapesJS, секторы ограничены `style_whitelist` (Цвет и фон / Текст / Отступы — не полный дефолтный набор GrapesJS).
6. Применение стиля (`background-color`, `text-align`) → отражается в `editor.getCss()`.
7. Ожидание троттлинга автосохранения (~20с) → индикатор «Сохранено»; **проверено дополнительно реальным REST GET** (не только по UI-индикатору): `doc_version=1`, сохранённый `project_data` содержит новый текст.
8. Клик «Опубликовать» → **проверено реальным анонимным `curl`** на фронтенд-URL: заголовок блока и применённый стиль (`background-color: rgb(255, 0, 0)`, `text-align: center`) присутствуют в HTML/CSS, отданном обычному посетителю — canvas и фронтенд идентичны.
9. Второй пользователь (`reviewer`, отдельно созданный REST-администратор, отдельный изолированный browser context — не то же самое, что "тот же пользователь в двух вкладках", когда `wp_check_post_lock()` намеренно не считает это конфликтом) открывает ТУ ЖЕ страницу → баннер «Страницу сейчас редактирует admin. Ваши изменения не будут сохранены.» + кнопка «Опубликовать» задизейблена.

## Осознанно упрощено в Phase 3 (не забыто, задокументировано)

- **Изображения**: слот хранит ID вложения как обычное число (текстовое поле), без drag-and-drop `wp.media`-пикера — контракт (ID, не URL) совпадает с уже проверенным в Phase 1 `ManifestValidator`, полноценный picker — понятный follow-up.
- **Rich-text**: обычная `<textarea>` с сырым HTML small-tag-набора (без WYSIWYG-тулбара) — контур 1 фильтрует на сохранении, полноценный тулбар — follow-up.
- **`component.stylable` не полностью ограничивает набор properties Style Manager сам по себе** — реальное ограничение сейчас достигается явной конфигурацией `styleManager.sectors` в `index.js` (общей для всех блоков), а не per-block через `stylable`; точное per-block ограничение свойств — уточнить в Phase 4/6.
- **"+" вставляет только в конец страницы**, не между конкретными блоками (нет hover-триггеров между блоками) — реордер уже возможен через ↑/↓ в тулбаре блока.
- **Конфликт-промпт при 409** — упрощённый (загрузить чужую версию / перезаписать своей), без полноценного визуального диффа — раздел 5.5 просит "предложить сравнение", здесь это сведено к двум явным кнопкам, не потеряно молча.
- Дев-only `window.__wpgjbEditor` (только для E2E-отладки) — не влияет на продакшен-поведение, можно оставить или убрать перед Phase 7 без разницы.
