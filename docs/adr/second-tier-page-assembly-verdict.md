# Вторая очередь: Сборка страницы из последовательности блоков — вердикт

Дата: 19 июля 2026. Второй пункт бэклога после закрытия MVP. Статус: **PASS**, проверено реальными REST-запросами и браузерным round-trip на живом WP Playground.

## Что реализовано

`includes/Rest/BlocksCatalogController.php` — `POST /wpgjb/v1/pages/assemble` заменяет 501-заглушку (была явно заложена в Phase 4 как точка расширения раздела 9, "можно застабить как 501"). Принимает `{title, blocks: [{block_id, slots}, ...]}`, создаёт НОВУЮ страницу за один вызов — для AI/внешнего клиента, которому неудобно делать N отдельных `insert-block` на ещё не существующую страницу.

Валидация/санитизация/рендер каждого элемента последовательности — ТЕ ЖЕ методы (`ManifestValidator::validate_values`, `ProjectDataSanitizer::sanitize_values`, `BlockRenderer::render_component`), что уже использует `insert_block()` — не вторая реализация правил (раздел 9 П7: человеческий и AI-путь дают идентичный формат `project_data`).

**Ключевое архитектурное решение — атомарность:** ВСЕ элементы последовательности валидируются ДО создания страницы. Если хоть один элемент невалиден (неизвестный `block_id`, не заполнено обязательное поле и т.п.), страница не создаётся вообще, а клиент получает ошибки СРАЗУ по всем проблемным элементам одним ответом (с индексом каждого), не по одной ошибке за раунд-трип — это лучше и для AI-клиента (может исправить весь батч за один проход), и не оставляет полусобранных черновиков-мусора при частичном сбое.

## Проверено

**Реальными REST-запросами (не только unit-тестом):**
- Пустой `blocks: []` → 422.
- Последовательность из 2 валидных блоков (`test-hero` + `test-cta`) → 201, реально создана страница (`post_id`), `project_data` содержит оба компонента.
- Смешанная последовательность (1 валидный + неизвестный `block_id` + блок без обязательного поля) → 422 с ошибками по индексам 1 и 2; **счётчик страниц до/после вызова не изменился** — подтверждает, что валидный элемент в невалидной последовательности НЕ создал частичный артефакт.

**Браузерный round-trip:** страница, собранная вызовом `/pages/assemble`, открыта в реальном GrapesJS-редакторе — оба блока на месте с правильным контентом ("Assembled Hero"/"Assembled CTA"), редактор её нормально сохраняет и публикует (кнопка "Опубликовать" отработала штатно).

**Регрессия:** добавлено 3 постоянных проверки в `tests/php/SecondTier/SecondTierRunner.php` (дев-эндпоинт `/wpgjb/v1/dev/second-tier`, теперь 10/10) — покрывают все три сценария выше через реальный REST-диспетч (`rest_do_request`), не интроспекцию. Полная регрессия по всем предыдущим фазам (`phase1/2/5/6/7`, `snapshots`) без изменений.

## Осознанно вне объёма

- Вставка НЕ в новую, а в СУЩЕСТВУЮЩУЮ страницу целой последовательностью (сейчас `assemble` создаёт только новую страницу) — для этого уже есть `insert-block` по одному блоку за вызов; не дублируется здесь.
- Порядок/логика "рекомендуемых следующих блоков" (`manifest.context.recommended_after`) не используется для автоматической подсказки последовательности — эндпоинт принимает последовательность как есть, не предлагает её сам.
