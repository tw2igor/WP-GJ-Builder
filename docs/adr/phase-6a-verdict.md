# Phase 6a: Надёжность + очистка — вердикт

Дата: 19 июля 2026. Статус: **PASS**, проверено реальным REST (дев-эндпоинт `/wpgjb/v1/dev/phase6`, 5/5, плюс регрессия phase1/phase2/phase5 — все PASS) и реальным браузером (Puppeteer против WP Playground), включая единственную по-настоящему разрушительную операцию фазы — "Полная очистка сейчас" — выполненную по-настоящему, не смоделированную. Скриншоты — `docs/adr/phase-6a-screenshots/`.

По решению пользователя из большого списка Phase 6 (раздел 11/12 спеки) в эту сессию взят приоритетный кусок: **надёжность + очистка**. Реальные изображения (`wp.media`, srcset/lazy/fetchpriority), инфраструктура per-page JS/CSS ассетов и замер целевых метрик раздела 11 — сознательно отложены на Phase 6b.

## Что реализовано

- `includes/Blocks/BlockFaultTolerance.php` — карантин "одного повреждённого/неизвестного блока" (раздел 12): проходит дерево `project_data`, заменяет структурно испорченные узлы (не массив там, где ожидался) и узлы со ссылкой на удалённый из библиотеки `data-wpb-block` невидимой заглушкой (`display:none`), логирует каждый случай в Diagnostics (канал `block-fault-tolerance`). Это НЕ про целый документ — тот случай уже был покрыт `DocumentRepository::STATUS_FROZEN` (невалидный JSON/неудачная миграция) до этой фазы; здесь — один узел среди иначе валидных.
- Подключено в ДВУХ местах: `Publisher::publish()` (страница публикуется, даже если один блок был удалён из библиотеки между вставкой и публикацией) и `DocumentRepository::get_for_edit()` (тот же документ открывается в редакторе без падения на неизвестном компоненте). `PublishController` теперь возвращает `quarantined` в ответе публикации — та же идея, что и `cascade` из Phase 5.
- `includes/Storage/RetentionPolicy.php` — настраиваемый лимит хранения (`wpgjb_revision_retention_count`, по умолчанию 10) для истории публикаций документа; **текущая опубликованная версия никогда не удаляется** — лимит применяется только к строкам `is_current=0`. Плановая очистка — `wp-cron` (`wpgjb_prune_revisions`, ежедневно), включается/выключается вместе с активацией/деактивацией плагина. Дополнительно: `before_delete_post` теперь чистит ВСЕ строки документов удалённого поста (раньше не было реализовано вовсе — строки переживали удалённую страницу без какого-либо способа до них добраться).
- `includes/Storage/Cleanup.php` + честный `uninstall.php` — раздел 12: "полная очистка (таблицы, опции, cron-задачи, транзиенты) через штатный uninstall-механизм". Ранее `uninstall.php` был заглушкой с `TODO (Phase 6)`. Реализация НЕ трогает `post_content` обычных страниц (готовый HTML остаётся читаемым без плагина) — удаляет только таблицу `wpb_documents`, посты `wpb_part` (части сайта — собственные данные плагина без публичного URL), postmeta `_wpb_built`/`_wpb_page_css`, опции плагина, cron, будущие `_transient_wpgjb_*`.
- Экран "Настройки" → новая секция "Хранение и очистка": число хранимых публикаций, галочка "удалить данные при удалении плагина" (по умолчанию выключена — тот же `Cleanup`, вызываемый из `uninstall.php`), и отдельная явная кнопка "Полная очистка сейчас" (раздел 12 явно требует ОБА механизма) — доступна, пока плагин активен, с `confirm()`-подтверждением и правильным различием: ручная кнопка пересоздаёт пустую таблицу сразу же (`DocumentsTable::maybe_upgrade()`), чтобы плагин остался рабочим; `uninstall.php` этого не делает (плагин целиком удаляется следом).

## Проверено

**REST (`/wpgjb/v1/dev/phase6`, 5/5 PASS):**
- 12 публикаций одного документа + прунинг с лимитом 5 → осталось 6 строк (5 истории + 1 текущая, текущая не тронута).
- Удаление поста через `wp_delete_post()` полностью очищает связанные строки документов (`before_delete_post`).
- Публикация с одним неизвестным (`does-not-exist-block-id`) блоком не бросает исключение, возвращает непустой `quarantined`, пишет инцидент в Diagnostics.
- Публикация со структурно испорченным узлом (строка вместо массива среди `components`) переживает это без падения.
- Документ с уже отсутствующим в библиотеке блоком открывается в редакторе (`status=ok`, не `frozen`) — проблемный узел заменён заглушкой.

**Регрессия:** Phase 1/2/5 дев-эндпоинты по-прежнему PASS после всех изменений.

**Браузер (Puppeteer):** секция "Хранение и очистка" рендерится и сохраняет значение (7 после сохранения, подтверждено перезагрузкой страницы). Кнопка "Полная очистка сейчас" нажата по-настоящему (не смоделирована): диалог `confirm()` перехвачен и подтверждён, редирект с уведомлением об успехе, **ноль фатальных ошибок**, и — ключевая проверка — сразу после очистки создание новой страницы через "Добавить в конструкторе" и открытие её в редакторе работает штатно (таблица документов пересоздана пустой автоматически). Регрессионный прогон REST-эндпоинтов phase1/phase2/phase5/phase6 после очистки — все снова PASS на свежих данных.

## Осознанно упрощено (не забыто)

- Регрессионные снимки версий блоков (раздел 12: "релиз обновления библиотеки тестируется на сохранённых JSON-снимках предыдущих версий каждого блока") — не реализованы в этой сессии, входят в оставшуюся часть Phase 6.
- Версионирование схемы НА УРОВНЕ ОТДЕЛЬНОГО ЭКЗЕМПЛЯРА блока (в отличие от уже существующего документо-уровневого `schema_version`) — тоже не реализовано; текущий карантин решает соседнюю, но более узкую задачу ("неизвестный/испорченный узел", а не "устаревшая версия конкретного блока").
- "Полная очистка сейчас" использует нативный `confirm()` вместо полноценного модального компонента — раздел 13 требует модальные подтверждения только для необратимых действий, это ровно тот случай, но полноценная модалка (как в `@wordpress/components`) — отдельная работа над UI-стандартами, не блокирует надёжность/очистку по сути.
