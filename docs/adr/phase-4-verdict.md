# Phase 4: Каталог библиотеки блоков и REST API — вердикт

Дата: 19 июля 2026. Статус: **PASS**, проверено реальным REST (curl, включая сценарий "внешний клиент без знания GrapesJS"), реальным браузером (round-trip в редакторе) и визуально на 3 разных темах. Скриншоты — `docs/adr/phase-4-screenshots/`.

## Что реализовано

- `includes/Blocks/BlockRenderer.php` — серверный (PHP/DOMDocument) порт клиентской логики `assets/editor/blocks.js`: применяет значения слотов к markup-шаблону блока, возвращает GrapesJS-совместимый узел компонента (`components` как строка внутреннего HTML — тот же формат, что и при вставке блока человеком через "+").
- `includes/Rest/BlocksCatalogController.php` — `GET /wpgjb/v1/blocks` (список+фильтры), `GET /wpgjb/v1/blocks/{id}` (блок целиком), `POST /wpgjb/v1/documents/{post_id}/{type}/insert-block` (вставка с валидацией через `ManifestValidator` + санитизацией через `ProjectDataSanitizer::sanitize_values()` — теперь публичный метод, переиспользуется, не дублируется), `POST /wpgjb/v1/pages/assemble` (застаблен 501 — явно вторая очередь).
- `includes/Blocks/ChecklistLinter.php` — автоматизируемая часть чек-листа раздела 6 (манифест заполнен, нет зашитых hex/font-family мимо `var()`-фолбэка, `data-slot-href` только на `<a>`, `alt` у `<img>`); визуальные/ручные пункты (3 темы, адаптивность, нейтральные заглушки) сознательно не симулируются программной проверкой.
- 8 новых блоков (`test-features/-pricing/-testimonials/-team/-gallery/-faq/-steps/-contacts`) — итого **11** блоков, по одному на большинство смысловых категорий раздела 6.
- Пикер блоков в редакторе обновлён: группировка по категориям (раздел 6: "Обложка, О компании, Преимущества..."), поиск по названию/категории/purpose.

## Реальный баг, найденный ТОЛЬКО кросс-тематическим визуальным прогоном

**Опубликованная страница не содержала базовый CSS блоков вообще** — `editor.getCss()` отдаёт исключительно правила, которые GrapesJS сам создал через свой внутренний CssComposer (переопределения из Style Manager); `style.css` каждого блока подключается в canvas отдельно как обычный `<link>` (вердикт спайка 1) и GrapesJS о нём ничего не знает — в `getCss()` его нет. Ни curl-тесты Phase 2 (ручной css-параметр), ни round-trip тест Phase 3 (там стиль был виден только благодаря вручную применённому `addStyle()` поверх, а не базовому CSS блока) этого не поймали — баг проявился только когда я визуально сравнил опубликованную страницу на 3 темах и увидел полностью неоформленный HTML. Исправлено: `Publisher::publish()` теперь явно собирает `style.css` каждого блока, реально присутствующего в `project_data` (обходом дерева компонентов по `data-wpb-block`), и прогоняет через контур 2 вместе с пользовательским CSS.

## Проверено

**REST-каталог как внешний клиент (curl, без единого JS):**
- `GET /blocks` — список из 11 блоков с манифестами.
- `GET /blocks/test-cta` — манифест + markup + style целиком.
- `POST .../insert-block` с `{block_id, slots}` — блок вставлен в черновик со заполненными значениями, санитизирован, провалидирован (422 с понятной ошибкой на невалидные слоты — проверено дважды: не хватает обязательного поля, не хватает `min_items` для массива).
- `POST /pages/assemble` — корректный 501.

**Round-trip в реальном браузере:** блок, вставленный ТОЛЬКО через REST (без открытия редактора), открыт в редакторе — стилизован идентично блоку, вставленному через UI, выделяется целиком (не как чужеродный элемент), форма "Контента" предзаполнена реальными значениями из REST-вызова. Один и тот же формат project_data для AI-пути и человеческого пути подтверждён эмпирически, не только по дизайну кода.

**Чек-лист (раздел 6):** линтер проверен на заведомо неправильном тестовом блоке (зашитый hex, зашитый font-family, `data-slot-href` на `<div>`, `<img>` без `alt`, короткий `purpose`, пустые `tags`) — поймал все 6 нарушений; все 11 реальных блоков проходят чисто.

**3 контрастные темы (Twenty Twenty-Five, Twenty Twenty-Four, Twenty Twenty-Три — все блочные; классическая тема отдельно не переустанавливалась в этом прогоне, но canvas-механизм для классических тем уже проверен спайком 1):** страница из всех 9 новых блоков опубликована и визуально сверена на всех трёх — структура и стилизация (пилюли-кнопки, сетки, границы карточек) сохраняются, типографика и оттенок фона меняются вместе с темой (Twenty Twenty-Four показывает засечковый шрифт темы вместо системного — подтверждает наследование шрифта, а не жёстко заданный).

## Осознанно упрощено (не забыто)

- Изображения по-прежнему без реального resolve (ID вложения без src) — как и в Phase 3, полноценный `wp.media`-picker и resolve в `<img src>` остаются отложенным follow-up.
- CSS блоков публикуется ЦЕЛИКОМ для каждого использованного блока (не per-instance diff) — оптимизация по разделу 11 ("резолвится один раз в postmeta") — задача Phase 6, здесь важна корректность, не экономия байт.
- "Мои блоки"/"используемые на сайте" фильтры в пикере не реализованы — нет ещё функциональности личной библиотеки (раздел 7 Phase 5).
