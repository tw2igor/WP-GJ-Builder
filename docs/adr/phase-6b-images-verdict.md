# Phase 6b (часть 1): Реальные изображения — вердикт

Дата: 19 июля 2026. Статус: **PASS**, проверено реальной загрузкой изображения через REST, реальной вставкой блока с этим вложением, и реальным браузером (Puppeteer): открытие wp.media, выбор вложения, применение к канвасу и панели. Скриншоты — `docs/adr/phase-6b-images-screenshots/`.

Из большого списка Phase 6 в эту итерацию взят самый заметный визуальный пробел продукта: image-слоты раньше хранили ТОЛЬКО ID вложения в `data-slot-img-id` — ни в редакторе, ни на опубликованной странице `<img>` никогда не получал `src` (подтверждено чтением кода ДО начала работы: `BlockRenderer.php`, `blocks.js`, все markup-шаблоны с image-слотами — ни разу не был реализован путь ID → реальная картинка).

## Что реализовано

- `includes/Admin/EditorPage.php` — `wp_enqueue_media()` в `maybe_enqueue()` (единственная PHP-функция, нужная чтобы `wp.media()` вообще заработал на экране редактора).
- `assets/editor/images.js` (новый модуль) — резолв ID вложения → `{src, srcset, sizes, alt, thumbnail}`: синхронно из уже загруженной Backbone-модели wp.media (сразу после выбора в пикере — без REST round-trip), либо асинхронно через `GET wp/v2/media/{id}` (открытие уже сохранённого документа).
- `assets/editor/index.js` — `renderImageField()`: настоящий `wp.media()`-пикер (превью-миниатюра, "Выбрать/Заменить изображение", "Убрать") вместо числового `<input>` с ID вложения. Значение слота по-прежнему сам ID — контракт с санитайзером и REST не менялся, только редакторский UI поверх него.
- `assets/editor/blocks.js` — `applyResolvedImagesToComponent()`: после вставки блока/изменения значений слотов резолвит все `[data-slot-img-id]` в уже живом дереве GrapesJS-компонентов (кэш → REST fallback), пишет `src/srcset/sizes/alt/loading|fetchpriority` через `component.addAttributes()` — то есть именно то представление, которое `editor.getHtml()` сериализует при публикации (архитектурный инвариант: HTML печётся на клиенте один раз, сервер его не пересобирает).
- `includes/Blocks/BlockRenderer.php` — `apply_image_attributes()`: тот же резолв для серверного (REST/AI) пути вставки блока, через штатные `wp_get_attachment_image_src()` + `wp_calculate_image_srcset()`/`wp_calculate_image_sizes()` (та же пара функций, которую ядро использует внутри `wp_get_attachment_image()`).
- Раздел 11: `loading="lazy"` по умолчанию; для слотов с манифестным `"first_screen": true` — `fetchpriority="high"` и БЕЗ `loading` (реализовано в обоих путях, JS и PHP, читает одно и то же поле манифеста).
- `ManifestValidator.php` — добавлена валидация `first_screen` (должен быть булевым, если указан) — ранее это метаполе принималось, но никак не проверялось и нигде не потреблялось.
- Исправлен сиротский слот: `blocks-library/test-hero/manifest.json` объявлял image-слот с `first_screen: true`, но в `markup.html` не было НИ ОДНОГО элемента с `data-slot="image"` — слот был физически недостижим. Добавлен `<img data-slot="image">` + стили (в т.ч. `:not([src]) { display: none }`, чтобы пустой слот не показывал битую иконку).

## Реальный баг, найденный ТОЛЬКО живым браузерным прогоном (не по коду)

**Наш webpack-бандл глушил `window.Backbone` ядра WP.** GrapesJS зависит от `backbone@1.4.1` как от обычной npm-зависимости; исходник `backbone.js` в СВОЁМ UMD-враппере безусловно пишет себя в `root.Backbone` при выполнении модуля — это поведение самого Backbone, не следствие webpack. Поэтому наш `editor.js` при загрузке молча подменял `window.Backbone` ядра (обычно 1.6.x) на бандленную GrapesJS-копию (1.4.1) — и `wp.media()` (Медиатека) продолжала ОТКРЫВАТЬСЯ (обёртка модалки уже была привязана к оригинальному Backbone на момент определения классов), но её content-регион (`AttachmentsBrowser` — сетка вложений/вкладки) переставал рендериться СОВСЕМ, без единой ошибки в консоли — тихий частичный отказ, который никакой статический разбор кода не поймал бы.

Диагностировано сравнением `window.Backbone.VERSION` на обычном экране "Медиатека" (`1.6.1`) и на экране редактора конструктора (`1.4.1` — после загрузки нашего бандла). **Исправлено** явным вызовом `window.Backbone.noConflict()` в самом начале `assets/editor/index.js`, сразу после того как импорты (включая GrapesJS) уже выполнились: это возвращает `window.Backbone` к оригиналу ядра, не затрагивая сам GrapesJS — он использует СВОЮ импортированную ссылку на Backbone из бандла напрямую, а не глобал, поэтому его код не читает `window.Backbone` повторно после инициализации.

## Проверено

**Сервер (реальная загрузка + REST-вставка):** сгенерирован настоящий JPEG 1600×900, загружен через `POST /wp/v2/media` (ядро сгенерировало реальные под-размеры 300/768/1024/1536/1600). **Браузер (Puppeteer, после фикса Backbone):**
- Канвас редактора реально ЗАГРУЗИЛ картинку (`img.complete === true`, реальные атрибуты `src/srcset/sizes/fetchpriority`), не заглушку.

## Осознанно НЕ реализовано в этой итерации

- **WebP-фолбэк** (раздел 11: "WebP с фолбэком на уровне загрузки"). Проверено эмпирически: реальная загрузка JPEG на этом инстансе WP НЕ создала WebP-сиблингов автоматически ни для одного под-размера — значит, автогенерация современных форматов здесь не включена по умолчанию. Включение через `wp_upload_image_mime_transforms`/аналогичные фильтры ядра и последующее потребление `sources['image/webp']` в `<picture>`-разметке — отдельная, ещё не сделанная работа; не стал угадывать точное поведение без эмпирической проверки на реальной установке.
- Апгрейд УЖЕ существующих (до этой фазы) сохранённых документов с `data-slot-img-id`, но без `src`, — подхватывается автоматически ТОЛЬКО при следующем открытии/редактировании этого слота (асинхронный резолв на загрузке документа не реализован специально для КАЖДОГО компонента дерева при открытии — только для вновь вставляемых/изменяемых). Для дев-библиотеки блоков без реальных пользователей это не проблема.
