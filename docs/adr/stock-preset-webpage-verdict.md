# Переход на стоковый GrapesJS (grapesjs-preset-webpage) — вердикт

Дата: 20 июля 2026. Статус: **PASS**, проверено реальным браузером (WP Playground + Puppeteer), не только код-ревью.

## Контекст

Пользователь отверг оба предыдущих прохода (Tilda-модалка, затем "Elementor-стиль" 3-колоночный кастомный layout): *"Ты срезал весь функционал GrapesJS... нужно взять весь их полноценный конструктор... не надо упрощать или делать похожим на Tilda/Elementor"*. Решение — подключить `grapesjs-preset-webpage` (официальный бесплатный community-пресет, то же что на demo grapesjs.com) КАК ЕСТЬ, без кастомных панелей, плюс 3-уровневая иерархия вставляемого контента (Элементы → Блоки → Страницы) с прицелом на будущую AI-сборку страниц.

## Что сделано

- **`grapesjs-preset-webpage@1.0.3`** подключён как GrapesJS-плагин. Пресет при загрузке вызывает `Panels.getPanels().reset([...])` — жёстко пересоздаёт панели `commands`/`devices-c`/`options`/`views` внутри своего контейнера, это не настраивается опциями.
- **WP-специфичный хром** (статус сохранения, ссылка "Назад", заголовок, баннер лока/конфликта) вынесен в СИБЛИНГ `.wpgjb-status-strip`/`#wpgjb-lock-banner` вне GrapesJS-контейнера (`#wpgjb-gjs-root`) — `Panels.reset()` его не трогает. Publish/Экспорт-JSON/Импорт-JSON добавлены КАК КНОПКИ в панель `options` пресета через `editor.Panels.addButton()`, не отдельным header.
- **Три уровня вложенности** в одном BlockManager (категории аддитивны, без конфликта со стоковыми блоками пресета — `pluginsOpts.blocks: []` явно подавляет 3 демо-блока link-block/quote/text-basic, чтобы они не путались с нашими тремя уровнями):
  - **Элементы** — не изменились (`elements.js`, 9 типов).
  - **Блоки** — не изменились (`blocks.js` + `registerBlocksInBlockManager` в `index.js`, 26 манифест-блоков).
  - **Страницы** (новое) — `templates-library/{id}/manifest.json` (та же форма `{title, blocks:[{block_id, slots}]}`, что уже принимает `POST /pages/assemble`), новый `includes/Blocks/TemplateLibrary.php` (зеркало `BlockLibrary.php`), новый `GET /wpgjb/v1/templates` в `BlocksCatalogController.php` (для будущего AI-модуля отдельно от бутстрапа редактора), новый `assets/editor/templates.js` — собирает HTML шаблона НА КЛИЕНТЕ через `renderBlockHtml()` по каждому блоку последовательности (не сервером — иначе вставленные блоки стали бы мёртвым HTML без `data-wpb-block`, нередактируемым формой "Контент" и неинспектируемым AI).
- **Форма "Контент"** (безопасное редактирование слотов манифеста) переехала из своей правой панели в кнопку-трейт "Редактировать контент…" (`registerBlockType`, blocks.js) → `editor.Modal.open()` со старой логикой без изменений, вынесенной в новый `assets/editor/content-form.js`.
- Стоковые Style/Trait/Layer Manager панели — больше не монтируются вручную, полностью отданы пресету.

## Проверено (реальный браузер)

- **Совместимость версий**: `grapesjs@0.22.16` (тестировался автором пресета на `^0.21.2`, несовместимость не была гарантирована формально) — реально работает без ошибок консоли; все 5 панелей (`commands/devices-c/options/views/views-container`) рендерятся; девайс-свитчер виден и работает "из коробки" (расходится с чтением исходников пресета, которое предполагало `showDevices=false` по умолчанию — эмпирика победила: реально `visible:true`).
- Наши кнопки ("Опубликовать"/"Экспорт страницы в файл (.json)"/"Импорт страницы из файла (.json)") реально присутствуют в панели `options` РЯДОМ со стоковыми (View components/Preview/Fullscreen/View code/Undo/Redo/Import/Clear canvas).
- Сиблинг-DOM (`.wpgjb-status-strip`) реально не тронут `Panels.reset()`.
- **36 блоков** в BlockManager = 9 Элементов + 26 Блоков + 1 Страница, сгруппированы в **20 категорий** (включая "Элементы" и "Страницы" как полноправные категории среди остальных 18 категорий Блоков) — сток-демо-блоки пресета (Basic) отсутствуют, подтверждает `blocks: []`.
- Вставка Блока (`services-grid`) → корректный `data-wpb-block`; вставка Элемента (`wpgjb-el-heading`) → тип `text`; вставка Страницы (`wpgjb-tpl-landing-page`) → **все 4 блока последовательности вставились как отдельные, каждый со своим `data-wpb-block`** — не мёртвый HTML, полностью редактируемый и инспектируемый.
- Trait-кнопка "Редактировать контент…" реально открывает Modal с формой (8 полей для `services-grid`), редактирование поля title реально применилось к `data-wpb-values` компонента.
- `rte:enable`-guard (защита текста внутри Блоков от прямого contenteditable в обход формы Контента) продолжает работать под новой панельной системой — независимая подсистема, подтверждено.
- Публикация (`POST /documents/{id}/{type}/publish`) — статус 200.
- Новый `GET /wpgjb/v1/templates` (для будущего AI-модуля, независимо от бутстрапа редактора) — возвращает полный инспектируемый контракт `{block_id, slots}` по каждому блоку шаблона.
- Полная регрессия: `dev/phase{1,2,5,6,7}`, `dev/second-tier`, `dev/checklist`, `dev/snapshots/compare` — все PASS, без изменений.
- Ноль ошибок консоли за весь сценарий (вставка Блока/Элемента/Страницы, Modal, публикация).

## Осознанно вне объёма

- Реальный drag-and-drop мышью не симулировался через Puppeteer (та же причина, что в прошлых сессиях: нативный HTML5 DnD через границу iframe плохо симулируется через CDP) — вставка проверялась через тот же публичный BlockManager API (`get(id).get('content')` → `append()`), которым в реальности пользуется сам механизм перетаскивания.
- Права доступа к полному функционалу (Import/код) — по прямому запросу пользователя НЕ гейтятся отдельной капабилити на этом этапе; двухконтурный санитайзер при save/publish остаётся единственным и достаточным рубежом защиты независимо от того, что вставлено через стоковый Import пресета.

## Что дальше

Из нового, вскрытого этим переходом: библиотека шаблонов "Страницы" сейчас содержит один демонстрационный шаблон (`landing-page`) — расширение каталога реальными шаблонами и, в перспективе, AI-модуль, собирающий страницы через уже готовый REST-контракт (`/wpgjb/v1/templates` + `/wpgjb/v1/pages/assemble`).
