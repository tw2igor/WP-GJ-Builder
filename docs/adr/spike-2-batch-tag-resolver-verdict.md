# Spike 2 (Phase 0): батч-резолвер динамических тегов — вердикт

Дата: 18 июля 2026. Статус: **PASS**, все 5 проверок пройдены — подтверждено дважды реальным выполнением на WordPress (сначала агентом на WP 7.0.2/PHP 8.2, затем независимо координирующей сессией на WP Playground latest/PHP 8.3).

## Архитектура

Плейсхолдер `{{wpb:tag_id;param=value;...}}` (формат из плана, без отклонений). `includes/DynamicTags/TagRegistry.php` — whitelist, каждый тег принадлежит ровно одной группе источника данных (`site`/`post`/`author`/`postmeta`); резолвер тега — чистая функция над уже загруженным контекстом, сам в БД не ходит. `includes/DynamicTags/BatchResolver.php`: (1) дешёвый regex-препроход собирает различные `tag_id`, реально присутствующие на странице, (2) `load_context()` делает запросы, сгруппированные по источнику — `get_userdata()` один раз, только если есть тег автора; `get_post_meta($post->ID)` один раз, только если есть тег postmeta (вне зависимости от числа разных ключей); пост/сайт — ноль доп. запросов, (3) один проход по дереву через `WP_HTML_Tag_Processor`, экранирование по позиции (текст/атрибут/URL).

## Проверено дважды, реальными числами запросов

- 5 вхождений тегов, 2 источника (post+author) → **2 запроса** (≤3).
- Все 4 группы разом, включая 3 разных ключа postmeta → **3 запроса** (≤3).
- Незарегистрированный `tag_id` не резолвится (whitelist держит периметр).
- Один и тот же "вредный" полезная нагрузка корректно экранирована и в text-узле, и в обычном атрибуте, и в URL-атрибуте (в независимой перепроверке координирующей сессией: `&quot;&gt;&lt;script&gt;...` в тексте/атрибуте, `href=""` — `esc_url` вычистил `javascript:`).

## Найденный и исправленный агентом баг (реальная польза от факта, что это НЕ просто трассировка кода)

`WP_HTML_Tag_Processor::set_attribute()`/`set_modifiable_text()` уже сами HTML-кодируют значения — вызов `esc_attr()`/`esc_html()` ДО передачи в процессор давал двойное кодирование (эмпирически подтверждено на реальном прогоне). Не дыра безопасности, но неверный вывод. Исправлено: текст/атрибут получают резолвленное значение как есть, `esc_url()` по-прежнему применяется явно для URL-атрибутов, поскольку у Tag Processor нет понятия "безопасность протокола".
