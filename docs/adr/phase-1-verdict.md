# Phase 1: Dual Storage и модель документа — вердикт

Дата: 18 июля 2026. Статус: **PASS** (после одной находки и исправления), проверено реальным выполнением на WordPress (WP Playground, PHP 8.3).

## Что реализовано

- `includes/Rest/DocumentsController.php` — реальные (не дев-only) REST-маршруты `GET/POST /wpgjb/v1/documents/{post_id}/{type}`: загрузка документа для редактирования и сохранение черновика. Капабилити-гейт: `wpgjb_edit_pages` + штатное `current_user_can('edit_post', $post_id)`. Публикации (запись в `post_content`) здесь сознательно нет — это Phase 2 по плану.
- `includes/Blocks/ManifestSchema.php` + `includes/Blocks/ManifestValidator.php` — схема и валидатор манифеста блока (раздел 9 спеки): `validate_manifest()` (форма самого манифеста) и `validate_values()` (значения слотов человека/AI — один и тот же метод для обоих потребителей, включая рекурсивную проверку вложенных `array.item_schema`).
- `blocks-library/test-hero/manifest.json` — тестовая фикстура манифеста для проверки валидатора.

## Найден и исправлен реальный баг валидатора

`validate_manifest()` изначально прерывался (`return`) сразу после обнаружения ЛЮБОГО отсутствующего обязательного поля верхнего уровня, не проверяя остальные присутствующие поля. Манифест с несколькими независимыми проблемами (нет `purpose`, неверный `section_type`, сломанный слот) показывал только одну ошибку вместо всех — заставило бы чинить по одной ошибке за прогон вместо разом. Обнаружено проверкой через реальный REST-вызов (`GET /wpgjb/v1/dev/phase1`), не код-ревью. Исправлено: каждая проверка теперь независимо гейтится через `array_key_exists()`, отсутствие одного поля не блокирует проверку остальных.

## Проверено (8 из 8 проверок, `GET /wpgjb/v1/dev/phase1`)

1. Валидный манифест (фикстура) проходит `validate_manifest()` без ошибок.
2. Сломанный манифест (нет `purpose`, неверный `section_type`, сломанный слот) даёт **4 отдельные ошибки** (после исправления бага).
3. Валидные значения слотов, включая вложенный `array.item_schema`, проходят `validate_values()` без ошибок.
4. Невалидные значения (нет обязательного `title`, превышена длина, мало элементов массива) дают 3 ошибки.
5. Анонимный REST-запрос без `wpgjb_edit_pages` → HTTP 403.
6. save → load через реальный REST: тот же `project_data`, `doc_version=1`, **`post_content` не тронут** (подтверждает "без рендера" — Phase 1 не пишет в `post_content`, это Phase 2).
7. GET на документ с повреждённым `project_json` → HTTP 200 + `status=frozen` (не 500) — то же структурное свойство, что и в Spike 3, теперь проверено через реальный production REST-путь, а не только через `DocumentRepository` напрямую.
8. Сохранение с заведомо устаревшим `doc_version` → HTTP 409 + `status=conflict`, не молчаливая перезапись (раздел 5.5).
