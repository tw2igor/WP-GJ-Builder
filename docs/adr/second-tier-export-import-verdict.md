# Вторая очередь: Экспорт/импорт страницы в файл — вердикт

Дата: 19 июля 2026. Статус: **PASS**, проверено реальным браузером (скачивание настоящего файла + загрузка файла обратно через `<input type="file">`) на живом WP Playground.

## Что реализовано

Полностью клиентская фича поверх уже существующей инфраструктуры — `project_data` уже спроектирован как чистый сериализуемый JSON (раздел "вторая очередь" плана: "держать `project_json` сериализуемо чистым — тогда экспорт добавляется аддитивно"), поэтому отдельный REST-эндпоинт не понадобился:

- **Экспорт** (`assets/editor/index.js`) — кнопка "Экспорт" вызывает `editor.getProjectData()` (тот же метод, что уже используется при обычном сохранении), оборачивает в `Blob` и скачивает как `wpb-{type}-{postId}-{дата}.json` через временный `<a download>`.
- **Импорт** — кнопка "Импорт" + скрытый `<input type="file" accept="application/json">`. Выбранный файл читается через `FileReader`, парсится, проверяется минимальная форма (`pages` — массив), пользователь явно подтверждает замену текущего содержимого (несохранённые правки будут потеряны), затем `editor.loadProjectData()` загружает данные в canvas и `onContentChanged()` запускает штатное автосохранение.

**Почему это безопасно без отдельной проверки контуром санитизации на импорте**: импортированные данные проходят ТОЧНО ТОТ ЖЕ путь, что и любое обычное редактирование — автосохранение шлёт их на `POST /documents/{id}/{type}`, который уже прогоняет `ProjectDataSanitizer` (контур 1) независимо от происхождения данных. Специального обхода нет и не создавался.

## Проверено

**Реальный браузер (не смоделировано):**
- Клик "Экспорт" на странице с 14 блоками → реально скачан файл `wpb-page-542-2026-07-19.json` (перехвачен через CDP `Page.setDownloadBehavior`) — содержит валидный `pages` массив с всеми 14 компонентами.
- Загрузка ДРУГОГО, заранее подготовленного JSON-файла (один блок `test-cta` с текстом "Imported CTA") через `input[type=file].uploadFile()` → диалог подтверждения с правильным текстом предупреждения → после подтверждения canvas показывает РОВНО 1 блок с текстом "Imported CTA" (было 14) — подтверждает полную замену содержимого, не слияние. Индикатор "Есть несохранённые изменения" корректно активировался.
- Ноль ошибок консоли за весь сценарий.

**Регрессия:** `phase1/2/5/6/7`, `second-tier`, `snapshots` — все PASS (в процессе проверки была одна транзиентная вспышка FAIL на phase5/phase6, не воспроизвелась при 3 повторных прогонах подряд — списано на моментальную нагрузку на WASM-инстанс Playground от параллельных Puppeteer-сессий этой долгой сессии разработки, не на регрессию: экспорт/импорт — чисто клиентская фича, не затрагивающая ничего, что проверяют эти PHP-раннеры).

## Осознанно вне объёма

- Импорт не проверяет, что блоки, упомянутые в файле, СУЩЕСТВУЮТ в библиотеке ЭТОГО сайта до загрузки в canvas — если файл экспортирован с другого сайта с другим набором блоков, несовпадающие блоки будут красиво деградировать через уже существующий `BlockFaultTolerance` (карантин при следующем сохранении/публикации), а не блокироваться на этапе импорта явным предупреждением "3 из 5 блоков не найдены". Более информативное предварительное предупреждение — возможное будущее улучшение UX, не блокирующее базовую функциональность.
- Нет пакетного экспорта/импорта нескольких страниц разом — только одна страница/часть сайта за раз, по одному REST/UI-циклу.
