# План поэтапной раскатки и отката (Phase 7)

Контекст: плагин разворачивается на хостинге для партнёров-веб-студий и их клиентов (см. `docs/builder-analysis.md`) — то есть один релиз потенциально затрагивает множество независимых сайтов одновременно. План ниже описывает, как раскатывать обновления плагина (включая этот, первый MVP-релиз) так, чтобы блокирующая проблема на одном сайте не превращалась в инцидент для всех, и как откатиться, если что-то пошло не так.

## Этап 0 — Предпосылки (выполнено к этой фазе)

- Двухконтурная санитизация проверена адверсариальным корпусом (спайк 4) + независимым security review (Phase 7) — критическая находка (см. `phase-7-security-review.md`) исправлена и закрыта регрессионным тестом.
- Чистое удаление проверено реальным прогоном (`docs/adr/phase-6a-verdict.md`) — деактивация/удаление плагина не требует специальной процедуры отката данных сайта.
- Целевые метрики производительности подтверждены (`docs/adr/phase-6d-verdict.md`).

## Этап 1 — Внутреннее дожитие (dogfooding)

- Установка ТОЛЬКО на тестовые/демо-сайты самого хостинга (не сайты реальных партнёров), минимум 3 разных темы (уже покрыто Phase 4/6 тестами, здесь — финальное подтверждение на staging-копии реальной инфраструктуры хостинга, не WP Playground).
- Критерий перехода на Этап 2: неделя без критичных инцидентов в диагностическом логе (`wpgjb_diagnostics_log` / экран "Настройки" → аудит) на dogfooding-сайтах, все 9 критериев раздела 15 подтверждены в РЕАЛЬНОЙ инфраструктуре хостинга (не только в WP Playground этой разработки).

## Этап 2 — Закрытая бета с партнёрами-волонтёрами

- Плагин доступен для АКТИВАЦИИ вручную (не автообновление) ограниченному числу партнёрских студий, согласившихся на раннее тестирование.
- `wpgjb_manage_settings`/`wpgjb_insert_raw_code` — по умолчанию только у администратора сайта (уже так по умолчанию, раздел 10) — партнёр сознательно решает, кому из своих клиентов открыть доступ, через экран "Настройки" → матрица прав.
- Мониторинг: диагностический канал `raw-code-audit` + `revision-cleanup` + `block-fault-tolerance` — точка расширения для централизованного сбора уже заложена в `Diagnostics::log()` (раздел 12 спеки), но сам центральный сборщик — вне объёма MVP; на этом этапе логи читаются вручную с каждого бета-сайта через экран "Настройки".
- Критерий перехода на Этап 3: минимум 5 партнёров, минимум 4 недели, ни одного инцидента severity "критично" (потеря контента, XSS, публичная утечка данных).

## Этап 3 — Постепенная раскатка по проценту сайтов

- Раскатка мелкими партиями (например, 10% → 25% → 50% → 100% партнёрских аккаунтов, интервал — не короче недели между партиями) — так, чтобы проблема, которая проявляется только на реальном разнообразии контента/тем (не воспроизводимая на dogfooding-сайтах), затронула минимум сайтов до обнаружения.
- Между партиями — явная проверка: количество новых записей в `raw-code-audit`/`block-fault-tolerance` каналах не должно аномально расти (сигнал реальной проблемы, а не просто использования фичи).
- Плановая очистка ревизий (`wpgjb_prune_revisions`, ежедневный wp-cron) уже settings-managed для каждого сайта индивидуально — не требует централизованного контроля при раскатке.

## Этап 4 — Полная раскатка (GA)

- Плагин доступен всем партнёрам хостинга без ограничений.
- `README`/onboarding-материалы (вне объёма этой сессии) — описывают капабилити-модель и то, что "Вставка кода"/Zero-режим — сознательный, редкий, аудируемый обход стандартной санитизации, не то, что стоит включать по умолчанию.

## Путь отката (на любом этапе)

Раздел 12 спеки прямо это гарантирует, проверено реальным прогоном в Phase 6a: **деактивация плагина не требует отдельной процедуры** — `Activation::deactivate()` не трогает ни таблицу документов, ни опубликованный HTML (`post_content` уже статичен и не зависит от плагина). Конкретно:

1. **Откат версии плагина** (баг в новом релизе): заменить файлы плагина на предыдущую версию — `wpb_documents`-таблица и её `schema_version` совместимы вперёд по дизайну миграционной цепочки (`MigrationRunner`), но откат НАЗад через версию схемы не тестировался явно в этой сессии — на практике безопаснее откатывать код БЕЗ отката уже мигрировавших данных (миграции идемпотентны и однонаправленны по замыслу, не пишутся как reversible).
2. **Откат конкретного сайта** (у партнёра проблема, у остальных — нет): деактивация плагина на этом сайте — все уже опубликованные страницы остаются полностью читаемыми (статический HTML), редактирование конструктором станет недоступно до реактивации/фикса. Это ровно то поведение, которое проверено в Phase 6a реальным прогоном "Полная очистка сейчас" + повторная активация.
3. **Откат целиком (все партнёры)**: массовая деактивация — тот же путь, что (2), просто на всех сайтах разом; данные каждого сайта (`wpb_documents`) не удаляются автоматически (только явной опцией "удалить при удалении" + подтверждением, раздел 12), поэтому реактивация без потерь возможна в любой момент после исправления причины отката.

## Что осталось вне объёма этого плана

- Централизованный сборщик диагностики со всех сайтов хостинга (раздел 12 отмечает это как желаемую точку расширения, не требование MVP).
- Автоматизированный canary-мониторинг (алерты по аномальному росту diagnostics-каналов) — на MVP-этапе это ручной процесс.
- Тестирование отката МИГРАЦИИ СХЕМЫ НАЗАД (только вперёд протестировано — Phase 1 спайк 3/дальнейшие фазы).
