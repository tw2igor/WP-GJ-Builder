# WP-Builder: документация для разработки

Конструктор страниц для WordPress, встроенный в хостинг. Концептуальная спецификация продукта.

Дата: 18 июля 2026
Аудитория документа: команда разработки, продакт, дизайнер интерфейса.
Фокус: концепт, требования и правила продукта. Реализация в коде — на усмотрение команды в рамках зафиксированных здесь принципов.

---

## 1. Продукт в одном абзаце

WP-Builder — бесплатный плагин-конструктор для сайтов WordPress на хостинге, предназначенный для партнёров (веб-студий и агентств) и их клиентов. Пользователь собирает страницу сверху вниз из готовых дизайнерских блоков-секций, которые автоматически выглядят в стиле активной темы сайта. Продуктовая модель — Tilda: минимум решений, которые пользователь должен принять сам, максимум готового качества по умолчанию. Технический фундамент — открытый фреймворк GrapesJS, встроенный в собственный плагин.

## 2. Продуктовые принципы (по модели Tilda)

Эти принципы — критерий при любом спорном решении в ходе разработки. Если фича противоречит принципу — фича проигрывает.

**П1. Контент первичен, блок — единица смысла.** Страница — это история, рассказанная сверху вниз; блок — законченное смысловое высказывание (обложка, «о нас», выгоды, отзывы, цены, контакты), а не техническая обёртка. Пользователь думает «добавлю блок с отзывами», а не «добавлю контейнер с тремя колонками». Отсюда: библиотека организована по смыслу контента, а не по типам разметки.

**П2. Библиотека курируется дизайнером.** Каждый блок платформенной библиотеки спроектирован дизайнером на основе типовых паттернов, с выверенной визуальной иерархией. Пользователь не может «сломать» блок — доступные ему настройки заранее ограничены безопасными рамками. Свобода — в выборе блока, а не в возможности произвольно двигать каждый пиксель.

**П3. Два режима работы с блоком: «Контент» и «Настройки».** Точно как в Tilda: у выделенного блока две кнопки. «Контент» — тексты, картинки, ссылки, пункты списка (то, что меняет каждый). «Настройки» — внешний вид: варианты раскладки, отступы, цветовая схема блока, скругления (то, что меняют осознанно). Стилевые настройки никогда не смешиваются с контентными в одной форме.

**П4. Свободный редактор — отдельный режим для продвинутых.** Аналог Zero Block в Tilda: полноценный canvas GrapesJS со свободным редактированием структуры (элементы, слои, произвольные стили) существует, но как осознанный переход «редактировать как Zero-блок», а не как режим по умолчанию. 90% работы происходит на уровне готовых блоков с ограниченными настройками; свободный режим — для оставшихся 10% и для создания собственных блоков.

**П5. По умолчанию — стиль темы.** Блок, вставленный на сайт, немедленно выглядит в стиле активной темы WordPress (цвета, шрифты, скругления) без каких-либо действий пользователя. Ручная настройка — это переопределение поверх наследуемого значения, всегда с видимой отметкой «изменено» и кнопкой «вернуть как в теме».

**П6. Никакой деградации производительности сайта.** Посетитель сайта никогда не загружает ни редактор, ни библиотеки конструктора. Страница отдаётся как заранее сгенерированный статический HTML. Это жёсткое правило, нарушение которого не может быть оправдано ни одной фичей.

**П7. Всё, что видит человек — доступно машине.** Каждый блок описан машиночитаемым манифестом, каждое действие редактора доступно через внутренний API. Будущий AI-модуль собирает страницы теми же операциями, что и человек, — параллельных механизмов нет.

## 3. Термины и модель данных

| Термин               | Определение                                                                                                                                    |
| -------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------- |
| **Страница**         | Стандартная страница WordPress, собранная конструктором. Живёт в обычном списке «Страницы».                                                    |
| **Блок**             | Готовая секция из библиотеки: полноширинная смысловая единица страницы (обложка, отзывы, прайс...).                                            |
| **Элемент**          | Составная часть блока (заголовок, кнопка, карточка). Пользователь работает с элементами только внутри «Контента» блока или в свободном режиме. |
| **Zero-режим**       | Свободное редактирование блока на canvas GrapesJS: структура, слои, произвольные стили.                                                        |
| **Мой блок**         | Блок, сохранённый пользователем в личную библиотеку сайта.                                                                                     |
| **Часть сайта**      | Шапка, подвал или сайдбар, собранные конструктором и привязанные к сайту условиями отображения.                                                |
| **Динамический тег** | Плейсхолдер, подставляющий данные WordPress (название сайта, заголовок записи, автор...) при рендере.                                          |
| **Манифест блока**   | Машиночитаемое описание блока: назначение, тип секции, контентные слоты, ограничения.                                                          |
| **Project data**     | Полное JSON-состояние страницы/блока в формате GrapesJS — источник правды для редактирования.                                                  |

**Правило двойного хранения (фундаментальное).** Каждая страница и часть сайта хранится в двух видах: (1) project data — JSON в собственной таблице плагина, из него редактор восстанавливает страницу без потерь; (2) статический HTML/CSS — результат рендера, сохраняется в `post_content` при публикации, его получает посетитель. Редактор никогда не восстанавливает страницу из HTML; фронтенд никогда не собирает страницу из JSON. Динамические теги в HTML хранятся как плейсхолдеры и резолвятся при отдаче страницы одним батч-проходом.

**Хранение project data — в собственной таблице** (`{prefix}_wpb_documents`: id, post_id, type [page|header|footer|sidebar|block|template], project_json, schema_version, updated_at, updated_by), не в `wp_postmeta` (postmeta с JSON-полями деградирует на объёмах) и не в автозагружаемых `wp_options` (источник хронических проблем производительности у конкурентов). Производные кэши — только с `autoload = no` и со сроком жизни.

## 4. Структура в админке WordPress

Принцип: не создавать параллельных миров. Минимум собственных экранов.

```
Меню WordPress
├── Страницы (стандартный экран WP — точка входа для страниц)
│     ├── колонка-метка «Конструктор» у собранных страниц
│     ├── действие «Редактировать в конструкторе»
│     └── кнопка «Добавить в конструкторе»
└── Конструктор (единственный собственный пункт меню)
      ├── Части сайта   (шапки, подвалы, сайдбары, мои блоки, мои шаблоны)
      └── Настройки     (вкладки: Тема и стили | Права и безопасность | Производительность | Диагностика)
```

Библиотека платформенных блоков не имеет отдельного экрана в админке — она открывается внутри редактора, в момент, когда нужна. Сам редактор — не пункт меню, а полноэкранный режим, открываемый со страницы или из «Частей сайта».

### 4.1. Экран «Части сайта»

Единый экран всего переиспользуемого контента: вкладки «Шапки», «Подвалы», «Сайдбары», «Мои блоки», «Мои шаблоны страниц». Карточки с превью-скриншотом, названием, статусом, сводкой условий отображения (для частей сайта), датой и автором изменения. Действия: редактировать, дублировать, экспортировать в файл, удалить. Поиск и пользовательские категории — для «Моих блоков».

Пустые состояния экранов — не пустота, а объяснение с действием: «Здесь появятся ваши сохранённые блоки. Выделите любой блок на странице и нажмите "Сохранить в мои блоки"» + ссылка на короткую инструкцию.

### 4.2. Экран «Настройки»

- **Тема и стили**: статус интеграции с темой («найдено N цветов, M шрифтов из theme.json» / «тема не отдаёт токены — работает только визуальное наследование»); просмотр токенов темы как палитры; глобальный custom CSS для публичных страниц; custom CSS для редактора; переключатель «блоки могут переопределять акцентный цвет темы».
- **Права и безопасность**: матрица «роль → возможности» (доступ к конструктору; Zero-режим; вставка произвольного HTML/JS; редактирование кода; глобальный CSS); белый список HTML-тегов для контента; аудит-лог вставок произвольного кода.
- **Производительность**: политика ревизий (лимит на страницу), управление кэшем HTML, кнопка «Пересобрать все страницы» (после смены темы).
- **Диагностика**: версия ядра, размер project data по страницам, журнал ошибок рендера/миграций.

## 5. Редактор страницы

Полноэкранный режим. Три зоны: шапка, canvas, контекстные панели. По умолчанию максимум пространства отдано canvas — панели появляются по действию пользователя (принцип Tilda: рабочая область чистая, настройки по Tab/клику).

### 5.1. Шапка редактора

- Слева: название страницы (инлайн-редактирование), статус, «← Назад».
- Центр: переключатель устройств (десктоп / планшет / мобильный, с реальными брейкпоинтами темы).
- Справа: undo/redo, история версий, индикатор сохранения («Сохранено» / «Сохранение…» / «Ошибка — повторим через N сек»; пользователь никогда не гадает, сохранилось ли), «Просмотр» (черновик в новой вкладке), «Опубликовать».

### 5.2. Canvas и работа с блоками (основной режим)

Canvas — iframe с реальными стилями активной темы: основной CSS темы, глобальные стили из theme.json, — то же, что видит посетитель. Критерий приёмки: собранная страница выглядит в редакторе идентично опубликованной, без «доводки после сохранения».

Страница — вертикальная последовательность блоков. Взаимодействия:

- **Добавление блока**: кнопка «+» между блоками и в конце страницы → открывается библиотека (см. раздел 6). Вставленный блок сразу в стиле темы.
- **Выделенный блок** показывает компактную панель действий (по образцу Tilda): **Контент**, **Настройки**, вверх/вниз, дублировать, сохранить в «Мои блоки», скрыть, удалить, «Редактировать в Zero-режиме» (если разрешено правами).
- **«Контент»** — панель с формой: тексты, изображения, ссылки, элементы списков («+ добавить карточку»), в полях — иконка динамических тегов (раздел 8). Никаких стилевых настроек здесь нет.
- **«Настройки»** — панель внешнего вида: варианты раскладки блока (если у блока их несколько), цветовая схема блока (светлая/тёмная/акцентная — из палитры темы), отступы сверху/снизу (шаги, не пиксели), скругления, выравнивание. Каждый параметр показывает унаследованное из темы значение; переопределение помечается индикатором с кнопкой «вернуть как в теме».
- Перетаскивание блоков — с явной линией-индикатором места вставки; все drag-операции дублируются кнопками (вверх/вниз) — и для скорости, и для доступности (WCAG 2.2: всё, что доступно перетаскиванием, доступно и без него).
- Горячие клавиши: Ctrl+C/V/D на уровне блока (в т.ч. между страницами одного сайта), Ctrl+Z/Shift+Z, Tab — скрыть/показать панели.

### 5.3. Zero-режим (свободное редактирование)

Переход явный, по кнопке на блоке. Внутри — полноценный GrapesJS: дерево слоёв, выделение вложенных элементов, произвольные стили через Style Manager (типографика, фон, границы, тени, box-model с визуальным редактором отступов), traits элементов, хлебные крошки пути выделения (Секция → Колонка → Карточка → Кнопка). Стилевые контролы предлагают токены темы первыми (палитра темы, шрифты темы), произвольные значения — вторыми. На вкладке «Дополнительно» элемента: CSS-класс, custom CSS элемента (через уникальный автогенерируемый селектор), видимость по устройствам, якорный ID.

Блок, отредактированный в Zero-режиме, помечается в структуре страницы как «кастомизированный». Новые блоки «с нуля» создаются тоже здесь — из простого каркаса-заготовки.

### 5.4. Настройки страницы

При отсутствии выделения панель показывает настройки страницы: фон, глобальный CSS-класс, якорная навигация. SEO-поля — только если на сайте нет SEO-плагина; при установленном Yoast/Rank Math конструктор не дублирует их поля, а показывает ссылку «SEO настраивается в …».

### 5.5. Сохранение, версии, конфликты

- Автосохранение при реальных изменениях (не по слепому таймеру), не чаще раза в 15-30 сек.
- Ревизии: одна обновляемая ревизия на сессию редактирования + постоянная ревизия на каждую публикацию; лимит хранения настраивается (по умолчанию 10-20 + все публикации), старые вычищаются по расписанию.
- Блокировка одновременного редактирования — через штатный механизм WordPress (Heartbeat): второй пользователь видит «страницу редактирует X», а не молча перезаписывает.
- Оптимистичная блокировка на сохранении: если версия на сервере новее открытой — предложить сравнение, не терять ни один вариант.
- История версий — просмотр и откат к любой сохранённой ревизии с превью.

## 6. Библиотека блоков

Открывается из редактора («+»). Полноэкранный оверлей.

- **Организация по смыслу** (принцип П1): Обложка, О компании, Преимущества, Услуги/Товары, Отзывы, Цены, Команда, Галерея, FAQ, Этапы работы, CTA, Контакты, Разное. Внутри категории — варианты дизайна одного смысла.
- Каждый блок — живое превью (реальный рендер в миниатюре, а не статичная картинка в чужом стиле: пользователь видит блок уже в цветах своей темы — это ключевое отличие от скриншотной библиотеки и прямое следствие П5).
- Переключатель источника: «Платформа / Мои блоки». Поиск. Фильтр «использованные на этом сайте».
- Клик — крупный предпросмотр; «Вставить» — блок появляется на странице в месте вызова «+».
- Вкладка «Страницы»: готовые страницы целиком (лендинг услуги, визитка, портфолио...) — «создать страницу из шаблона» или «вставить все блоки в текущую».

**Требования к каждому платформенному блоку** (чек-лист выпуска блока в библиотеку):
1. Все цвета/шрифты — через CSS-переменные темы с фолбэками; ни одного зашитого hex и font-family.
2. Семантическая разметка (заголовки h2/h3 по иерархии, реальные button/a, alt у изображений).
3. Проверен минимум на 3 темах с контрастными палитрами.
4. Адаптивен на всех брейкпоинтах без ручной донастройки.
5. Заполнен манифест (раздел 9).
6. Ассеты блока (если есть JS — слайдер, табы) подключаются на странице только при фактическом присутствии блока.
7. Изображения-заглушки — нейтральные, с корректными пропорциями под слоты блока.

## 7. Части сайта: шапка, подвал, сайдбар

Собираются в том же редакторе, в контексте части сайта: canvas показывает часть в окружении-заглушке (упрощённое «тело страницы» — видно контекст, редактировать нельзя), библиотека фильтруется по уместности (в шапку — навигация, логотип, CTA; не «отзывы»).

**Условия отображения** (модель Elementor Theme Builder). При публикации части — диалог условий: весь сайт / все страницы / все записи / конкретные страницы / рубрика / тип записи / главная / архивы / 404; каждое условие — «включить» или «исключить»; условий может быть несколько. Конфликт двух шапок на одной странице разрешается специфичностью (конкретная страница > рубрика > весь сайт), при равной — предупреждение пользователю прямо в диалоге. На карточке части — индикатор наличия условий.

**Внедрение в тему.** Для блочных тем — замещение template parts штатным механизмом WordPress. Для классических — перехват `get_header`/`get_footer` с честной индикацией уровня поддержки («полное замещение / вставка над-под / тема несовместима»). Не обещать невозможного: если тема не позволяет корректно заместить шапку, показать это до начала работы, а не после.

**Специальные блоки для частей сайта**: меню WordPress (привязка к существующим menu locations, стилизация под тему), логотип сайта (динамический тег), переключатель «липкой» шапки, копирайт с автогодом.

Изменение опубликованной шапки/подвала = сброс HTML-кэша всех затронутых страниц; операция явная, с сообщением «Обновление применится ко всем страницам сайта».

## 8. Динамические данные

Модель Elementor Dynamic Tags, упрощённая до нужного.

**UX.** У текстовых полей, полей изображений и ссылок в «Контенте» — иконка динамических данных. Список источников по группам. Вставленный тег отображается в поле как чип («Название сайта»), в canvas — как реальное значение с подсветкой при наведении.

**Набор первой очереди.**
- Сайт: название, описание, URL, логотип, иконка.
- Страница/запись: заголовок, отрывок, дата публикации/обновления, featured image (+alt), URL.
- Автор: имя, аватар, био, ссылка.
- Архив (для частей сайта): название и описание рубрики/таксономии.
- Произвольное поле записи (postmeta по ключу). ACF — второй очередью.

**Правила рендера (жёсткие).**
1. В статическом HTML тег хранится как плейсхолдер, не как запечённое значение.
2. Все теги страницы резолвятся одним батч-проходом при отдаче (1-3 подготовленных запроса, не N запросов на N тегов).
3. Только белый список зарегистрированных тегов с типизированными параметрами; никакого исполнения кода из контента.
4. Каждый тег экранирует вывод по контексту (текст / атрибут / URL).
5. Реестр тегов машиночитаем и перечислим через API (тот же список видят человек в UI и AI-модуль).

## 9. Манифесты блоков и готовность к AI-сборке

Библиотека — не набор картинок, а машиночитаемый каталог. У каждого блока (платформенного и «моего») — манифест:

- **Идентификация**: стабильный ID, версия схемы, название.
- **Тип секции** — из фиксированного словаря (hero, about, features, services, pricing, testimonials, team, gallery, faq, steps, cta, contacts, header, footer...); свободные теги — дополнительно.
- **Назначение** — 1-2 предложения естественным языком, когда блок уместен («Первый экран с крупным заголовком и одной CTA-кнопкой; для лендинга услуги»). Пишется автором блока.
- **Контентные слоты** — редактируемые поля с типами и ограничениями: `title (строка, ≤60)`, `subtitle (строка, ≤160)`, `cta_label (строка, ≤25)`, `image (изображение, горизонтальное, ≥1200px)`, `items (массив 3-6 × {icon, title, text})`. Манифест слотов — одновременно контракт для AI-наполнения, валидация ввода человека и документация блока. Форма «Контента» (5.2) генерируется из этого же описания слотов — один источник правды.
- **Контекст**: рекомендуемые соседи («после hero»), допустимость повторов, ниши (опционально).
- **Требования**: используемые динамические теги, подключаемые ассеты.

Для «моих блоков» манифест собирается автоматически (тип — выбор при сохранении, слоты — из структуры), описание — опционально, с пояснением «улучшит будущую AI-сборку».

**Внутренний API каталога** (REST, под правами):
- каталог блоков с манифестами (фильтры: тип, теги, ниша);
- блок целиком (project data + манифест);
- вставить блок в страницу (ID блока + значения слотов → обновлённый project data);
- собрать страницу из последовательности `[{block_id, slots}]` → черновик.

Редактор пользуется этим же API. AI-модуль (вне рамок этого документа) получает готовый контракт: выбирает блоки по манифестам, заполняет слоты контентом, собирает страницу — не зная ничего о внутренностях GrapesJS. Страницы и шаблоны также тегируются (тип страницы, ниша, состав секций — автоматически из блоков).

## 10. Безопасность

Два контура, оба обязательные:

**Контур редактирования.** Серверная валидация всего входящего при каждом сохранении: HTML — по белому списку тегов/атрибутов, CSS — по белому списку свойств с синтаксической проверкой; `<script>` и `on*`-атрибуты запрещены по умолчанию. Право на произвольный HTML/JS-код (спец-блок «Вставка кода») и на Zero-режим — отдельные флаги в матрице прав, по умолчанию выключены; включение — осознанное действие администратора. Аудит-лог: кто и когда вставил произвольный код.

**Контур публикации.** Финальный HTML прогоняется через санитизацию ещё раз непосредственно перед записью в `post_content` — независимо от проверок контура редактирования (защита от прямых обращений к API сохранения в обход редактора). Экранирование всех динамических значений по контексту. CSP — опциональная настройка сайта.

Мультитенантный контекст (тысячи сайтов одного хостинга) повышает цену любой уязвимости редактора: security-ревью перед массовым релизом обязательно; обновления плагина раскатываются поэтапно (staged rollout) с возможностью отката, не на все сайты одновременно.

## 11. Производительность

**Фронтенд (посетитель).**
- Страница — заранее сгенерированный статический HTML; ничего не собирается на лету (кроме батч-резолва динамических тегов).
- CSS/JS конкретного блока подключается только на страницах, где блок реально есть; общий рантайм конструктора — один маленький файл, и только если на странице есть интерактивные блоки.
- Изображения: автогенерация размеров и `srcset` штатным механизмом WordPress, `loading="lazy"` для всего ниже первого экрана, `fetchpriority="high"` и без lazy — для изображения первого экрана; WebP с фолбэком на уровне загрузки. Всё автоматически, без действий пользователя.
- Явная инвалидация кэша страницы при публикации (стандартные хуки, понятные кэш-плагинам хостинга); изменение части сайта инвалидирует все затронутые страницы.

**Редактор.**
- Загрузка редактора — только на его собственном экране; ни один ассет конструктора не подключается ни в остальной админке, ни на фронтенде.
- Мягкие предупреждения о проблемах: загружено изображение аномального веса, чрезмерная вложенность в Zero-режиме — с конкретной рекомендацией, не блокируя работу.

**БД.**
- Project data — собственная таблица (см. раздел 3), никаких JSON-мегаполей в postmeta и автозагружаемых кэшей в options.
- Политика ревизий с лимитами и плановой очисткой.

**Целевые метрики (приёмка MVP):** страница из ~10 платформенных блоков на shared-хостинге без кэш-плагина: TTFB сопоставим со статической страницей Gutenberg (±15%), суммарный CSS/JS конструктора на странице ≤ 60 КБ, PageSpeed (mobile) ≥ 85 на эталонной теме.

## 12. Надёжность

- **Версионирование схемы блоков.** У каждого экземпляра блока на странице — версия схемы. Обновление библиотеки сопровождается миграциями: устаревший JSON прогоняется через цепочку миграций при открытии; неудача миграции — блок «замораживается» в последнем валидном HTML (страница продолжает работать), инцидент логируется.
- **Отказоустойчивый рендер.** Один повреждённый/неизвестный блок не роняет страницу: пропускается с записью в журнал диагностики, остальная страница публикуется.
- **Регрессионные снимки.** Релиз обновления библиотеки блоков тестируется на сохранённых JSON-снимках предыдущих версий каждого блока.
- **Чистое удаление.** Полная очистка при удалении плагина (таблицы, опции, cron-задачи, транзиенты) через штатный uninstall-механизм; галочка «удалить данные при удалении» (по умолчанию выключена) + отдельная явная «Полная очистка». Контент сайтов (страницы с готовым HTML) при удалении плагина остаётся читаемым — статический HTML не зависит от плагина.
- **Наблюдаемость.** Единый формат журнала ошибок (рендер, миграции, конфликты сохранения) с точкой расширения под централизованный сбор со всех сайтов хостинга.

## 13. Стандарты интерфейса

- Панели и меню редактора следуют дизайн-системе админки WordPress (@wordpress/components-совместимый визуальный язык); собственный стиль — только внутри canvas (там стиль темы сайта).
- Доступность: полная работа с клавиатуры (видимый фокус ≥2px с контрастом ≥3:1, логичный Tab-порядок, без ловушек), ARIA-разметка панелей и уведомлений, все drag-действия имеют кнопочные альтернативы. Тест скринридером — пункт чек-листа релиза.
- Уведомления: тосты с «Отменить» для обратимых действий; модальные подтверждения — только для необратимых (удаление страницы, полная очистка).
- Онбординг: пропускаемый тур из 2-3 шагов при первом входе; пустые состояния всегда содержат следующее действие.
- Язык интерфейса: русский и английский с самого начала (i18n-готовность всех строк).

## 14. Состав MVP и вторая очередь

**MVP:**
- Редактор: canvas со стилями темы, блочный режим («Контент»/«Настройки»), библиотека, «Мои блоки», undo/redo, автосохранение, ревизии, блокировки, индикатор сохранения.
- Библиотека платформы: стартовый набор (ориентир — 60-100 блоков по всем смысловым категориям + 10-15 готовых страниц), все с манифестами.
- Zero-режим (за флагом прав).
- Шапка и подвал с условиями отображения (сайдбар — вторая очередь).
- Динамические теги первой очереди.
- Двухконтурная безопасность, матрица прав, аудит-лог.
- Производительность по разделу 11, включая целевые метрики.
- API каталога — read-only + вставка блока.
- Чистое удаление, миграции схемы, отказоустойчивый рендер.

**Вторая очередь:** сайдбары, ACF-теги, экспорт/импорт в файл, API сборки страницы целиком, централизованная телеметрия, полный WCAG-аудит, интеграционные ссылки на SEO-плагины, лимиты/тарифные точки расширения, AI-модуль.

## 15. Критерии приёмки продукта (сквозные)

1. Пользователь без обучения собирает лендинг из 6-8 блоков за 15 минут, и он выглядит в стиле темы без ручной стилизации.
2. Страница в редакторе и опубликованная страница визуально идентичны.
3. Смена темы сайта: страницы конструктора адаптируются к новой палитре/шрифтам без редактирования (проверка на 3 контрастных темах).
4. Метрики производительности раздела 11 выполнены.
5. Два пользователя не могут молча перезаписать работу друг друга.
6. Повреждение одного блока не ломает ни страницу, ни редактор.
7. Пользователь без права на код физически не может вставить исполняемый скрипт ни через один интерфейс, включая прямые API-запросы.
8. Удаление плагина оставляет читаемые страницы и не оставляет данных в БД (при включённой очистке).
9. Внешний скрипт, зная только REST API каталога, может получить список блоков с манифестами и вставить блок с заполненными слотами в черновик страницы.
